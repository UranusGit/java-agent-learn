## 0. Web 项目特有的问题

CLI / 桌面 app 不会有这些，但 Web 一定有：

| 问题 | 后果 |
|------|------|
| 用户开 2 个 tab 看 session A，在 tab 1 审批，tab 2 不知道 | 重复审批 / 状态不一致 |
| 手机切到后台 30s 后 iOS Safari 冻结 WS | 重连后 missed events 看起来"卡住" |
| 用户在地铁里发的消息，断网期间走丢 | 消息永久丢失，agent 没收到 |
| 团队场景：3 个人同时看同一 task | 谁在审 / 谁在线 / 谁改了什么都不知道 |
| localStorage 5MB 限制 | 缓存的活动流爆掉 |
| 多设备：手机发的，电脑没刷新看不到 | 需要跨设备同步 |

CLI 单进程不需要解决这些；Web 必须做。

### 2.1 Visibility API + 心跳

iOS Safari 把 tab 切到后台 30s 后会冻 WebSocket，回来时连接已死。需要：

  useEffect(() => {
    const onChange = () => {
      if (document.visibilityState === 'visible') onVisible();

    document.addEventListener('visibilitychange', onChange);

  private heartbeatTimer?: number;

  connect() {
    this.ws = new WebSocket(url);
    this.ws.onopen = () => this.startHeartbeat();
    this.ws.onclose = () => this.stopHeartbeat();
    // ... 08 章的其余逻辑

      if (this.ws?.readyState === WebSocket.OPEN) {

### 2.3 后台冻结策略

| 状态 | 行为 |
|------|------|
| visible + 在审 | 心跳 25s，WS 保持 |
| hidden < 5min | 心跳 25s（保持长任务可观察） |
| hidden > 5min | 主动断 WS，避免无谓心跳；visible 时立即重连 |
| hidden + 手机省电模式 | 浏览器会自动断，无需自己处理 |

```ts
// 本代码仅作学习材料参考
let hiddenSince = 0;
document.addEventListener('visibilitychange', () => {
  if (document.visibilityState === 'hidden') {
    hiddenSince = Date.now();
  } else if (Date.now() - hiddenSince > 5 * 60_000) {
    ws.forceReconnect();
  }
});
```

### 2.4 IndexedDB 缓存活动流

localStorage 5MB 不够，活动流上 IndexedDB：

```ts
// 本代码仅作学习材料参考
// src/lib/eventDb.ts
import { openDB } from 'idb';

const dbPromise = openDB('webclaude', 1, {
  upgrade(db) {
    db.createObjectStore('events', { keyPath: ['sessionId', 'seq'] });
    db.createObjectStore('meta', { keyPath: 'sessionId' });
  },
});

export async function putEvents(sessionId: string, events: AgentEvent[]) {
  const db = await dbPromise;
  const tx = db.transaction('events', 'readwrite');
  for (const e of events) {
    await tx.store.put({ ...e, sessionId });
  }
  await tx.done;
}

export async function getEventsSince(sessionId: string, lastSeq: number): Promise<AgentEvent[]> {
  const db = await dbPromise;
  return db.getAllFromIndex('events', 'by-session-seq', IDBKeyRange.bound([sessionId, lastSeq], [sessionId, Infinity]));
}

// LRU 淘汰：每个 session 最多保留最近 10000 个 event
export async function pruneOldEvents(sessionId: string, keepCount = 10000) {
  const db = await dbPromise;
  const all = await db.getAllFromIndex('events', 'by-session-seq');
  const sessionEvents = all.filter(e => e.sessionId === sessionId);
  if (sessionEvents.length <= keepCount) return;
  const toDelete = sessionEvents.slice(0, sessionEvents.length - keepCount);
  const tx = db.transaction('events', 'readwrite');
  for (const e of toDelete) await tx.store.delete([sessionId, e.seq]);
  await tx.done;
}
```

> IndexedDB 索引需要在 `upgrade` 里建。简化代码省略。

### 2.5 离线消息队列

```ts
// 本代码仅作学习材料参考
// src/lib/offlineQueue.ts
class OfflineQueue {
  private queue: QueuedMessage[] = [];

  push(msg: WSMessage) {
    this.queue.push({ msg, at: Date.now(), attempts: 0 });
    this.persist();
  }

  async flush(ws: SessionWS) {
    while (this.queue.length && ws.isOpen()) {
      const item = this.queue[0];
      try {
        ws.send(item.msg);
        this.queue.shift();
        this.persist();
      } catch (e) {
        item.attempts++;
        if (item.attempts > 5) {
          // 死信：放回队列、提示用户
          toast.error('消息发送失败：' + (item.msg as any).text?.slice(0, 30));
          this.queue.shift();
        }
        break;
      }
    }
  }

  private persist() {
    localStorage.setItem('offline_queue', JSON.stringify(this.queue.slice(-50)));
  }
}
```

## 3. Layer 2：跨标签页同步

### 3.1 问题：默认每 tab 一条 WS

08 章的设计是每 tab 一条 WS，会出现：

- tab 1 和 tab 2 都连同一 session，后端推 event 时都收到，前端各自维护一份 store → 状态可能漂移
- 用户在 tab 2 答了审批，tab 1 的 QuestionCard 还显示
- 后端 session 连接数翻倍

### 3.2 方案 A：BroadcastChannel（推荐 v1）

每个 tab 仍各自 WS，但 tab 之间用 BroadcastChannel 互通：

```ts
// 本代码仅作学习材料参考
// src/lib/crossTab.ts
class CrossTabBridge {
  private channel: BroadcastChannel;

  constructor(sessionId: string) {
    this.channel = new BroadcastChannel(`session:${sessionId}`);
    this.channel.onmessage = (e) => this.onPeerMessage(e.data);
  }

  // 当前 tab 收到 WS event 后，转发给其他 tab
  broadcastFromWs(event: AgentEvent) {
    this.channel.postMessage({ kind: 'event', event });
  }

  // 当前 tab 用户做了动作（审批 / 取消 / 发消息），通知其他 tab
  broadcastAction(action: { type: string; payload: unknown }) {
    this.channel.postMessage({ kind: 'action', action });
  }

  private onPeerMessage(msg: PeerMessage) {
    if (msg.kind === 'event') {
      // 来自兄弟 tab 的 event，更新本 tab store（不再走 WS）
      useActivityStore.getState().append(msg.event);
    } else if (msg.kind === 'action') {
      // 兄弟 tab 的用户动作
      handlePeerAction(msg.action);
    } else if (msg.kind === 'presence') {
      usePresenceStore.getState().updateTab(msg.tabId, msg.state);
    }
  }

  // 选举 leader tab：负责 WS 连接（避免每 tab 都连）
  electLeader() {
    // 简化：用 sessionStorage + 定期心跳
    // 复杂版用 Web Locks API
    navigator.locks?.request('ws-leader', () => this.runLeaderLoop());
  }
}
```

### 3.3 方案 B：SharedWorker（更彻底，复杂）

把 WS 整个搬到 SharedWorker，所有 tab 共享：

```ts
// 本代码仅作学习材料参考
// public/shared-worker.ts
const ports = new Set<MessagePort>();
let ws: WebSocket | null = null;
const subscribedSessions = new Set<string>();

self.onconnect = (e) => {
  const port = e.ports[0];
  ports.add(port);
  port.onmessage = (ev) => handleMessage(ev.data, port);
  port.start();
};

function handleMessage(msg: any, port: MessagePort) {
  if (msg.type === 'subscribe') {
    subscribedSessions.add(msg.sessionId);
    ensureWs(msg.sessionId);
  } else if (msg.type === 'send') {
    ws?.send(JSON.stringify(msg.payload));
  }
}

function broadcastToPorts(data: any) {
  for (const p of ports) {
    try { p.postMessage(data); } catch {}
  }
}
```

tab 侧：

```ts
// 本代码仅作学习材料参考
const worker = new SharedWorker('/shared-worker.js', { name: 'ws' });
worker.port.onmessage = (e) => {
  const event = e.data;
  useActivityStore.getState().append(event);
};
worker.port.postMessage({ type: 'subscribe', sessionId });
```

**取舍**：
- BroadcastChannel 简单，所有浏览器都支持；
- SharedWorker 更省连接（一个浏览器只 1 条 WS），但 Safari 支持有坑、调试麻烦。
- v1 用 BroadcastChannel，规模上来再迁 SharedWorker。

### 3.4 Leader 选举（避免每 tab 都连）

```ts
// 本代码仅作学习材料参考
// 用 Web Locks API 选 leader
async function electLeader() {
  while (true) {
    try {
      await navigator.locks.request('ws-leader', { mode: 'exclusive' }, async () => {
        // 我是 leader，负责 WS
        startWs();
        // 阻塞在这个 lock 上直到 tab 关闭或网络断
        return new Promise<void>(() => {});
      });
    } catch {
      // 失去 leader，等待重试
    }
  }
}
```

只有 leader tab 连 WS，其他 tab 通过 BroadcastChannel 接收。leader tab 关闭时 lock 自动释放，新 tab 接管。

## 4. Layer 3：跨用户实时协作

### 4.1 后端 WebSocket topic

```java
// 本代码仅作学习材料参考
// SessionTopicRegistry.java
@Component
public class SessionTopicRegistry {

    // sessionId → 多个 WS 连接（来自不同用户 / 不同 tab）
    private final Map<UUID, Set<String>> subscribers = new ConcurrentHashMap<>();

    public void subscribe(UUID sessionId, String wsId) {
        subscribers.computeIfAbsent(sessionId, k -> ConcurrentHashMap.newKeySet()).add(wsId);
    }

    public void unsubscribe(UUID sessionId, String wsId) {
        subscribers.getOrDefault(sessionId, Set.of()).remove(wsId);
    }

    public Collection<String> subscribersOf(UUID sessionId) {
        return subscribers.getOrDefault(sessionId, Set.of());
    }
}
```

### 4.2 AgentEventBus 多播

17 章的事件总线扩展为多播：

```java
// 本代码仅作学习材料参考
@Component
public class AgentEventBus {

    private final SessionTopicRegistry topics;
    private final Map<String, WebSocketSession> sessions;

    public void emit(AgentEvent event) {
        persist(event);  // 落 agent_events 表
        // 广播给所有订阅该 session 的连接
        for (String wsId : topics.subscribersOf(event.sessionId())) {
            WebSocketSession ws = sessions.get(wsId);
            if (ws != null && ws.isOpen()) {
                send(ws, event);
            }
        }
    }
}
```

### 4.3 鉴权：能否看 session

不是同一租户的人不能订阅：

```java
// 本代码仅作学习材料参考
@Override
public void afterConnectionEstablished(WebSocketSession ws) {
    UUID sessionId = extractSessionId(ws);
    TenantContext ctx = currentTenant(ws);
    SessionEntity session = sessionRepo.findById(sessionId).orElseThrow();

    // 1. 同租户 + 同 user → 允许
    if (session.getTenantId().equals(ctx.tenantId()) &&
        session.getUserId().equals(ctx.userId())) {
        topics.subscribe(sessionId, ws.getId());
        return;
    }

    // 2. 同租户 + 显式 share token → 只读
    String shareToken = extractShareToken(ws);
    if (shareToken != null && shareService.validate(shareToken, sessionId, ctx)) {
        ws.getAttributes().put("readonly", true);
        topics.subscribe(sessionId, ws.getId());
        return;
    }

    // 3. 其他 → 拒绝
    ws.close(CloseStatus.POLICY_VIOLATION);
}
```

### 4.4 在线状态

后端记录每个 session 的活跃连接：

```java
// 本代码仅作学习材料参考
public record Presence(String userId, String tabId, String device, Instant lastSeen) {}

@Component
public class PresenceTracker {

    private final Map<UUID, Map<String, Presence>> presence = new ConcurrentHashMap<>();

    public void heartbeat(UUID sessionId, Presence p) {
        presence.computeIfAbsent(sessionId, k -> new ConcurrentHashMap<>())
                .put(p.userId() + ":" + p.tabId(), p);
    }

    public Collection<Presence> whoIsOnline(UUID sessionId) {
        return presence.getOrDefault(sessionId, Map.of()).values().stream()
                .filter(p -> p.lastSeen().isAfter(Instant.now().minusSeconds(60)))
                .toList();
    }

    @Scheduled(fixedRate = 30_000)
    public void broadcastPresence() {
        presence.forEach((sessionId, map) -> {
            Collection<Presence> online = whoIsOnline(sessionId);
            eventBus.emit(AgentEvent.system(sessionId, "presence_update", online));
        });
    }
}
```

## 8. 安全注意

- BroadcastChannel 只在 same-origin 内传播，天然安全；
- SharedWorker 同样 same-origin；
- presence 数据只暴露 user 显示名 + 是否在线，**不暴露 token / ip / user-agent 全量**；
- 跨用户场景必须经 share token 鉴权（[23 章](./23-Web安全与可分享性.md)）；
- 离线队列里若有敏感数据，flush 后立即清空，不长期持久化；
- IndexedDB 数据是同源可读，敏感字段（如 model_request 的完整 prompt）按 [23 章](./23-Web安全与可分享性.md) 加密或仅存 metadata。

## 9. 对照：与"08 章单 tab 单人 WS"的差异

| 维度 | 08 章 | 本章 |
|------|-------|------|
| 连接数 | 1 tab × 1 WS | N tab × 1 WS（Leader 模式）|
| 状态同步 | 服务端推 → 单 tab | 服务端推 + BroadcastChannel 分发 |
| 后台冻结 | 重连即可 | 重连 + missed events + IndexedDB 缓存 |
| 多人协作 | 无 | WS topic 多播 + presence |
| 跨设备 | 无 | 偏好 / 草稿跨设备同步 |
| 离线 | 丢消息 | 队列 + 重发 |

**结论**：08 章解决"单客户端消息可达"，本章解决"多端 + 多人 + 多设备一致性"，是 Web 项目独有的复杂度。

---

## 10. 避坑：跨 tab / 实时协作常踩的雷

| 雷 | 现象 | 规避 |
|----|------|------|
| Leader 选举风暴 | 多 tab 抢 Leader 同时检测对方死了 | Web Locks + 心跳超时双校验 |
| presence 隐私泄露 | 谁在线被未授权人看到 | §4 presence 仅 project 内可见 + opt-out |
| BroadcastChannel 兼容 | Safari 老版本不支持 | 降级 localStorage storage event |
| IndexedDB 配额超 | 离线缓存爆 5MB | LRU + size 上限 + 用户告警 |
| iOS WS 强冻结 | 后台 30s 后被杀 | §3 visibility 重连 + push notification 兜底 |
| 同一事件双收 | WS + BroadcastChannel 都收到 | 去重靠 event id |
| presence 心跳抖动 | 网络抖动频繁上下线 | 防抖 + 5s grace period |
| Yjs CRDT 冲突 | 协同编辑冲突不可合并 | 用 Yjs 自动合并，禁止覆盖式更新 |
