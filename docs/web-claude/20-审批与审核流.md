## 1. 工具执行审批（B）

### 1.1 触发条件

PermissionMiddleware 返回 `ask`（05 章）：

- 默认 `default` 模式下，写操作 / 高危读操作；
- 租户管理员配置了"X 工具需要审批"规则；
- PreToolUse hook 返回 `ask`（12 章）。

### 1.2 流程

```
Agent → tool_use(Bash, rm temp/)
  ↓
PermissionMiddleware.evaluate
  ↓ 返回 ASK
QuestionBus.requestApproval(
  decisionKind="tool_approval",
  toolName="Bash",
  toolInput={"command": "rm temp/"},
  reasoning="command matches 'rm' pattern"
)
  ↓ emit decision/requested
前端弹 ToolApprovalCard
  ↓ 用户操作
answer = { allow: true, edit: null, denyReason: null }
  ↓
QuestionBus.submit → PermissionMiddleware 返回 allow
  ↓ 工具执行
```

### 1.3 审批选项

```json
{
  "kind": "tool_approval",
  "payload": {
    "toolName": "Bash",
    "toolInput": { "command": "rm temp/" },
    "risk": "medium",          // low/medium/high/critical
    "reasoning": "command matches rm pattern",
    "preview": "rm temp/"       // 给用户看的简化版
  },
  "options": [
    { "value": "allow",     "label": "允许" },
    { "value": "allow_once", "label": "本次允许（同类不再问）"},
    { "value": "edit",      "label": "修改命令后允许" },
    { "value": "deny",      "label": "拒绝" },
    { "value": "defer",     "label": "稍后决定（离线模式）"}
  ]
}
```

### 1.4 allow_once：会话内信任

用户选 "allow_once" 后，本次会话内同类调用自动放行：

```java
// 本代码仅作学习材料参考
@Component
public class SessionTrustCache {

    private final Map<UUID, Set<String>> cache = new ConcurrentHashMap<>();

    public void trust(UUID sessionId, String toolName, String pattern) {
        cache.computeIfAbsent(sessionId, k -> ConcurrentHashMap.newKeySet())
            .add(toolName + ":" + pattern);
    }

    public boolean isTrusted(UUID sessionId, String toolName, String pattern) {
        return cache.getOrDefault(sessionId, Set.of()).contains(toolName + ":" + pattern);
    }
}
```

### 1.5 edit：用户改命令再允许

用户可以修改 agent 提议的命令：

```json
answer: {
  "action": "edit",
  "editedInput": { "command": "rm -rf temp/" }  // 用户改的版本
}
```

执行的是用户改过的版本。

### 1.6 defer：离线模式

接 12 章 Hooks 的 defer。用户离线时，把审批推到队列，下次回来时处理：

```java
// 本代码仅作学习材料参考
if (!isUserOnline(sessionId)) {
    deferredQueue.push(approval);
    notifyUser(sessionId, "你有 N 条待审批");
    // 返回 default decision（deny + 提示）
    return Answer.defaulted(qid, "deny (deferred)");
}
```

## 3. Diff Review（D）

Coder agent 完成实现后，让用户审 diff。

### 3.1 触发时机

- 每个 feature 完成后（harness 模式）；
- 修改 `immutable_paths` 中的文件时（虽然被 guard 拦，但仍可选 review）；
- 用户配置"每个 commit 都 review"。

### 3.2 流程

```
Coder 完成
  ↓
git diff HEAD~1 HEAD
  ↓
emit decision/diff_review
  payload: {
    files: [
      { path: "src/Register.tsx", additions: 45, deletions: 2, diff: "..." },
      { path: "tests/test_register.py", additions: 30, deletions: 0, diff: "..." }
    ],
    commitSha: "abc1234",
    feature: "F012",
    summary: "实现了用户注册，含表单验证 + 测试"
  }
  ↓
前端 DiffReviewCard
  ↓ 用户操作
answer: {
  action: "approve" | "reject" | "comment",
  comments: [
    { file: "src/Register.tsx", line: 23, text: "这里建议用 useMemo" }
  ]
}
  ↓
approve → git push / 进入下一个 feature
reject → agent 修改后重新 review
comment → agent 看评论，决定是否改
```

### 3.3 UI：分文件展示

```tsx
// 本代码仅作学习材料参考
```

## 4. 多角色协作审批（E）

任务需要 leader + reviewer 都批才执行。

### 4.1 流程

```
agent 请求高危操作
  ↓
emit decision/multi_approver
  payload: {
    requiredApprovers: ["leader", "reviewer"],
    currentApprovals: [],
    payload: {...}
  }
  ↓
leader 看到 → approve
  ↓
reviewer 看到 → approve
  ↓
两个都批 → 执行
任一拒绝 → 取消
超时 → 默认拒绝
```

### 4.2 实现

```java
// 本代码仅作学习材料参考
public class MultiApprover {

    public void requestApproval(UUID sessionId, List<String> roles, Object payload) {
        for (String role : roles) {
            PendingQuestion q = PendingQuestion.builder()
                .kind("multi_approver")
                .payload(Map.of("role", role, "allRequired", roles, "payload", payload))
                ...
                .build();
            bus.persist(q);
            bus.emitRequest(q);
        }
    }

    public void onApprove(UUID questionId, String role) {
        PendingQuestion q = repo.findById(questionId);
        // 检查所有 required 都已 approve
        List<PendingQuestion> all = repo.findByGroupId(q.groupId());
        if (all.stream().allMatch(p -> p.answer().get("approved").equals(true))) {
            // 全通过 → 触发原操作
            bus.completeOriginal(q.toolCallId(), Answer.approved());
        }
        if (all.stream().anyMatch(p -> p.answer().get("approved").equals(false))) {
            // 任一拒绝 → 取消
            bus.completeOriginal(q.toolCallId(), Answer.rejected());
        }
    }
}
```

## 5. 任务阶段确认（F）

Harness 协议（13 章）的每个阶段切换前，让用户确认：

```
phase: init → booting（自动）
booting → working（用户确认"基线 OK，开始？"）
working → verifying（自动）
verifying → committing（用户确认"测试通过，提交？"）
committing → next_feature（用户确认"进入下一个？"）
```

### 5.1 实现

```java
// 本代码仅作学习材料参考
public void transitionPhase(UUID sessionId, String fromPhase, String toPhase) {
    if (requiresUserConfirmation(fromPhase, toPhase)) {
        PendingQuestion q = PendingQuestion.builder()
            .kind("task_phase_confirmation")
            .payload(Map.of("from", fromPhase, "to", toPhase,
                "feature", currentFeature,
                "summary", summarizeWork()))
            .build();
        bus.persist(q);
        bus.emitRequest(q);
        // 挂起直到用户确认
    } else {
        doTransition(sessionId, toPhase);
    }
}

private boolean requiresUserConfirmation(String from, String to) {
    return (from.equals("booting") && to.equals("working"))
        || (from.equals("verifying") && to.equals("committing"));
}
```

### 5.2 异步任务无人值守

长程任务用户不在线，phase confirmation 应**自动通过**或配置为自动：

```java
// 本代码仅作学习材料参考
if (ctx.isAutonomousMode()) {
    // 自动通过所有 phase confirmation
    return Answer.defaulted(qid, "approved");
}
```

## 6. 异常处理询问（G）

agent 遇到非预期情况，主动问用户怎么办。

### 6.1 场景

- 测试连续失败 3 次同一 feature；
- git 冲突无法自动解决；
- 需要的 API 文档访问不到；
- 用户的初始需求被现实证明不可行。

### 6.2 流程

agent 调 `AskUser` 工具，kind=free-text，prompt=问题描述。

```
agent：测试失败 3 次，错误：TypeError: cannot read 'user.name' of undefined。
       看起来是 User model 缺字段。选项：
       a) 我加 user.name 字段
       b) 你确认下数据库 schema
       c) 跳过这个 feature
```

### 6.3 卡住检测

agent 主动调 `ReportBlocker` 工具（不是 AskUser）：

```java
// 本代码仅作学习材料参考
@Component
public class ReportBlockerTool implements Tool {

    @Override
    public CompletableFuture<ToolResult> apply(Map<String, Object> input, ToolContext ctx) {
        // 把 task 状态标 blocked
        taskService.markBlocked(ctx.taskId(), (String) input.get("reason"));
        // 推送通知给用户
        notifyUser(ctx.userId(), "任务卡住：" + input.get("reason"));
        // session end_turn
        return CompletableFuture.completedFuture(ToolResult.text("已上报阻塞，session 结束"));
    }
}
```

## 8. 超时降级

所有 decision 都有超时：

| kind | 默认超时 | 超时行为 |
|------|---------|---------|
| clarification | 5 min | 用 defaultDecision 或 cancel |
| tool_approval | 1 min | deny（安全优先）|
| task_plan_review | 1 day | 用初始 plan |
| diff_review | 1 day | approve（不阻塞流水线）|
| multi_approver | 1 day | 任一超时 = 拒绝 |
| task_phase_confirmation | 5 min | autonomous mode 时 approve |
| exception | 1 day | cancel task |

## 9. 审计与撤销

### 9.1 审计表

```sql
-- 本代码仅作学习材料参考
CREATE TABLE decision_audit (
    id UUID PRIMARY KEY,
    decision_id UUID NOT NULL,
    session_id UUID NOT NULL,
    user_id UUID NOT NULL,
    action VARCHAR(32) NOT NULL,    -- allow/deny/edit/approve/comment/revise
    payload JSONB NOT NULL,
    reason TEXT,
    at TIMESTAMP NOT NULL DEFAULT NOW()
);
```

所有动作都记录，可追溯到具体人。

### 9.2 撤销

某些操作可以撤销：

- 工具执行后 → git revert；
- 文件写入 → 从 git 恢复；
- 任务 commit → git reset + 强制 push（高危，需二次确认）；
- feature 完成 → reopen feature（标 passes=false）。

实现：

```java
// 本代码仅作学习材料参考
@PostMapping("/api/decisions/{id}/undo")
public void undo(@PathVariable UUID id) {
    DecisionAudit a = auditRepo.findById(id);
    switch (a.kind()) {
        case "tool_approval" -> undoTool(a);
        case "diff_review_approve" -> gitRevert(a.payload().commitSha());
        case "task_phase_confirm" -> revertPhase(a);
    }
}
```

## 11. 对照：与"每类场景各写一套 handler"的差异

| 维度 | 各自为政（反例）| 统一抽象（本章）|
|------|---------------|---------------|
| 事件类型 | 6 种独立事件 | 1 种 decision + kind |
| 前端组件 | 6 个独立 Card | 共用 Card + kind 分支 |
| 超时策略 | 每类各写 | §8 统一 |
| 审计 | 各表各存 | decision_audit 一表 |
| 撤销 | 各写 cancel | §9 统一 |
| 跨场景复用 | 几乎为零 | 19 章 QuestionBus 直接复用 |

**结论**：抽象代价是把 kind 多态带进来，但换来了"加一类 HITL 只加一行注册"，远比写一整套 handler 划算。

## 13. 避坑：HITL 场景常踩的雷

| 雷 | 现象 | 规避 |
|----|------|------|
| 审批永久挂死 | 用户关浏览器，session 卡住 | §8 超时降级（默认 5min 自动拒）|
| 多角色审批冲突 | 两人同时批 / 一人批一人拒 | §4 写 quorum + 锁状态机 |
| 超时漏降级 | 定时器漏跑或并发取消 | 用 ScheduledExecutor + 幂等 cancel |
| Diff Review 误批 | 大 diff 一眼滑过 | §3 强制分行预览 + 危险行高亮 |
| 撤销窗口已执行 | 用户撤时工具已跑完 | §9 写入 cancel 不保证停，配合 06 沙箱 kill |
| 决策事件丢 | WS 断开 decision 没到 | 复用 08 章 seq 重连 + 落 DB 后再 ACK |
| AskUser 卡队列 | 后续问题被前一个挡住 | QuestionBus 单独队列 + 超时自动弹下一个 |
| 审批权限越权 | 普通用户批管理员操作 | §4 角色 RBAC + 第一关校验 |
