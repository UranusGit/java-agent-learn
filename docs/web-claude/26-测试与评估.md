## 0. 测试金字塔

```
              ╱─ Eval 案例集（LLM judge） ─╲         <- 慢、贵、定性
           ╱                              ╲
        ╱──── E2E（Playwright） ────────╲         <- 慢、覆盖关键流程
      ╱                                  ╲
    ╱─────── 集成测试（Testcontainers）─────╲       <- 中、覆盖跨模块
  ╱                                          ╲
 ╱──────── 单元测试（JUnit / Vitest）──────────╲     <- 快、覆盖算法
```

| 层 | 比例 | 速度 | 覆盖 |
|----|------|------|------|
| 单元 | 70% | < 10ms/个 | 算法 / 工具内部逻辑 / 状态机 |
| 集成 | 20% | < 1s/个 | 跨模块、真实 DB / Redis / S3 |
| E2E | 7% | 10-60s/个 | 用户视角关键流程 |
| Eval | 3% | 30s-5min/个 | Agent 行为 / 安全 / 上下文 |

```java
    @Test
    void topo_sort_respects_dependencies() {
        Subagent a = sub("A", List.of());
        Subagent b = sub("B", List.of("A"));
        Subagent c = sub("C", List.of("A"));
        Subagent d = sub("D", List.of("B", "C"));

        // 或 "A", "C", "B", "D"
    }

    @Test
    void detects_cycle() {
        Subagent a = sub("A", List.of("B"));
        Subagent b = sub("B", List.of("A"));
        assertThrows(CyclicDependencyException.class, () -> scheduler.topoSort(List.of(a, b)));
    }
```

## 5. Agent Eval 框架

### 5.1 设计

每个 eval case：
- **输入**：用户消息 + 沙箱初始状态 + 可选 mock 工具；
- **期望**：断言（终态 / 工具调用序列 / 安全事件 / 时间预算）；
- **判定**：rule-based 或 LLM judge。

### 5.2 表结构

```sql
-- 本代码仅作学习材料参考
CREATE TABLE eval_cases (
    id UUID PRIMARY KEY,
    suite VARCHAR(64) NOT NULL,        -- tool_use / safety / long_task / context_engineering
    name VARCHAR(255) NOT NULL,
    initial_state JSONB,               -- 沙箱文件 / DB seed
    user_messages JSONB NOT NULL,      -- 一轮或多轮
    assertions JSONB NOT NULL,         -- 断言列表
    budget_ms INT NOT NULL DEFAULT 60000,
    tags TEXT[],
    UNIQUE (suite, name)
);

CREATE TABLE eval_runs (
    id UUID PRIMARY KEY,
    case_id UUID NOT NULL REFERENCES eval_cases(id),
    started_at TIMESTAMP NOT NULL,
    ended_at TIMESTAMP,
    passed BOOLEAN,
    actual_output JSONB,
    assertion_results JSONB,
    cost_usd DECIMAL(10,4),
    model VARCHAR(64)
);
```

### 5.3 断言 DSL

```java
// 本代码仅作学习材料参考
public sealed interface Assertion {
    record ToolCalled(String tool, int minTimes) implements Assertion {}
    record ToolNotCalled(String tool) implements Assertion {}
    record FileContains(String path, String pattern) implements Assertion {}
    record NoSecurityEvent() implements Assertion {}
    record FinalTextContains(String pattern) implements Assertion {}
    record LlmJudge(String criteria) implements Assertion {}  // 用第二个模型判定
    record WithinTurns(int max) implements Assertion {}
    record WithinBudgetTokens(int max) implements Assertion {}
}

public record AssertionResult(boolean passed, String reason) {}
```

### 5.4 Runner

```java
// 本代码仅作学习材料参考
@Component
public class EvalRunner {

    public EvalRunResult run(EvalCase c) {
        // 1. 准备沙箱
        Sandbox s = sandboxFactory.freshFromState(c.initialState());

        // 2. 跑 agent loop
        SessionEntity session = sessionLauncher.launchForEval(s.getId());
        for (String msg : c.userMessages()) {
            agentLoop.run(session.getId(), List.of(), msg, ...).blockLast();
        }

        // 3. 收集终态
        EvalContext ctx = new EvalContext(
            messageRepo.findBySessionOrdered(session.getId()),
            securityEventRepo.findBySession(session.getId()),
            sandboxFileReader.readAll(s.getId()),
            tokenUsage(session.getId())
        );

        // 4. 跑断言
        List<AssertionResult> results = c.assertions().stream()
            .map(a -> evaluator.evaluate(a, ctx))
            .toList();

        boolean passed = results.stream().allMatch(AssertionResult::passed);
        saveRun(c, ctx, results, passed);
        return new EvalRunResult(passed, results, ctx);
    }
}
```

### 5.5 LLM Judge

```java
// 本代码仅作学习材料参考
@Component
public class LlmJudgeEvaluator {

    public AssertionResult evaluate(LlmJudge j, EvalContext ctx) {
        String prompt = """
            你是评估员。判断 agent 的输出是否满足以下标准：
            
            标准：%s
            
            用户输入：%s
            Agent 最终输出：%s
            
            只输出 JSON：{"pass": true/false, "reason": "..."}
            """.formatted(j.criteria(), ctx.userMessages(), ctx.finalText());

        JudgeVerdict v = judgeModel.ask(prompt);
        return new AssertionResult(v.pass(), v.reason());
    }
}
```

## 11. 避坑：测试与评估常踩的雷

| 雷 | 现象 | 规避 |
|----|------|------|
| Mock 漂移 | 真实 API 改了 mock 没改 | 定期重录 + Mock 对比 |
| Eval 误判 | LLM judge 不稳定 | 多次投票 + 人工抽样 |
| E2E flaky | 时序不稳定 | 显式 wait + retry + 隔离 |
| 集成测试慢 | 每次起容器 30s | Testcontainers reuse + 并行 |
| 单元测试覆盖率高无意义 | 覆盖率 90% 但 happy path | 看变异测试 |
| Eval 案例污染 | judge 模型 = 被测模型 | 用更强模型 judge |
| CI 队列堵 | 全套跑 1h | 分层并行 + 关键路径优先 |
| 测试数据写死 | 真实环境不通用 | 用 fixture + 参数化 |
| 性能基准环境抖动 | JMH 在 CI 噪声大 | 独立机器 + 多轮预热 |
| 混沌测试误伤生产 | 注入到 prod | 独立环境 + 标记 + 自动回滚 |
