## 1. 目标拆解与术语统一

### 1.1 用户原始需求（保留原话）

> "我想写一个网页版的 claude，并且拥有长程任务的能力。"

进一步分解：

| 子能力 | 解释 | 难点 |
|--------|------|------|
| 网页版 UI | 浏览器中可对话、可看工具调用、可看 diff | 实时性、富文本渲染、流式输出 |
| Agent 内核 | 复用 Claude Code 的 Agent Loop / 工具系统 / 权限模型 | 不能直接抄 CLI 入口，需要 Web-friendly 的 Engine |
| 长程任务 | 任务可以跑数小时甚至数天，可断点续跑 | 状态持久化、跨 session 上下文恢复、外部触发 |
| 真实可用 | 给其他人用，不是 demo | 多租户、配额、安全、可观测 |

### 1.2 把"长程任务"翻译成工程问题

"长程"不是"一次跑很久"，而是"任务的生命周期超过单次会话的生命周期"。具体表现：

1. **上下文会被压缩 / 截断**：长任务必然触发 Claude Code 的 `autocompact` / `microcompact`，必须保证压缩后还能继续工作；
2. **执行可能被中断**：用户关浏览器、网络掉线、服务重启、模型限流 —— 每一种都需要可恢复；
3. **状态必须外部化**：进度、已完成项、待完成项不能只活在 LLM 的"记忆"里，要落到文件 / DB；
4. **需要外部驱动器**：Cron、事件、用户审批 —— 这些是触发"下一回合"的外部信号源。

**这正是 Anthropic 在 *Effective Harnesses for Long-Running Agents* 一文中强调的核心**：

> "Long-running agents must work in discrete sessions, each new session starts with no memory."

→ 工程上的等价表述：**长程任务 = 状态机 + 多个独立会话 + 状态外部化**。

### 1.3 子问题清单（设计阶段逐一回答）

1. Web UI 如何承载流式输出与人工干预？
2. Agent Loop 在 Web 服务端如何启动、调度、隔离？
3. 会话与长程任务的关系是什么？（1 个任务 = N 个会话？）
4. 工具系统如何兼顾"用户看得见"和"权限可控"？
5. 权限模型如何从单用户扩展到多租户？
6. 状态如何外部化（feature list、progress、artifacts）？
7. 中断与恢复的语义是什么？（关浏览器 ≠ 取消任务）
8. 如何复用现有 ai-serving 基础设施？
9. 浏览器侧的安全沙箱如何选型？
10. 可观测性指标有哪些？

## 2. Claude Code 源码精要（基于 45 章源码分析）

> 出处：`/Volumes/data/远端代码仓/VibeCoding/agent-docs/claudecode源码分析/`
> 重点章节：ch02 / ch05 / ch20 / ch27 / ch30 / ch45

### 2.1 四层架构（ch02）

Claude Code 内部分四层：

```
┌────────────────────────────────────────────┐
│  UI 层    TTY 渲染 / Ink 组件 / 输入处理     │
├────────────────────────────────────────────┤
│  Engine 层 Agent Loop / 工具调度 / 流式输出   │
├────────────────────────────────────────────┤
│  Service 层 MCP / 权限 / 配置 / 鉴权         │
├────────────────────────────────────────────┤
│  Infrastructure 层 文件 / 进程 / 网络 / 缓存  │
└────────────────────────────────────────────┘
```

关键 facade：`QueryEngine.ts` —— 对外暴露的统一入口；核心循环：`query.ts`。

**对 Web 的启示**：
- **UI 层必须替换**（TTY → Web），这是最大改造点；
- **Engine 层几乎可以平移**：AsyncGenerator 驱动的 Agent Loop 本身就与 TTY 无关；
- **Service 层需要扩展**：权限模型要支持多租户，配置要支持远端加载；
- **Infrastructure 层需要替换**：本地 FS → 沙箱 FS，本地进程 → 远端容器。

### 2.2 Agent Loop 与 AsyncGenerator（ch05）

`query()` 是一个 **AsyncGenerator**，每次 `yield` 一个 `State`：

```ts
async function* query(userMessage, options): AsyncGenerator<State> {
  let state = State.initial(userMessage);
  while (true) {
    state = await stepForward(state);
    yield state;
    if (shouldTerminate(state)) return;
  }
}
```

**State 对象核心字段**（设计 Web 后端时可直接借用）：

| 字段 | 含义 | Web 适配要点 |
|------|------|--------------|
| `messages` | 当前消息数组 | 落库；前端订阅增量 |
| `toolUseID` | 工具调用 ID | 用来匹配流式片段 |
| `transition` | 当前阶段标记 | 前端据此切换 UI 状态 |
| `abortController` | 中断信号 | 必须支持父-子级联 |
| `usage` | token 消耗 | 落库做配额 |
| `parentUuid` | 父消息 UUID | DAG 关系，用于压缩 |

**5 个终止条件**（Web 后端必须支持）：
1. 模型主动 end_turn；
2. 用户中断（abort）；
3. 超出预算（context budget 触发）；
4. 模型 fallback 也失败；
5. 进入 wait-for-input 等待审批。

**3 层中断**：
- L1：单工具调用级 AbortController；
- L2：回合级（一次 query）AbortController；
- L3：session 级（用户主动 / 超时）。

→ **Web 适配**：每一层都要映射到 HTTP/WebSocket 的取消语义。L3 在 Web 上特别重要 —— 用户关闭浏览器后，任务应该被「挂起」还是「取消」？这是产品决策，需要明确。

**模型 fallback 与 max_output_tokens 恢复**：
- 当模型返回空内容 + `max_output_tokens` 标志时，触发 `stripSignatureBlocks` 并 fallback 到更小的模型；
- 这是"渐进降级"原则的体现。

→ **Web 适配**：fallback 路径必须可视化（用户能看到"当前模型降级到 X"）。

### 2.3 权限系统（ch20）

**6 种模式**：
1. `default` —— 首次询问，记忆用户选择；
2. `acceptEdits` —— 自动允许编辑；
3. `plan` —— 只规划不执行；
4. `bypassPermissions` —— 全部放行（危险）；
5. `autoAccept` —— 自动接受所有工具调用；
6. `yolo` —— 完全无确认（高危）。

**3 种规则行为**：`allow` / `deny` / `ask`。

**规则 DSL 语法**：`ToolName(content)`，例如 `Bash(rm -rf:*)` 表示禁止任何 `rm -rf` 开头的命令。

**Deny-first 原则**：deny 一票否决，allow 顺序判定，最后才落 `ask`。

**8 个规则来源**（优先级从高到低）：
1. CLI flag
2. session 内存
3. enterprise policy
4. local project `.claude/settings.json`
5. user 全局 `~/.claude/settings.json`
6. plugin
7. default
8. code

→ **Web 适配**：规则来源必须重排，把 enterprise / tenant policy 提到最高；用户偏好也要分级（租户级 vs 项目级 vs 会话级）。这是多租户的关键改造点。

### 2.4 后台任务（ch27）

**两层 Task 模型**：
- **AppState Task**：进程内运行时任务，关进程即失；
- **File-based Task**：持久化任务，跨进程存活。

**File-based Task 关键字段**：
- `blocks` / `blockedBy`：DAG 依赖；
- 文件锁（最多 30 次重试）；
- 确定性 jitter（用 task ID 哈希）。

**Cron**：
- **durable Cron**：跨进程，落盘；
- **session-only Cron**：进程内，关进程即失。

**Scheduler Lock**（leader election）：
- 同一时刻只有一个 worker 持锁调度；
- 防止多副本重复触发。

→ **Web 适配**：File-based Task 与 Cron 几乎可以直接平移；Scheduler Lock 在多副本部署下是必需品（参考 Redis 分布式锁 / DB SELECT FOR UPDATE）。

### 2.5 会话持久化（ch30）

**存储格式**：JSONL，每行一条消息；通过 `parentUuid` 形成 DAG。

**关键设计**：
- **Compact boundary**：压缩发生时插入特殊消息，标记前后不可拼接；
- **Lazy materialization**：消息按需加载，不全量读入内存；
- **Batch write queue**：写消息进队列，批量落盘；
- **recoverOrphanedParallelToolResults**：进程崩溃时，恢复"父亲已写但孩子未写"的孤儿结果；
- **Sidecar 文件**：与主 JSONL 并列的辅助文件（计数、索引等）。

→ **Web 适配**：
- JSONL 在 Web 上同样可用，但建议直接落到对象存储 + DB 索引（CLI 是单机，Web 是多机）；
- Compact boundary 的概念必须保留 —— 它是"长程任务可续跑"的物理基础；
- Lazy materialization 在 Web 后端尤其重要，否则长 session 会撑爆内存。

### 2.6 七大架构模式与 8 条实践建议（ch45）

**七大模式**：

| 模式 | 一句话 | Web 上的落点 |
|------|--------|-------------|
| Agent Loop | AsyncGenerator 驱动 | Engine 层 |
| Tool Registry | 工具按需注册、按权限过滤 | Service 层 |
| Permission Middleware | 所有工具调用过中间件 | Service 层（必须扩展） |
| Context Budget | 多级压缩 | Engine 层 |
| Multi-Agent Delegation | 主 Agent 委派子 Agent | Service 层 + 调度 |
| Streaming Pipeline | 全链路流式 | 必须 WebSocket/SSE |
| Progressive Degradation | 模型 / 上下文 / 工具逐级降级 | 全栈 |

**8 条实践建议（ch45 原文摘录）**：
1. 用 AsyncGenerator 表达 Agent Loop；
2. 工具调用前过权限中间件；
3. 上下文预算优先级 > 截断；
4. 流式输出贯穿全链路；
5. 子 Agent 隔离上下文；
6. 持久化所有状态到文件；
7. 模型 / 工具 / 上下文都要有降级路径；
8. 中断要分层、可级联。

## 3. 官方文档三件套（精读笔记）

### 3.1 Hooks 系统（https://code.claude.com/docs/en/hooks）

#### 3.1.1 五种 Hook 类型与约 30 个事件

| Hook 类型 | 触发点 | 典型用途 |
|-----------|--------|----------|
| `command` | 执行 shell 命令 | 自定义脚本、文件操作 |
| `http` | 发 HTTP 请求 | 通知外部系统、调用 webhook |
| `mcp_tool` | 调用 MCP 工具 | 复用 MCP 生态 |
| `prompt` | 注入 prompt | 动态调整 prompt |
| `agent` | 触发子 agent | 委派任务 |

3 种节奏（cadence）：
- **每回合（every turn）**：每次 user → assistant 都触发；
- **每工具调用（per tool call）**：每次工具调用前后；
- **每会话（per session）**：session 开始 / 结束。

约 30 个事件，按生命周期分类：
- Session lifecycle（start / end / compact）
- User prompt（submit / change）
- Tool（pre / post / error）
- Model（request / response / fallback）
- Agent（delegate / return）

#### 3.1.2 决策控制：allow / deny / ask / defer

**优先级**：`deny > defer > ask > allow`。

- `deny`：一票否决；
- `defer`：本轮先放行，下一轮再决定 —— **这是非交互模式（`-p`）的关键**；
- `ask`：要求用户确认；
- `allow`：放行。

**PreToolUse 的 defer 模式**（重点）：
- 在 CLI `-p` 非交互模式下，`ask` 会卡死；
- Anthropic 推荐用 `defer`：先放行本轮工具调用，把"需要人工决策"放到下一回合；
- 这让"非交互长任务"成为可能。

→ **Web 适配**：Web 上同时存在"交互"和"非交互"两种场景：
- 用户在线 → 用 `ask`（弹审批 UI）；
- 用户离线 / 后台跑 → 用 `defer`，下次唤醒时处理。

#### 3.1.3 异步 Hook（`async: true`）

异步 Hook 不能阻塞当前回合，但可以在下一回合注入上下文。适合：
- 长耗时的审计日志写入；
- 外部系统状态查询（不要求立即响应）。

#### 3.1.4 权限规则语法（`if` 字段）

支持基于工具内容的 DSL：
- `Bash(rm -rf:*)` —— 禁止 `rm -rf` 命令；
- `Write(~/.ssh/**)` —— 禁止写 SSH 目录；
- `Read(.env)` —— 禁止读 `.env`。

#### 3.1.5 路径占位符

- `${CLAUDE_PROJECT_DIR}` —— 当前项目目录；
- `${CLAUDE_PLUGIN_ROOT}` —— 插件根目录；
- `${CLAUDE_PLUGIN_DATA}` —— 插件数据目录。

→ **Web 适配**：Web 上每个用户的"项目目录"必须是隔离的（沙箱），这些占位符要重定义为"用户工作区根目录"。

#### 3.1.6 HTTP Hook 协议

HTTP Hook 收到 JSON 作为 POST body，返回 JSON 决策。这让 Hook 可以是任意外部服务。

→ **Web 适配**：HTTP Hook 是天然的"扩展点"，Web 平台可以让租户管理员配置 Hook 来接入自己的审计 / 审批系统。

#### 3.1.7 安全最佳实践

- 校验所有输入；
- 引号包裹变量；
- 阻止路径穿越；
- 使用绝对路径；
- 跳过敏感文件（`.env`、`.ssh`、token 文件）。

→ **Web 适配**：Web 上还要加一层"租户隔离"，Hook 执行不能跨租户读写。

### 3.2 Agent View 与 Background Sessions（https://code.claude.com/docs/en/agent-view）

#### 3.2.1 整体模型

- `claude agents` 命令打开"Agent View"，专门管理多个后台会话；
- 会话按状态分组：`Needs input` / `Working` / `Completed` / `Ready for review`；
- 后台会话由 **per-user supervisor 进程** 驱动，与终端解耦。

#### 3.2.2 关键交互

- **Peek（空格）**：在面板中快速预览 / 回复，不切走；
- **Attach（Enter/→）**：完整进入会话；
- **Dispatch（分发）**：三种方式 —— agent view 输入、`/background` 或 `/bg`、`claude --bg`。

#### 3.2.3 文件编辑隔离

- 每个后台会话的文件操作隔离在 **git worktree** 下（`.claude/worktrees/<id>`）；
- 会话结束时 worktree 被删除；
- 这避免了多 session 同时编辑同一个文件的冲突。

→ **Web 适配**：
- Git worktree 模型可以直接借鉴，但 Web 上更推荐**容器 / VM 级隔离**（更安全）；
- 每个 background session = 一个沙箱 = 一个工作区；
- 会话结束 → 沙箱销毁，但**产出的 artifacts 必须先持久化**。

#### 3.2.4 持久化与恢复

- 会话在以下情况下持续存活：自动更新、supervisor 重启、机器睡眠；
- **机器关机后会话丢失**（这是产品边界）；
- 后台子 agent 在下次唤醒时从 transcript 恢复。

→ **Web 适配**：Web 平台可以做比 CLI 更好 —— 用数据库 + 对象存储把 session 真正持久化，跨机器迁移也能恢复。

#### 3.2.5 Supervisor 进程

- 每个 user 一个 supervisor 进程；
- 预热 worker（pre-warmed），收到任务立即移交；
- supervisor 负责：shell 命令、动态工作流、后台子 agent。

→ **Web 适配**：
- supervisor 进程在 Web 上等价于"per-user agent worker pool"；
- 用 K8s / Nomad 调度，按用户分配 worker；
-预热 worker 池可以显著降低首字延迟。

#### 3.2.6 每会话目录

- `~/.claude/jobs/<id>` —— 每个会话独立目录；
- `CLAUDE_JOB_DIR` 环境变量 —— 会话内可访问自己的目录。

#### 3.2.7 设置与凭证继承

不同 dispatch 方式有不同的继承规则：
- agent view dispatch 继承当前会话；
- `/bg` 继承当前 + 部分覆盖；
- `claude --bg` 全新启动。

#### 3.2.8 限制（产品边界）

- 限流仍然适用（同一账号的并发限制）；
- 会话是**本地**的，不跨机器；
- 会话删除时 worktree 也删除。

### 3.3 长程 Agent 的有效 Harness（https://www.anthropic.com/engineering/effective-harnesses-for-long-running-agents）

这是本项目最重要的参考，直接定义了"长程任务"的工程范式。

#### 3.3.1 核心论断

> "Long-running agents must work in discrete sessions, each new session starts with no memory."

→ 工程含义：**不要试图让一个会话从头跑到尾，而是拆成多个独立会话，靠外部状态串起来。**

#### 3.3.2 两种失败模式（必须规避）

| 失败模式 | 表现 | 规避手段 |
|----------|------|----------|
| 一次性把 app 做完 | 上下文耗尽、忘记之前做了什么 | 拆成多 session，状态外部化 |
| 过早宣布完成 | 没充分测试就 stop | feature_list + 自动化测试 |

#### 3.3.3 双 Agent 模式

**Initializer Agent（首次运行）**：
- 读需求；
- 生成 `feature_list.json`（核心产物）；
- 创建 `claude-progress.txt`；
- 写 `init.sh`（启动开发服务器 + 基础端到端测试）；
- 第一个 git commit（展示已添加的文件）。

**Coding Agent（后续每个 session）**：
- 读 progress + feature_list + git log；
- 选最高优先级的"未完成"feature；
- 实现 + 测试；
- 更新 progress + 提交；
- 写一段 summary。

#### 3.3.4 `feature_list.json` 的设计（关键）

- **JSON 而非 Markdown**：便于程序化解析；
- **200+ features**：覆盖完整应用所需的所有功能；
- 每个 feature 一开始 `passes: false`；
- **强指令：不许删 / 改测试**（防止 agent 作弊）。

→ **Web 适配要点**：
- Web 平台天然适合做"feature_list 管理 UI"；
- 每个 feature 在 UI 上有状态徽章（未做 / 进行中 / 通过 / 失败）；
- 测试结果回写 feature_list —— 这就是"长程任务进度可视化"的核心。

#### 3.3.5 `claude-progress.txt`

- 简单文本日志；
- 每完成一项追加一条；
- 下次 session 第一步读它。

→ **Web 适配**：可以升级为 DB 表 + 时间线 UI，但**保留"线性追加"语义**（避免并发写冲突）。

#### 3.3.6 `init.sh`

- 启动 dev server；
- 跑基础端到端测试；
- 让 agent 可以"先验证基线再做下一项"。

#### 3.3.7 增量推进原则

- 一次只做一个 feature；
- 每次提交带描述性 commit message；
- 写 summary（给下一次 session 看）。

#### 3.3.8 测试方法

- 显式 prompt 让 agent 用 **Puppeteer MCP** 做浏览器自动化测试；
- "像人一样测试"，不只是跑单元测试。

→ **Web 适配**：Web 平台可以内置 Playwright / Puppeteer 沙箱，agent 可以"自己开浏览器测自己"。

#### 3.3.9 起步流程（每个 session 开头）

1. `pwd` —— 确认工作目录；
2. 读 git log、progress 文件；
3. 读 feature_list；
4. 选最高优先级未完成 feature；
5. 跑 init.sh 确认基线；
6. 实现 → 测试 → 提交 → 更新进度。

#### 3.3.10 未来方向（文章提到）

- **多 agent 架构**：测试 agent / QA agent / 清理 agent 分工；
- **泛化**：不只用于全栈 web 应用，也可用于其他领域。

→ 这是本项目 Phase 2/3 的天然路线图。

## 4. CLI → Web 的迁移映射

> 这是设计阶段的核心输入。把 CLI 概念逐一映射到 Web，找出"必须改造"和"可平移"。

### 4.1 顶层映射表

| 维度 | CLI 形态 | Web 形态 | 改造程度 |
|------|----------|----------|----------|
| 入口 | `claude` 命令 | HTTPS + WebSocket | 全新 |
| UI | TTY / Ink | React + 流式 | 全新 |
| 输入 | stdin | WebSocket / HTTP | 全新 |
| 输出 | stdout 流 | WebSocket 推送 | 全新 |
| 中断 | Ctrl+C | WS close / abort 按钮 | 重做语义 |
| 会话存储 | 本地 JSONL | 对象存储 + DB 索引 | 改造 |
| 任务调度 | 本地 Cron + Task | 分布式 Cron + Job DB | 改造 |
| 工具沙箱 | 本机 FS | 容器 / VM | 改造 |
| 权限 | 单用户 8 源 | 多租户分层 | 扩展 |
| Agent Loop | AsyncGenerator | 同上 + 网络边界 | 几乎平移 |
| Tool Registry | 进程内 | 同上 + 远端 MCP | 平移 |
| Context Budget | 同进程 | 同上 | 平移 |
| 多 Agent | 同进程 subagent | 跨进程 subagent | 改造 |
| 流式管道 | AsyncIterator | WebSocket / SSE | 改造 |
| 渐进降级 | 同上 | 同上 + 多租户限流 | 平移 + 扩展 |

### 4.2 关键改造点详解

#### 4.2.1 流式输出（必须 WebSocket 或 SSE）

CLI 上 AsyncGenerator 直接 yield；Web 上必须：
- 用 WebSocket 长连接推送增量；
- 用 SSE 也行（更简单，但单向）；
- 必须有「断线重连 + 位移续传」机制（参考 JSONL 的 `parentUuid`）。

#### 4.2.2 中断语义（产品决策）

CLI 上 `Ctrl+C` = 立即取消；Web 上必须区分：
- **关闭浏览器** ≠ 取消任务（否则长程任务无法实现）；
- **取消按钮** = 真正取消；
- **超时** = 挂起，等用户回来。

→ 设计建议：用三层中断 + 显式状态机。

#### 4.2.3 工具沙箱（安全边界）

CLI 工具操作本机文件；Web 必须隔离：
- 选项 A：Docker 容器（成熟、重）；
- 选项 B：gVisor / Firecracker（更轻、更安全）；
- 选项 C：WebAssembly（前沿，但生态弱）；
- 选项 D：远端 VM（最重，但完全隔离）。

→ 推荐 A 起步，B 作为 v2 升级路径。

#### 4.2.4 多租户权限（最复杂）

CLI 是单用户；Web 必须支持多租户。权限规则来源重排：

| 优先级 | 来源 | 说明 |
|--------|------|------|
| 1 | Platform policy | 平台运营方强制（合规红线） |
| 2 | Tenant policy | 租户管理员配置 |
| 3 | Project policy | 项目级配置 |
| 4 | User preference | 用户个人偏好 |
| 5 | Session runtime | 会话内临时决定 |
| 6 | Default | 内置默认 |

#### 4.2.5 Background Session → Background Job

CLI 上是 supervisor 进程；Web 上等价于：
- 分布式任务队列（Celery / Temporal / 自研）；
- per-tenant worker pool；
- 持久化到 DB，崩溃可恢复。

#### 4.2.6 Hooks → Webhook + Plugin 系统

CLI 上是 shell / HTTP / MCP hook；Web 上：
- HTTP hook 直接复用；
- shell hook 改为"沙箱内执行"；
- MCP hook 直接复用；
- 新增 "Webhook" 形式（更通用）；
- 新增 "UI Plugin" 形式（前端注入组件）。

### 4.3 Artifacts 模型（Web 独有）

CLI 上"产出"是文件；Web 上要把产出做成一等公民：

| 类型 | 渲染方式 | 用途 |
|------|----------|------|
| 代码 | Monaco editor | 直接看 diff |
| Markdown | 富文本 | 文档型产出 |
| 图片 | 缩略图 + 大图 | 设计稿 |
| SVG | 直接渲染 | 图表 |
| HTML | iframe 沙箱 | 实时预览 |
| React 组件 | 沙箱执行 | 可交互 demo |
| 表格 | 表格组件 | 数据展示 |
| Mermaid | 流程图 | 架构图 |

→ 这是 Claude Cowork 的 Live Artifacts 思路（详见 §5.3）。

## 7. 关键风险与未决问题

设计阶段必须正面回答：

### 7.1 技术风险

| 风险 | 影响 | 缓解方向 |
|------|------|----------|
| WebSocket 长连接稳定性 | 用户体验差 | 自动重连 + 位移续传 |
| 沙箱冷启动慢 | 首字延迟高 | 预热 worker 池 |
| 多租户权限漏配 | 安全事故 | 默认拒绝 + 单元测试 |
| LLM 限流 | 任务中断 | 多 key 轮转 + 配额隔离 |
| 长任务上下文丢失 | 任务失败 | compact boundary + 状态外部化 |
| Hook 慢 / 死循环 | 整体卡住 | 异步 hook + 超时熔断 |

### 7.2 产品风险

| 风险 | 影响 | 缓解方向 |
|------|------|----------|
| "网页版"用户期待高保真终端 | 体验落差 | 用 xterm.js + tmux 模拟 |
| 长任务进度不可见 | 用户焦虑 | feature_list + timeline UI |
| 沙箱中跑代码"不真实" | 信任问题 | git 集成 + 导出能力 |
| 多租户配额争议 | 客诉 | 透明展示 + 自助升级 |

### 7.3 未决问题（设计阶段必须拍板）

1. **语言栈**：Java（Spring Boot 4 + Spring AI 2.0 + WebFlux）还是 TypeScript（Node.js + 自研）？
   - 倾向：Java，与现有 demo01 项目一致，复用 ai-serving。
2. **沙箱技术**：Docker 起步还是直接上 Firecracker？
   - 倾向：Docker 起步，预留接口。
3. **会话存储**：JSONL 文件还是直接进 DB？
   - 倾向：JSONL 落对象存储 + DB 索引（兼顾 CLI 兼容性和查询效率）。
4. **多 agent 模型**：v1 内置还是延后？
   - 倾向：v1 只做主 agent + 工具，v2 加 subagent。
5. **Artifact 沙箱执行**：v1 做不做 React 组件实时渲染？
   - 倾向：v1 只做静态 artifact，v2 加动态。
6. **认证方式**：SSO 还是自建账号？
   - 倾向：v1 自建账号（开发用），v2 加 SSO。
7. **部署形态**：单机 docker-compose 还是 K8s？
   - 倾向：开发用 compose，生产 K8s。

## 8. 调研结论

### 8.1 项目定位

**本项目 = 多租户 Web 平台 + Agent Engine + 长程任务调度 + 容器沙箱 + Live Artifacts**。

不是 CLI 的简单复刻，而是把 Claude Code 的 Engine 层抽出来，套上 Web UI + 多租户基础设施，再加上长程任务的状态机驱动。

### 8.2 核心创新点（相对 Claude Code CLI）

1. **多租户**：从单用户到多租户，权限、配额、隔离全部重做；
2. **Web UI**：流式输出 + Artifacts + 任务时间线；
3. **真正持久化**：跨机器、跨重启的 session 恢复（CLI 做不到）；
4. **长程任务 UI**：feature_list + progress + 自动化测试，全部可视化；
5. **可观测性**：tracing / metrics / logs 三件套，企业级。

### 8.3 实施路线建议

```
Phase 0（基础）：
  - 项目骨架（Spring Boot 4 + WebFlux + 前端）
  - 单租户 / 单 session / 单工具（echo）跑通
  - WebSocket 流式输出

Phase 1（MVP）：
  - Agent Loop + Tool Registry + Permission Middleware
  - JSONL session 持久化
  - 简单 Artifacts 渲染（代码 / Markdown）
  - Docker 沙箱

Phase 2（长程任务）：
  - feature_list + progress 模型
  - Cron + Background Session
  - 状态机驱动的任务调度
  - Playwright 测试集成

Phase 3（多租户 + 企业）：
  - 接 ai-serving 推理网关 / 多租户
  - 配额 / 成本治理
  - SSO / 审计 / Hooks
  - K8s 部署 + 高可用

Phase 4（多 agent）：
  - subagent 委派
  - 角色分工（Planner / Coder / Tester）
  - 任务依赖 DAG
```
