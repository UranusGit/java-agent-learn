| 场景 | 后果 |

## 1. 数据分类

### 1.1 分类标准

| 等级 | 类型 | 例子 | 加密 | 审计 | 保留 |
|------|------|------|------|------|------|
| L4 | 极敏感 | 密码 / 凭据 / 私钥 | 强制 + KMS | 全操作 | 按法规 |
| L3 | 敏感 PII | 邮箱 / 手机 / 身份证 | 强制 | 全操作 | 按 DPA |
| L2 | 业务数据 | session / artifact / message | 静态加密 | 写操作 | 按租户配置 |
| L1 | 配置 | 项目名 / slug | 默认 | 变更 | 短期 |
| L0 | 公开 | 公共分享链接内容 | 不加密 | 不审计 | 按用户 |

## 2. 加密

### 2.1 传输加密

- 全站 HTTPS(TLS 1.3,禁用 1.0 / 1.1 / 1.2 中的弱算法);
- 内部服务间也 TLS(mTLS,backend ↔ DB ↔ Redis ↔ ai-serving);
- WebSocket 用 wss://;
- HSTS:`Strict-Transport-Security: max-age=31536000; includeSubDomains; preload`;
- 证书自动续期(Let's Encrypt / ACM)。

### 2.2 静态加密

| 存储 | 加密 |
|------|------|
| Postgres | TDE(Transparent Data Encryption) / 磁盘级 LUKS / AWS RDS Encryption |
| Redis | 磁盘加密(AWS ElastiCache at-rest) |
| S3 / MinIO | SSE-KMS(每对象独立 data key) |
| Backup | 加密上传 |

### 2.3 字段级加密(敏感字段)

```java
// 本代码仅作学习材料参考
@Component
public class FieldCipher {

    private final KmsClient kms;

    // DEK cache:每条数据用独立 data key,DEK 用 KMS 包裹后存
    public String encrypt(String plaintext, String tenantId) {
        // 1. 拉 / 生成 DEK
        byte[] dek = dekCache.get(tenantId, () -> {
            GenerateDataKeyResponse r = kms.generateDataKey(tenantId);
            return r.plaintext();
        });

        // 2. AES 加密
        byte[] iv = secureRandom(12);
        byte[] cipher = aesGcmEncrypt(dek, iv, plaintext.getBytes(UTF_8));

        // 3. 拼装:iv + cipher + tag,base64
        return Base64.getEncoder().encodeToString(
            ByteBuffer.allocate(iv.length + cipher.length)
                .put(iv).put(cipher).array()
        );
    }

    public String decrypt(String ciphertext, String tenantId) {
        byte[] all = Base64.getDecoder().decode(ciphertext);
        byte[] iv = Arrays.copyOfRange(all, 0, 12);
        byte[] cipher = Arrays.copyOfRange(all, 12, all.length);
        byte[] dek = dekCache.get(tenantId, ...);
        return new String(aesGcmDecrypt(dek, iv, cipher), UTF_8);
    }
}
```

## 3. 密钥管理

### 3.1 KMS 架构

```
┌────────────────────────────────────────┐
│ KMS Provider(AWS KMS / Vault / 自建)    │
│                                        │
│  CMK(Customer Master Key)              │
│    ├─ Tenant CMK(per tenant)           │
│    └─ System CMK(全局)                  │
│        │                               │
│        ▼ generateDataKey                │
│  DEK(Data Encryption Key)              │
│    └─ per record / per file            │
└────────────────────────────────────────┘
```

### 3.2 密钥层级

| 层 | 用途 | 数量 | 轮转 |
|----|------|------|------|
| CMK | 包裹 DEK | 每 tenant 1 个 | 1 年 |
| Tenant DEK | 加密 tenant 数据 | 每 tenant 几个 | 90 天 |
| Session DEK | 加密单 session | 每 session 1 个 | 不轮转 |
| Token signing | 签 JWT | 1 个 | 30 天 |
| Cookie encryption | 加密 cookie | 1 个 | 30 天 |

### 3.3 密钥轮转

```java
// 本代码仅作学习材料参考
@Scheduled(cron = "0 0 4 1 * *")  // 每月 1 号
public void rotateTenantKeys() {
    for (UUID tenantId : tenantRepo.findAll()) {
        KmsKey old = keyRepo.findActive(tenantId);
        KmsKey newKey = kms.createKey(tenantId);

        // 新写用新 key
        keyRepo.activate(tenantId, newKey.getId());

        // 异步 re-encrypt 老数据
        reencryptService.submit(tenantId, old.getId(), newKey.getId());
    }
}

// 后台 re-encrypt
public void reencrypt(UUID tenantId, UUID oldKeyId, UUID newKeyId) {
    int offset = 0;
    while (true) {
        List<UserEntity> batch = userRepo.findByKmsKey(oldKeyId, offset, 100);
        if (batch.isEmpty()) break;

        for (UserEntity u : batch) {
            String plain = fieldCipher.decrypt(u.getEmail(), oldKeyId);
            String recipher = fieldCipher.encrypt(plain, newKeyId);
            u.setEmail(recipher);
            u.setKmsKey(newKeyId);
        }
        userRepo.saveAll(batch);
        offset += 100;
    }
}
```

### 4.3 审计 Service

```java
// 本代码仅作学习材料参考
@Component
public class AuditLogger {

    private final BlockingQueue<AuditEvent> buffer = new LinkedBlockingQueue<>(100_000);

    public void log(String action, Object details) {
        AuditEvent e = AuditEvent.builder()
            .tenantId(currentTenantId())
            .actorId(currentUserId())
            .actorType(currentActorType())
            .action(action)
            .ip(currentIp())
            .userAgent(currentUserAgent())
            .requestId(currentRequestId())
            .details(details)
            .severity("info")
            .createdAt(Instant.now())
            .build();

        if (!buffer.offer(e)) {
            // 队列满了 → 直接同步落库 + 告警
            emergencyFlush(e);
        }
    }

    @Scheduled(fixedRate = 1_000)
    public void flush() {
        List<AuditEvent> batch = new ArrayList<>(500);
        buffer.drainTo(batch, 500);
        if (!batch.isEmpty()) {
            auditRepo.batchInsert(batch);
            // 同步到不可篡改存储(S3 + Object Lock)
            s3AuditSink.archive(batch);
        }
    }
}
```

// 本代码仅作学习材料参考
@PostMapping("/api/users/me/gdpr-delete")
public ResponseEntity<?> gdprDelete(@RequestBody GdprDeleteRequest req) {
    // 双重确认
    if (!req.confirmPassword().equals(...)) {
        throw new BadRequestException();
    }

    UUID userId = currentUserId();
    GdprRequest r = GdprRequest.builder()
        .id(UUID.randomUUID())
        .userId(userId)
        .status("pending")
        .verificationPeriod(Instant.now().plus(14, ChronoUnit.DAYS))  // 14 天反悔期
        .build();
    gdprRepo.save(r);

    // 立即登出
    tokenService.revokeAll(userId);
    mailService.sendGdprNotice(userId, r.getVerificationPeriod());

    return ok("我们将在 14 天后删除您的所有数据");
}

@Scheduled(cron = "0 0 3 * * *")
public void executeGdprDeletion() {
    for (GdprRequest r : gdprRepo.findPendingPastVerification()) {
        gdprService.deleteAll(r.getUserId());
        r.setStatus("completed");
        gdprRepo.save(r);
    }
}

public void deleteAll(UUID userId) {
    // 1. 业务数据
    sessionRepo.deleteByOwner(userId);
    messageRepo.deleteByUser(userId);
    artifactRepo.deleteByUser(userId);
    memoryRepo.deleteByUser(userId);

    // 2. 审计日志 anonymize(法律要求保留审计,但去掉 PII)
    auditRepo.anonymizeActor(userId);

    // 3. 计费记录(保留 7 年法律规定)
    invoiceRepo.anonymizeUser(userId);

    // 4. SSO
    externalIdentityRepo.deleteByUser(userId);

    // 5. 用户
    userRepo.deleteById(userId);

    // 6. 备份(下次轮转自然消失,或主动标记 purge)

## 6. 合规认证

### 6.1 SOC 2 Type II

要求:
- 完整的安全策略文档;
- 访问控制(本章 §4);
- 加密(本章 §2);
- 监控告警(17 章 + 23 章);
- 变更管理(27 章 CI/CD);
- 事件响应(本章 §7);
- 业务连续性(27 章灾备);
- 审计日志(本章 §4);
- 持续运营(连续 6-12 个月观察期)。

### 6.2 ISO 27001

要求:
- ISMS(信息安全管理体系);
- 风险评估 + 处置;
- 14 个控制域(人力 / 物理 / 通信 / 访问 / 加密 / ...);
- 内部审计 + 持续改进。

### 6.3 HIPAA(医疗)

要求:
- BAA(业务伙伴协议);
- PHI(受保护健康信息)加密;
- 审计日志保留 6 年;
- 风险评估年度;
- 物理安全(数据中心);
- 员工培训。

### 6.4 FedRAMP(美国联邦)

要求极严,一般 SaaS 不做。

### 6.5 国内合规

- **等级保护**(等保 2.0 / 3 级):网络安全审查;
- **数据安全法**:数据分类分级;
- **个人信息保护法**(PIPL):用户同意 / 跨境评估;
- **算法备案**:用 LLM 需备案(深度合成算法);
- **数据本地化**:中国用户数据不出境。

## 7. 事件响应

### 7.1 事件分级

| 级别 | 描述 | 响应时间 |
|------|------|---------|
| SEV-0 | 数据泄露 / 全站宕机 | 立即 |
| SEV-1 | 部分功能挂 / 安全漏洞被利用 | 1h |
| SEV-2 | 性能严重退化 / 单租户故障 | 4h |
| SEV-3 | 局部问题 / 体验受损 | 1 工作日 |

### 7.2 流程

```
检测 → 分类 → 隔离 → 通知 → 修复 → 复盘 → 报告
```

### 7.3 事件手册

```java
// 本代码仅作学习材料参考
public class IncidentPlaybook {

    public void onDataBreach(BreachEvent e) {
        // 1. 立即隔离(关闭受影响服务 / 撤销 token / 改密钥)
        if (e.severity() == SEV_0) {
            tokenService.revokeAll();
            kmsService.rotateAll();
        }

        // 2. 评估范围
        BreachScope scope = breachAnalyzer.analyze(e);

        // 3. 通知
        // - 内部:CTO / 安全 / 法务 / PR
        notifyInternal(scope);
        // - 用户:72h 内(SOC2 / GDPR 要求)
        if (scope.affectedUsers().size() > 0) {
            mailService.sendBreachNotice(scope.affectedUsers());
        }
        // - 监管:GDPR 72h 内报 DPA
        regulatorService.reportIfRequired(scope);

        // 4. 修复
        fixVulnerability(e.entry());

        // 5. 复盘
        postmortemService.create(e, scope, actions);

        // 6. 归档到 audit
        audit.logCritical("incident.response", scope);
    }
}
```

## 14. 避坑:数据治理与合规审计常踩的雷

| 雷 | 现象 | 规避 |
|----|------|------|
| KMS 单点 | KMS 挂了所有解密失败 | 多 AZ + 缓存 + 紧急访问 |
| 审计日志膨胀 | 全操作写入撑爆存储 | §4 分级 + 冷归档 + 采样 |
| 误删生产数据 | 保留策略误执行 | §5 软删 + 双人审批 + 沙箱演练 |
| 跨境违规 | 数据出境未告知 | §8 地理分区 + 用户告知 |
| KMS 凭据泄露 | 主密钥失守 | HSM + 多签 + 不导出 |
| 审计被篡改 | 管理员改审计 | §4 WORM 存储 + 独立账户 |
| 用户撤回同意后残留 | 数据散落各处 | §9 全量清单 + 级联 |
| incident 响应慢 | 没演练 + 没联系人 | §7 季度演练 + Runbook |
| 第三方传数据漏 | 没签 DPA | §8 强制协议 + 白名单 |
| 备份未加密 | 备份磁带被脱 | §2 备份同等级加密 |
