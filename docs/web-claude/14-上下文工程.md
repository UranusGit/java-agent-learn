# 14 - 上下文工程：压缩管线 + 工具结果治理 + Prompt 装配

> 本章是 Claude Code 源码 ch05 的核心延伸。04 章只把 Agent Loop 的"骨架"做了出来，
> 但**真正决定长程任务能不能跑下去的，是上下文（context）的处理**——这正是 Anthropic 把
> "Context Engineering"列为 Agent 工程第一性问题的原因。
>
> 完成后：
> - 4 级压缩管线（applyToolResultBudget → snipCompactIfNeeded → microcompact → autocompact）；
> - 工具结果按预算裁剪、按类型压缩；
> - System Prompt 的分层装配（platform / tenant / project / session）；
> - 模型 fallback 与 max_output_tokens 恢复。

---

## 0. 为什么单独拉一章

调研笔记 §2.2 提到了 Claude Code 的 `max_output_tokens` 恢复、模型 fallback、Context Budget，但**没有展开成可实现的工程**。04 章只是把 State + 终止条件做了出来，**完全没有碰上下文压缩**。

结果是：你跑长任务时必然撞上：
- 上下文超长 → 模型报 `max_output_tokens` 错；
- 工具结果（特别是 Bash 输出几 MB 日志）瞬间把上下文打爆；
- autocompact 触发后，agent "忘了"之前做过的关键决定；
- 多 session 长程任务，前一个 session 的关键状态没传递到下一个。

这些问题不在 04 章 / 07 章解决，**整个长程任务就是空架子**。

## 1. 上下文预算的四个阶段

Claude Code 的上下文处理是**四级管线**，每一级处理不同场景：

```
[原始 messages]
       ↓
┌─────────────────────────────────────────┐
│ Stage 1: applyToolResultBudget           │  工具结果单条裁剪
│   - 每条 tool_result 不得超过 N tokens   │
│   - 超过 → 截断 + 保留头尾 + 提示已截断   │
└─────────────────────────────────────────┘
       ↓
┌─────────────────────────────────────────┐
│ Stage 2: snipCompactIfNeeded            │  历史快照压缩
│   - 当前 tokens 接近上限（>80%）          │
│   - 把最早的 K 条消息折叠成一条 summary   │
└─────────────────────────────────────────┘
       ↓
┌─────────────────────────────────────────┐
│ Stage 3: microcompact                    │  块内字段压缩
│   - tool_use 的 input 太长 → 折叠         │
│   - tool_result 内部冗余 → 删除已读确认   │
│   - assistant 消息里的"思考过程"折叠      │
└─────────────────────────────────────────┘
       ↓
┌─────────────────────────────────────────┐
│ Stage 4: autocompact                     │  整体压缩（最后手段）
│   - 全部历史 → 一条 summary message       │
│   - 插入 CompactBoundary 标记             │
│   - 后续拼接不可跨 boundary               │
└─────────────────────────────────────────┘
       ↓
[发给模型]
```

四个阶段**串行执行**，前一级没解决才走下一级。

## 2. Token 计数

没有准确的 token 数，预算管理就是猜。

### 2.1 估算策略（不依赖外部 tokenizer）

```java
// 本代码仅作学习材料参考
@Component
public class TokenEstimator {

    // 经验值：英文 ~4 字符/token，中文 ~1.5 字符/token
    public int estimate(String text) {
        if (text == null) return 0;
        int ascii = 0;
        int nonAscii = 0;
        for (int i = 0; i < text.length(); i++) {
            if (text.charAt(i) < 128) ascii++;
            else nonAscii++;
        }
        return ascii / 4 + (int) (nonAscii / 1.5);
    }

    public int estimate(Message m) {
        int sum = 4;  // role + 结构开销
        for (ContentBlock b : m.content()) {
            if (b instanceof ContentBlock.Text t) sum += estimate(t.text());
            else if (b instanceof ContentBlock.ToolUse tu) {
                sum += 20 + estimate(tu.name()) + estimate(tu.input().toString());
            } else if (b instanceof ContentBlock.ToolResult tr) {
                sum += 20 + estimate(tr.content());
            }
        }
        return sum;
    }

    public int estimateAll(List<Message> msgs) {
        return msgs.stream().mapToInt(this::estimate).sum();
    }
}
```

### 2.2 精确策略（用模型的 tokenizer）

可选：通过推理网关的 `/v1/tokenize` 接口（如果网关提供）。每次 batch tokenize 一批消息，缓存结果。

> v1 用估算即可，v2 升级到精确。

## 3. Stage 1：applyToolResultBudget

**核心问题**：Bash 输出可能几 MB，Read 大文件几千行——这些原始结果不能直接塞进上下文。

### 3.1 单条裁剪策略

```java
// 本代码仅作学习材料参考
@Component
public class ToolResultBudget {

    private static final int PER_RESULT_LIMIT = 8000;   // 单条 8k tokens
    private static final int HEAD = 2000;
    private static final int TAIL = 4000;

    private final TokenEstimator estimator;

    public ContentBlock.ToolResult apply(ContentBlock.ToolResult original) {
        if (estimator.estimate(original.content()) <= PER_RESULT_LIMIT) {
            return original;
        }
        String content = original.content();
        // 保留头 + 尾，中间用占位符
        String head = content.substring(0, findCharOffset(content, HEAD));
        String tail = content.substring(findCharOffsetFromEnd(content, TAIL));
        String truncated = head
            + "\n\n... [truncated " + (content.length() - head.length() - tail.length())
            + " chars] ...\n\n"
            + tail;
        return new ContentBlock.ToolResult(
            original.toolUseId(),
            truncated,
            original.isError()
        );
    }

    private int findCharOffset(String content, int targetTokens) {
        // 近似：4 char/token
        return Math.min(content.length(), targetTokens * 4);
    }

    private int findCharOffsetFromEnd(String content, int targetTokens) {
        return Math.max(0, content.length() - targetTokens * 4);
    }
}
```

### 3.2 按工具类型的差异化裁剪

不同工具的结果裁剪策略不同：

| 工具 | 默认裁剪 | 特殊处理 |
|------|---------|---------|
| Bash | 头尾各保留 | 如果是测试输出，保留所有 FAIL/ERROR 行 |
| Read | 行号保留 | 大文件分段读 |
| Grep | 命中行保留 | hits > 200 → "另有 N 条命中未显示" |
| Edit | 只返回 ok/error | 不返回文件内容（已在 tool_use 里）|
| Write | 只返回 ok | 同上 |

实现：在 `Tool` 接口加 `default ToolResultBudget budget() { ... }` 让工具自定义。

### 3.3 接入位置

在 Agent Loop 把 tool_result 加入 messages 前：

```java
// 本代码仅作学习材料参考
ContentBlock.ToolResult trimmed = toolResultBudget.apply(rawResult);
Message resultMsg = Message.toolResult(parentId, trimmed);
state = state.append(resultMsg);
```

## 4. Stage 2：snipCompactIfNeeded

**触发条件**：当前 messages 总 tokens > 模型上限的 80%。

### 4.1 折叠策略

把最早的 K 条消息（不含 system）合并成一条 summary：

```
[原 messages]
  system: ...
  user: 我叫小张
  assistant: 你好小张
  user: 帮我创建 hello.py
  assistant: [tool_use Write]
  tool_result: ...
  user: 跑一下
  assistant: [tool_use Bash]
  tool_result: ...
  ← 这里是当前上下文边界 →

折叠后：
  system: ...
  system: [SNIPPED] 之前 8 条消息：用户叫小张，已创建 hello.py 并运行通过
  user: 跑一下
  assistant: [tool_use Bash]
  tool_result: ...
```

### 4.2 实现

```java
// 本代码仅作学习材料参考
@Component
public class SnipCompactor {

    private static final double THRESHOLD = 0.8;
    private static final int KEEP_RECENT = 10;

    private final TokenEstimator estimator;
    private final ChatClient summaryClient;  // 用小模型做摘要

    public List<Message> compactIfNeeded(List<Message> msgs, int modelLimit) {
        int used = estimator.estimateAll(msgs);
        if (used < modelLimit * THRESHOLD) return msgs;

        // 取最早的 K 条（保留最近 KEEP_RECENT 条不动）
        int cutAt = msgs.size() - KEEP_RECENT;
        if (cutAt <= 0) return msgs;

        List<Message> toSummarize = msgs.subList(0, cutAt);
        List<Message> toKeep = msgs.subList(cutAt, msgs.size());

        String summary = summarize(toSummarize);
        Message summaryMsg = Message.system(
            "[SNIPPED] 之前 " + toSummarize.size() + " 条消息的摘要：\n" + summary
        );

        List<Message> result = new ArrayList<>();
        result.add(msgs.get(0));  // 保留原始 system
        result.add(summaryMsg);
        result.addAll(toKeep);
        return result;
    }

    private String summarize(List<Message> msgs) {
        // 用小模型（如 haiku）做摘要，prompt:
        // "请用 500 字以内总结以下对话的关键信息和已完成的工作..."
        String conversation = msgs.stream()
            .map(m -> m.role() + ": " + flatten(m))
            .collect(Collectors.joining("\n"));
        return summaryClient.prompt()
            .system("你是对话摘要助手。提取关键事实、决定、未完成任务。简洁。")
            .user(conversation)
            .call()
            .content();
    }

    private String flatten(Message m) {
        return m.content().stream()
            .filter(b -> b instanceof ContentBlock.Text)
            .map(b -> ((ContentBlock.Text) b).text())
            .collect(Collectors.joining());
    }
}
```

### 4.3 关键：保留"系统关键事实"

折叠时不要漏掉关键约束。可以让用户标记某些消息为 `pinned: true`，compactor 跳过：

```java
if (m.pinned()) {
    // 不参与折叠
}
```

例如：用户的原始需求、不可违反的约束、当前任务目标 —— 都应该 pin。

## 5. Stage 3：microcompact

**触发条件**：snip 之后仍超 90% 上限。

不整体折叠，而是**对单条消息内部冗余做清理**：

| 清理项 | 例子 |
|--------|------|
| tool_use 的 input 已被 tool_result 确认 → 替换为 placeholder | `[Bash: 'ls -la' executed]` |
| assistant 消息的"思考过程"（如果有）→ 折叠 | `<thinking>...</thinking>` → `[thought elided]` |
| 同一文件的多次 Read → 只保留最后一次 | 历史 Read 替换为 `[Read a.py earlier]` |
| tool_result 内的"OK"类确认 → 完全删除 | `success` `done` 等无信息量回复 |

实现：遍历每条 message，按规则做内容替换。

```java
// 本代码仅作学习材料参考
@Component
public class Microcompactor {

    public List<Message> apply(List<Message> msgs) {
        // 1. 找出已被确认的 tool_use
        Set<String> confirmedToolUseIds = msgs.stream()
            .flatMap(m -> m.content().stream())
            .filter(b -> b instanceof ContentBlock.ToolResult)
            .map(b -> ((ContentBlock.ToolResult) b).toolUseId())
            .collect(Collectors.toSet());

        // 2. 替换已确认 tool_use 的 input 为占位符
        return msgs.stream().map(m -> {
            if (!"assistant".equals(m.role())) return m;
            List<ContentBlock> newContent = m.content().stream().map(b -> {
                if (b instanceof ContentBlock.ToolUse tu && confirmedToolUseIds.contains(tu.id())) {
                    return new ContentBlock.ToolUse(tu.id(), tu.name(),
                        Map.of("__elided__", "input collapsed after confirmation"));
                }
                return b;
            }).toList();
            return new Message(m.id(), m.parentUuid(), m.role(), newContent,
                m.transition(), m.tokensIn(), m.tokensOut());
        }).toList();
    }
}
```

## 6. Stage 4：autocompact

**触发条件**：前三阶段都走完仍然超 95% 上限。

这是"核弹级"压缩：所有历史（除了原始 system + 最近 1 条 user/assistant）压成一条 summary。**必须插入 CompactBoundary 标记**（07 章已定义），后续拼接时不能跨 boundary。

```java
// 本代码仅作学习材料参考
@Component
public class Autocompactor {

    public List<Message> apply(List<Message> msgs, int modelLimit) {
        // 1. 提取要压缩的部分（除 system 和最近 2 条）
        Message system = msgs.get(0);
        List<Message> tail = msgs.subList(Math.max(1, msgs.size() - 2), msgs.size());
        List<Message> middle = msgs.subList(1, Math.max(1, msgs.size() - 2));

        if (middle.isEmpty()) return msgs;

        // 2. 调小模型压缩
        String summary = summarizeWithHaiku(middle);

        // 3. 拼装
        UUID boundary1 = UUID.randomUUID();
        UUID boundary2 = UUID.randomUUID();
        return List.of(
            system,
            CompactBoundary.compactStart(boundary1),
            Message.system(boundary1, "[AUTO-COMPACT] " + summary),
            CompactBoundary.compactEnd(boundary2),
            ...tail
        );
    }
}
```

**关键设计**：boundary 之后的消息发送给模型时，**前缀要明确告诉模型"你之前有压缩历史，参考 summary"**：

```
system: 你是一个编码 agent。
system: [AUTO-COMPACT] 之前你已完成 F001（用户注册）、F002（登录）...
user: 继续做 F003
```

## 7. 模型 fallback 与 max_output_tokens 恢复

### 7.1 max_output_tokens 恢复

模型偶尔会返回**空内容 + max_output_tokens 标志**（实际是模型想输出但被截断）。Claude Code 的处理：

1. 检测到空内容 + finish_reason = "max_output_tokens"；
2. 调用 `stripSignatureBlocks`：从 messages 末尾移除可能的签名/格式块（这些块是导致截断的常见原因）；
3. 重试一次。

```java
// 本代码仅作学习材料参考
if (response.emptyContent() && "max_output_tokens".equals(response.finishReason())) {
    messages = stripSignatureBlocks(messages);
    retry();
}
```

### 7.2 模型 fallback

如果当前模型连续 N 次失败或超时，降级到更小/更稳定的模型：

```java
// 本代码仅作学习材料参考
@Component
public class ModelFallbackChain {

    private static final List<String> CHAIN = List.of(
        "claude-sonnet-4-6",
        "claude-haiku-4-5",
        "claude-haiku-3-5"
    );

    public String nextModel(String current) {
        int idx = CHAIN.indexOf(current);
        if (idx < 0 || idx >= CHAIN.size() - 1) return null;
        return CHAIN.get(idx + 1);
    }
}
```

fallback 时要：
- 告知前端（UI 显示 "model degraded to X"）；
- 记录审计日志（为什么降级）；
- 不在 fallback 链上来回弹跳（一旦降级，本次会话内保持）。

## 8. System Prompt 的分层装配

### 8.1 四层来源

```
┌──────────────────────────────────────────┐
│ Layer 1: Platform                        │  平台强制（合规、安全）
│   - 禁止生成恶意代码                       │
│   - 必须遵守 tenant 隔离                  │
├──────────────────────────────────────────┤
│ Layer 2: Tenant                          │  租户管理员配置
│   - 公司安全规范                          │
│   - 代码风格约束                          │
├──────────────────────────────────────────┤
│ Layer 3: Project                         │  项目级 CLAUDE.md
│   - 本项目的技术栈、目录结构              │
├──────────────────────────────────────────┤
│ Layer 4: Session                         │  会话级（用户、任务、当前阶段）
│   - "你正在做 F003 用户注册功能"          │
│   - 用户上传的临时指令                    │
└──────────────────────────────────────────┘
```

### 8.2 装配顺序

```java
// 本代码仅作学习材料参考
@Component
public class SystemPromptAssembler {

    public String assemble(TenantContext ctx, Session session, Task task) {
        StringBuilder sb = new StringBuilder();

        // Layer 1
        sb.append(loadResource("/prompts/platform.md"));

        // Layer 2
        sb.append(tenantPromptService.get(ctx.tenantId()));

        // Layer 3
        Path claudeMd = session.worktree().resolve("CLAUDE.md");
        if (Files.exists(claudeMd)) {
            sb.append("\n\n## Project Guide\n").append(Files.readString(claudeMd));
        }

        // Layer 4
        if (task != null) {
            sb.append("\n\n## Current Task\n");
            sb.append("Task: ").append(task.title()).append("\n");
            sb.append("Requirement: ").append(task.requirement()).append("\n");
        }

        // Hook 注入的 prompt（12 章）
        for (String injected : ctx.injectedPrompts()) {
            sb.append("\n\n## Injected\n").append(injected);
        }

        return sb.toString();
    }
}
```

### 8.3 Prompt 优先级 vs Token 成本

System prompt 通常每轮都发，token 成本高。优化：
- 静态部分（Layer 1-2）缓存（网关侧 prompt caching）；
- 动态部分（Layer 3-4）每次现拼；
- 合并相同的 `<instructions>` 块。

## 9. 工具结果消息（tool_result）的完整生命周期

### 9.1 完整链路

```
Tool 实际执行
    ↓ 返回原始 ToolResult（可能很大）
ToolResultBudget.apply             ← Stage 1 裁剪
    ↓ 适中大小的 ToolResult
转成 message，加入 State.messages
    ↓
进入下一轮 Agent Loop
    ↓ 检查预算
SnipCompactor.compactIfNeeded      ← Stage 2 折叠
    ↓
Microcompactor.apply               ← Stage 3 块内压缩
    ↓
Autocompactor.apply                ← Stage 4 整体压缩
    ↓
最终发给模型
```

### 9.2 错误结果的处理

工具失败时，tool_result 的 isError=true。要确保：
- 错误信息被裁剪时**保留**关键堆栈（不被头尾截断丢掉）；
- agent 看到 error 后能自纠正（不要把"file not found"压缩成"OK"）；
- 同一工具连续失败 → 标记"可能死循环"，提示 agent 换思路。

```java
// 本代码仅作学习材料参考
if (result.isError()) {
    // 错误结果优先保留，不被 budget 截断
    return result;
}
```

### 9.3 并行工具结果

Claude Code 支持单 turn 内并行多次 tool_use。所有 tool_result 在下一轮前**必须全部到位**。否则模型会"看见"部分结果，产生幻觉。

```java
// 本代码仅作学习材料参考
List<ContentBlock.ToolUse> toolUses = extractToolUses(assistantMsg);
CompletableFuture<Void> allOf = CompletableFuture.allOf(
    toolUses.stream().map(tu -> executeTool(tu)).toArray(CompletableFuture[]::new)
);
allOf.join();  // 等全部完成
// 才能进下一轮
```

孤儿恢复（07 章 `recoverOrphanedParallelToolResults`）就是处理"崩溃时部分 tool_result 没写"的场景。

## 10. 长程任务的上下文交接

### 10.1 跨 session 的状态传递

11 章 harness 模式下，每个 session 都是全新 context。要靠**外部状态**把上一个 session 的关键信息传给下一个：

```
session-N 结束时：
  1. 写 claude-progress.txt（人类可读）
  2. 写 .harness/session-N.summary.json（机器可读）
  3. 更新 feature_list.json 的 passes 字段
  4. git commit + push

session-N+1 启动时：
  1. 读 claude-progress.txt + git log
  2. 读 .harness/session-N.summary.json
  3. 选下一个 feature
  4. 注入 system prompt：包含前 N 个 session 的摘要
```

### 10.2 summary.json 的结构

```json
{
  "session_id": "uuid",
  "feature_done": "F012",
  "commits": ["abc1234", "def5678"],
  "decisions_made": [
    "使用 bcrypt 而不是 argon2",
    "API 路径定为 /api/v1/auth/register"
  ],
  "things_to_remember": [
    "用户的数据库密码必须从 env 读"
  ],
  "next_session_hint": "F013 依赖 F012 的 User model，路径 src/models/User.ts"
}
```

### 10.3 Session 接力 Prompt

下一个 session 的 Layer 4 注入：

你是 Coding Agent。前一个 session 已完成 F012 用户注册。

## 11. 接入 Agent Loop

修改 `AgentLoopV2`，在调模型前过 4 级管线：

```java
// 本代码仅作学习材料参考
private void callModel(List<Message> history, String userInput,
                       FluxSink<State> sink, AbortHandle abort) {
    // 1. 装配 system prompt
    String system = systemPromptAssembler.assemble(ctx, session, task);

    // 2. 4 级压缩管线
    List<Message> compacted = history;
    compacted = snipCompactor.compactIfNeeded(compacted, modelLimit);
    compacted = microcompactor.apply(compacted);
    if (estimator.estimateAll(compacted) > modelLimit * 0.95) {
        compacted = autocompactor.apply(compacted, modelLimit);
    }

    // 3. 调模型（带 fallback）
    callModelWithFallback(system, compacted, userInput, sink, abort);
}

private void callModelWithFallback(String system, List<Message> history,
                                    String userInput, FluxSink<State> sink,
                                    AbortHandle abort) {
    String model = ctx.model();
    int retries = 0;
    while (retries < 3) {
        try {
            doStreamCall(system, history, userInput, model, sink, abort);
            return;
        } catch (MaxOutputTokensExceeded e) {
            history = stripSignatureBlocks(history);
            retries++;
        } catch (ModelUnavailableException e) {
            String next = fallbackChain.nextModel(model);
            if (next == null) throw e;
            model = next;
            sink.next(state.withTransition("model_fallback:" + model));
            retries++;
        }
    }
}
```
