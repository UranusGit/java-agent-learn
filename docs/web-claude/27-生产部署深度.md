## 0. 单机 vs 集群

| 维度 | docker-compose（99 章） | K8s（本章） |
|------|----------------------|-------------|
| 节点数 | 1 | N |
| WebSocket 广播 | 同进程 | 跨节点（Redis pub/sub） |
| 沙箱调度 | 单机 Docker | 跨节点 Docker / 远程沙箱 |

## 3. WebSocket 跨节点广播

### 3.1 问题

99 章是单进程 `Map<sessionId, Set<wsId>>`，K8s 下 5 个 Pod，订阅信息分散。

### 3.2 方案：Redis pub/sub

```java
// 本代码仅作学习材料参考
@Component
public class RedisTopicRegistry implements SessionTopicRegistry {

    private final StringRedisTemplate redis;
    private final String selfPodId = UUID.randomUUID().toString();
    private final Map<String, Set<String>> localSubscribers = new ConcurrentHashMap<>();

    @PostConstruct
    public void init() {
        // 订阅 broadcast channel
        redis.subscribe(this::onBroadcast, "session-events");
    }

    @Override
    public void subscribe(UUID sessionId, String wsId, WebSocketSession ws) {
        localSubscribers.computeIfAbsent(sessionId.toString(), k -> ConcurrentHashMap.newKeySet()).add(wsId);
        localSessions.put(wsId, ws);
        redis.opsForSet().add("subscribers:" + sessionId, selfPodId + ":" + wsId);
    }

    @Override
    public void emit(AgentEvent event) {
        // 1. 本地发
        broadcastLocally(event);
        // 2. 发到 Redis pub/sub 让其他 Pod 收
        redis.convertAndSend("session-events", om.writeValueAsString(event));
    }

    private void onBroadcast(Message msg, byte[] pattern) {
        AgentEvent e = om.readValue(msg.getBody(), AgentEvent.class);
        broadcastLocally(e);
    }

    private void broadcastLocally(AgentEvent e) {
        Set<String> subs = localSubscribers.get(e.sessionId().toString());
        if (subs == null) return;
        for (String wsId : subs) {
            WebSocketSession ws = localSessions.get(wsId);
            if (ws != null && ws.isOpen()) send(ws, e);
        }
    }
}
```

## 4. 沙箱调度

### 4.1 单机 Docker 不够用

K8s 集群里 backend Pod 想跑 Docker（DinD）会引入安全 + 性能问题。两条路：

### 4.2 方案 A：Kubernetes Job per Sandbox

把每个沙箱变成一个 K8s Job：

```java
// 本代码仅作学习材料参考
@Component
public class K8sSandboxManager implements SandboxManager {

    private final KubernetesClient client;

    public Sandbox startForTask(UUID sessionId, UUID taskId) {
        Pod pod = new PodBuilder()
            .withNewMetadata()
                .withName("sandbox-" + sessionId)
                .withLabels(Map.of("app", "webclaude-sandbox", "session", sessionId.toString()))
            .endMetadata()
            .withNewSpec()
                .withRestartPolicy("Never")
                .withActiveDeadlineSeconds(3600L)   // 1h 强制终止
                .addNewContainer()
                    .withName("sandbox")
                    .withImage("web-claude-sandbox:v2")
                    .withCommand("sleep", "infinity")
                    .addNewVolumeMount().withName("workspace").withMountPath("/workspace").endVolumeMount()
                    .withNewResources()
                        .withRequests(Map.of("cpu", new Quantity("500m"), "memory", new Quantity("512Mi")))
                        .withLimits(Map.of("cpu", new Quantity("1"), "memory", new Quantity("1Gi")))
                    .endResources()
                .endContainer()
                .addNewVolume().withNewEmptyDir().withSizeLimit(new Quantity("2Gi")).endEmptyDir().withName("workspace").endVolume()
            .endSpec()
            .build();

        client.pods().resource(pod).create();
        return waitForReady(pod, Duration.ofSeconds(60));
    }

    public void stop(UUID sessionId) {
        client.pods().withLabel("session", sessionId.toString()).delete();
    }
}
```

### 4.3 方案 B：远程沙箱集群

单独跑一组"沙箱专用节点"（带 Docker daemon），backend 通过 gRPC 调用：

```
backend (5 pods) ──┐
                   ├─→ sandbox-cluster (1 master + N workers)
                   │   - master 维护池
                   │   - 每个 worker 跑 N 个 sandbox 容器
```

优点：sandbox 资源完全隔离、可独立扩缩。

### 4.4 沙箱预热池

```java
// 本代码仅作学习材料参考
@Component
public class SandboxWarmPool {

    private final BlockingQueue<Sandbox> pool = new LinkedBlockingQueue<>(10);

    @Scheduled(fixedRate = 5_000)
    public void refill() {
        while (pool.size() < 5) {
            try {
                Sandbox s = sandboxManager.startCold();
                pool.offer(s);
            } catch (Exception ignored) {}
        }
    }

    public Sandbox checkout(Duration timeout) throws InterruptedException {
        Sandbox s = pool.poll(timeout.toMillis(), TimeUnit.MILLISECONDS);
        if (s == null) s = sandboxManager.startCold();  // 池空了现场起
        return s;
    }

    public void returnToPool(Sandbox s) {
        if (!pool.offer(s)) sandboxManager.stop(s.sessionId());
    }
}
```

冷启动 5-10s → 池化后 100ms。

## 9. SLI / SLO

### 9.1 SLI 定义

| SLI | 计算 | 目标 |
|-----|------|------|
| 可用性 | 200 响应 / 总请求 | 99.5% |
| 流式响应延迟 | P95 first token < 1s | 99% |
| WS 连接成功率 | 握手成功 / 握手请求 | 99% |
| 长程任务完成率 | completed / started | 90% |
| 沙箱启动延迟 | P95 < 200ms（热池）| 99% |
| Eval 通过率 | safety suite 100% | 100% |

### 9.2 错误预算

99.5% 可用性 = 每月 21.6 min downtime。超过 → 暂停发版、改专注稳定。

### 9.3 Grafana 仪表板

- 总览：SLO、错误预算、QPS、p95；
- 单租户：sessions / tokens / cost / errors；
- 沙箱：active / cold start / pool size；
- 任务：active / completion rate / avg feature passes；
- Eval：pass rate / regression。

## 10. 灾备（DR）

### 10.1 RTO / RPO

| 数据 | RTO | RPO |
|------|-----|-----|
| Postgres | 5min | 0（同步复制） |
| S3 artifacts | 1h | 0（CRR） |
| Redis | 30min | 60s（AOF + 异步） |
| ConfigMap | 即时 | 0（git） |

### 10.2 Postgres HA

```
primary ──streaming──→ replica1（同步）
        └─streaming──→ replica2（异步）
                  ↓
            Patroni 管理 failover
```

故障切换：primary 挂 → Patroni 提升一个 replica → DNS 切换 → 应用重连。

### 10.3 跨区域 DR

- 主区域：us-east-1
- 备区域：us-west-2（Postgres 异步复制 + S3 CRR）
- DNS 用 Route53 health check，主挂后 5min 切到备

### 10.4 演练

每季度跑一次 DR 演练：

1. 在备区域启一套完整环境；
2. 切流量；
3. 验证；
4. 切回。

```yaml
            / sum(rate(http_requests_total{job="webclaude-backend"}[2m]))
```

错误率 > 1% → 自动回滚。

## 13. 运维手册（Runbook）

### 13.1 故障分级

| 级别 | 描述 | 响应 |
|------|------|------|
| P0 | 服务全挂 / 数据丢失 | 5min |
| P1 | 关键功能挂 / SLO 跌 | 15min |
| P2 | 部分功能异常 | 1h |
| P3 | 体验问题 | 1 工作日 |

### 13.2 常见故障处理

**A. backend pod 大量 OOM**

```
查：kubectl top pod
判断：哪个 session 占内存大（可能 unbounded messages）
处理：限流该 session、扩容、查 14 章压缩是否生效
```

**B. WS 跨节点广播丢消息**

```
查：webclaude_redis_pubsub_lag
判断：Redis 队列积压
处理：扩 Redis、查慢命令
```

**C. 沙箱冷启动暴涨**

```
查：webclaude_sandbox_cold_start_total
判断：池空了
处理：扩预热池大小、查沙箱节点是否被驱逐
```

**D. DB 连接耗尽**

```
查：pg_stat_activity
判断：连接泄漏 / 慢查询
处理：杀长查询、查 N+1、调 HikariCP
```

**E. AI 网关 429 暴涨**

```
查：ai-serving 网关日志
判断：租户 quota 超 / 上游限流
处理：通知 ai-serving team、临时降级到 fallback model（14/18 章）
```

### 13.3 On-call 值班

- 7x24 分组轮值（每组 1 主 1 备）；
- PagerDuty 推送 P0/P1；
- 每个 alert 有对应 runbook 链接。

### 17.3 单租户成本

```sql
-- 本代码仅作学习材料参考
SELECT tenant_id,
       SUM(cost_usd) AS total_cost,
       SUM(usage_tokens) AS total_tokens
FROM sessions
WHERE created_at > NOW() - INTERVAL '30 days'
GROUP BY tenant_id
ORDER BY total_cost DESC;
```
