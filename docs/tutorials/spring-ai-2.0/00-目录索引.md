## 0. 为什么单独搞一个 2.0 目录

Spring AI 2.0 不是 1.0 的小版本升级，而是范式迁移：

| 范式 | 1.0 | 2.0 |
|------|-----|-----|
| Agent Loop | 手写 while + 手动管理 tool calls | `ToolCallingAdvisor` 原生递归迭代 |
| 结构化输出 | `BeanOutputConverter` 转一次 | `StructuredOutputValidationAdvisor` 校验 + 重试 |
| 拦截机制 | `CallAdvisor` / `StreamAdvisor` | 同名但 API 收敛 + 工具链统一 |
| 会话 | `MessageWindowChatMemory` 内存版 | `spring-ai-session` event-sourced 可重放 |
| 工具生态 | `@Tool` 仅本进程 | `@Tool` + `@McpTool` 跨进程跨语言 |
| 可观测 | Micrometer | Micrometer + OTel GenAI Semantic Conventions |

> 旧目录里的"自研 AgentLoop / BeanOutputConverter"在 2.0 都被原生取代。把旧目录当历史，把本目录当未来。

## 2. 三大反复出现的设计原则

整个 2.0 学习会反复强化这三条：

1. **Workflow > Agent**（来自 Anthropic《Building Effective Agents》）—— 能用确定性 DAG 解决的，不要用自主 Agent。11、10 全程体现。
2. **失败要被 LLM 看到**（来自 Claude Code 源码）—— Tool 异常不要抛，要返回 `ToolResult.error(stderr)`，让 LLM 自我修复。04、14 详谈。
3. **三重保护**（maxTurns / 预算 / 死循环检测）—— Agent 必须有终止保护。14 详谈。

## 4. 五大 Workflow 模式学习路径

Anthropic《Building Effective Agents》(2024-12-19) 是企业 Agent 的金科玉律。

```
读 11 理解 5 个模式怎么封装成 Advisor（理论 + API + 综合实战）
   ↓
读 14 知道 5 个 Advisor 怎么防失控（生产化）
   ↓
读 10 知道 5 个 Advisor 撑不住时怎么升级到编排引擎（未来）
```

每个模式都遵循"模式定义 → 适用场景 → 反模式 → Advisor 实现 → 完整 demo → 验收"六段结构。

## 5. MCP 体系学习路径（重点新增）

MCP 是 2026 起 Agent 工具接入的事实标准。本系列把 MCP 主题集中在 05-07，外加 32 综合应用：

```
读 05 理解 MCP 协议（三类能力 + 四种传输 + Server/Client 入门）
   ↓
读 06 学会开发（注解 / Provider / 原生三层 + 鉴权限流可观测）
   ↓
读 07 进阶生态（HTTP API → MCP / Hub / 跨语言 / 性能安全测试）
   ↓
读 32 综合应用（多源检索 Agent 把 MCP 当核心接入层）
```

> MCP 不是"被 CLI 冲击"的边缘技术，反而是 2026 服务端 Agent 工具生态的主流协议。详见 32 篇 §15。
