## 1. 三大 Java LLM 框架对比

### 1.1 简表

| 框架 | 出身 | 特点 | 适合 |
|------|------|------|------|
| **Spring AI** | Spring 官方 | Spring 生态深度集成；API 稳定；2.0 GA 后成熟 | Spring Boot 项目首选 |
| **LangChain4j** | 社区（独立） | LangChain 思路；功能丰富；社区活跃 | 已经在用 LangChain（Python）的团队 |
| **Spring AI Alibaba** | 阿里巴巴 | 国产模型深度支持；Graph 编排；企业级特性 | 国产化场景；需要多 Agent 编排 |

### 1.2 详细对比

| 维度 | Spring AI | LangChain4j | Spring AI Alibaba |
|------|-----------|-------------|-------------------|
| **核心 API 稳定性** | 高（GA） | 中 | 中（迭代快） |
| **Spring 生态** | 原生 | 适配 | 原生 |
| **多 Agent 编排** | 不带 | 简单 | 强（Graph） |
| **RAG 工具链** | 完整 | 完整 | 完整 + 增强 |
| **MCP 支持** | 完整 Server+Client | 部分 | 完整 |
| **国产模型支持** | 通义、DeepSeek 都支持 | 通义、DeepSeek | 通义、DeepSeek、智谱等更强 |
| **可观测性** | OTel 原生 | OTel | OTel + 自家 |
| **文档质量** | 高 | 中（社区） | 中（中英文混合） |
| **生产案例** | 多 | 中 | 国内多 |

### 1.3 Spring AI 2.0 GA 的关键事实

2024 年 5 月 GA 后，Spring AI 已经从"实验性"变成"生产可用"：

- API 稳定承诺（minor 版本兼容）。
- Spring Boot 3.3+ / 4.x 原生支持。
- 与 Spring Cloud / Spring Security / Spring Data 完美集成。
- 多种 VectorStore 适配器（pgvector、Milvus、Redis 等）。

## 2. 选型决策树

```
你是 Spring Boot 项目吗？
├── 否 → 用 LangChain4j 或 Quarkus 生态
└── 是 →
    你需要复杂多 Agent 编排吗？
    ├── 是 → Spring AI + Spring AI Alibaba Graph
    └── 否 →
        你需要深度国产化（信创）吗？
        ├── 是 → Spring AI Alibaba（深度集成通义、达摩院等）
        └── 否 → 纯 Spring AI（GA 稳定）
```

### 2.1 不要混用

**反模式**：项目里同时用 Spring AI 和 LangChain4j。

- 两套抽象不一致（ChatClient vs AiServices）。
- Memory 不互通。
- Tool 定义重复。
- 团队学习成本翻倍。

**例外**：可以混用底层 HTTP Client（如 OkHttp）或独立工具（如 Langfuse Java SDK），但 LLM 抽象层必须统一。

## 3. 什么时候自研

### 3.1 该自研的信号

| 信号 | 说明 |
|------|------|
| 框架抽象阻碍你 | 你需要 LLM 输出流式 token 级控制，但框架只给到消息级 |
| 性能瓶颈在框架本身 | 框架的反射开销 / 序列化在大规模场景下扛不住 |
| 业务模型与框架冲突 | 你的 Agent 模型完全不同（如基于规则的专家系统） |
| 团队有能力 | 你有 LLM 工程师 + 资深 Java 工程师，且业务长期 |
| 极度轻量 | 就一个简单 ChatClient 调用，引入整个框架太重 |

### 3.2 不该自研的信号

| 信号 | 说明 |
|------|------|
| "我想自己实现一个 RAG" | 99% 的自研 RAG 都不如直接用 spring-ai-rag |
| "框架的 Tool 注解不够灵活" | 大概率是没看文档；ToolCallback 接口足够灵活 |
| "我自己写更快" | 短期是，长期维护成本远超你预估 |
| "想学一下原理" | 学习可以自研，但不要上生产 |

### 3.3 折中：框架 + 自研关键模块

```
框架（Spring AI）：ChatClient、Advisor、Tool、Memory
    +
自研：
    ├── 业务 Advisor（如安全、审计、计费）
    ├── 业务工具（业务领域工具）
    ├── 自定义 RAG 管道（如果框架的不够）
    └── 自定义编排（如果 Graph 不够灵活）
```

**关键**：用框架的"骨架"，自研的"肉"。骨架决定集成生态，肉决定业务独特性。

## 4. ADR：记录架构决策

**ADR（Architecture Decision Record）** 是工程化团队的标准实践 —— 把每个关键决策记录下来，未来的人能理解"为什么当时这样选"。

### 4.1 ADR 模板

```markdown
# ADR-001: 选择 Spring AI 作为 LLM 抽象层

**Status**: Accepted
**Date**: 2026-07-17
**Decider**: 张三（架构组）

## Context

公司需要构建内部 AI 平台，支持多个业务方接入。需要选型 LLM 调用、RAG、Tool、Memory 的抽象层。

候选方案：
- Spring AI 2.0（Spring 官方）
- LangChain4j 0.35
- Spring AI Alibaba 1.0
- 自研

## Decision

选择 Spring AI 2.0 + Spring AI Alibaba Graph（编排）。

## Rationale

1. **Spring 生态原生**：公司技术栈 100% Spring Boot，原生集成降低成本。
2. **2.0 GA 稳定**：API 已承诺兼容性，可以长期投资。
3. **MCP Server/Client 完整**：未来 Agent 跨进程互操作需要。
4. **多 VectorStore 适配**：pgvector / Milvus / Redis 都有现成实现。

## Alternatives Considered

- **LangChain4j**：放弃原因：与 Spring 生态集成需要适配，社区文档质量中等。
- **Spring AI Alibaba 全家桶**：放弃原因：纯 Alibaba 锁定，未来切其他模型可能受限。但 Graph 模块单独使用没问题。
- **自研**：放弃原因：投入产出比不合理，3 人月起步。

## Consequences

**正面**：
- 接入 Spring Boot Actuator 即可获得完整可观测性。
- 切换 VectorStore 只改 Bean 配置。
- 未来切换 LLM 供应商（OpenAI → Anthropic）只改依赖。

**负面**：
- 强绑定 Spring 生态（如果未来切 Quarkus 就麻烦）。
- Spring AI 2.0 才刚 GA，部分高级功能（如多 Agent）需要 Alibaba Graph 补充。

## Compliance

- 所有 LLM 调用必须通过 ChatClient，禁止直接调 OpenAI SDK。
- 所有 Tool 必须用 @Tool 注解，禁止裸 ToolCallback。
- 所有 Memory 必须用 ChatMemory 接口，禁止自己维护历史。
```

## 5. 几个经典选型决策（参考）

### 5.1 VectorStore 选型 ADR

```markdown
# ADR-003: 选择 PostgreSQL + pgvector 作为向量库

## Context
需要存储 100 万级文档 chunk，支撑 RAG 检索。候选：pgvector、Milvus、Qdrant、Elasticsearch。

## Decision
pgvector。

## Rationale
1. 100 万规模 pgvector 完全够用（用 HNSW 索引）。
2. 已经在用 PostgreSQL，不增加运维负担。
3. 可以用 RLS 做 row-level 多租户隔离。
4. 关系数据和向量在同一事务，一致性天然保证。

## Consequences
- 规模超过 1000 万时需要迁移到 Milvus。
- 检索性能受 PG 配置影响（需要 tuning shared_buffers、work_mem 等）。
```

### 5.2 多 Agent 编排 ADR

```markdown
# ADR-006: 选择 Spring AI Alibaba Graph 作为多 Agent 编排

## Context
需要 Router + Pipeline 模式的多 Agent 编排。候选：手撸状态机、Alibaba Graph、Spring StateMachine。

## Decision
Spring AI Alibaba Graph。

## Rationale
1. 与 Spring AI ChatClient 原生集成。
2. 状态机式表达直观。
3. 支持条件边、并行节点。
4. 内置 trace / 错误处理。

## Consequences
- 强依赖 Alibaba 维护。
- 简单场景（< 3 节点）反而比手撸复杂。
```

### 5.3 观测平台 ADR

```markdown
# ADR-008: 选择 Langfuse（自托管）作为 LLM 观测平台

## Context
需要 LLM trace、prompt 管理、eval 数据。候选：自建、Langfuse、Datadog LLM、LangSmith。

## Decision
Langfuse 自托管。

## Rationale
1. 开源，数据在自己机房。
2. 业界标杆，社区活跃。
3. Java SDK 直接接 Spring AI Advisor。
4. 未来如果不够用可以平滑迁移到 Datadog LLM。

## Consequences
- 需要自维护 Langfuse 部署（K8s）。
- Java SDK 还在迭代，部分功能需要适配。
```

## 6. 反模式速查

### 6.1 "上来就全家桶"

```xml
<!-- ❌ 反模式：什么都引 -->
<dependency>spring-ai-openai</dependency>
<dependency>langchain4j-openai</dependency>
<dependency>spring-ai-alibaba</dependency>
<dependency>simple-openai</dependency>
```

后果：依赖冲突、行为不一致、维护噩梦。

### 6.2 "过早多模型"

需求还没明确，先引入 5 个模型适配器。

后果：开发复杂度爆炸，模型切换的收益还没出现。

**建议**：先单模型跑通业务，再加多模型。

### 6.3 "过早微服务"

一上来就把每个 Agent 拆成独立服务。

后果：调试地狱、延迟翻倍、运维复杂。

**建议**：单体先跑通，性能 / 团队规模到了瓶颈再拆。

### 6.4 "Prompt 写在代码里且没版本"

后果：改 prompt 要发版，无法 A/B 测试。

见 [`./12-评估闭环与Prompt版本管理.md`](./12-评估闭环与Prompt版本管理.md)。

### 6.5 "没有评估集"

后果：每次改动都靠主观感觉，长期看质量是退化的。

见 [`./12-评估闭环与Prompt版本管理.md`](./12-评估闭环与Prompt版本管理.md)。

## 7. 自研的最小集合

如果你坚持自研，下面是**最小集合**，不至于完全重造轮子：

### 7.1 必须有

- **HTTP Client 封装**：每个 provider 一个 adapter（OpenAI / Anthropic / DeepSeek）。
- **统一响应模型**：把不同 provider 的响应抽象成统一 record。
- **Tool 注册表**：手动维护 `Map<String, ToolCallback>`。
- **Memory**：List<Message>，自己写 append / truncate。

### 7.2 强烈建议直接用框架

- **VectorStore**：开源的都成熟，自研不划算。
- **RAG 编排**：`RetrievalAugmentationAdvisor` 已经够好。
- **Observability**：OTel 集成自己写很费时间。

### 7.3 自研的核心成本

| 模块 | 工时估算 |
|------|---------|
| HTTP 适配（3 个 provider） | 2 周 |
| 统一响应模型 | 1 周 |
| Tool 注册 + 调用循环 | 2 周 |
| Memory + 持久化 | 1 周 |
| Advisor 链 | 2 周 |
| RAG 检索 + 重组 | 3 周 |
| OTel 集成 | 2 周 |
| **合计** | **~3 人月** |

3 人月起步，这就是自研的代价。如果你的业务独特性收益 > 3 人月 + 长期维护成本，才考虑自研。

## 8. 国产化场景的特殊考虑

### 8.1 模型选型

国内主流：

- **通义千问**（阿里）：API 兼容 OpenAI 格式，国产第一梯队。
- **DeepSeek**（深度求索）：开源模型 + API，性价比极高。
- **智谱 GLM**（清华系）：API + 开源。
- **文心一言**（百度）：API。
- **Moonshot Kimi**：长文本能力强。
- **自托管**：Llama 3 / Qwen2 / DeepSeek 自托管，完全私有化。

### 8.2 选型考虑

| 场景 | 推荐 |
|------|------|
| 通用对话（中文） | DeepSeek-V3 / 通义 Qwen-Max |
| 代码生成 | DeepSeek-Coder / Claude |
| 长文档处理 | Kimi（200 万 context） |
| 严格合规（不出境） | 自托管 Qwen2 / DeepSeek |
| 嵌入模型 | bge-large-zh（开源） / 通义 Embedding |

### 8.3 Spring AI 接入国产模型

DeepSeek 用 OpenAI 兼容 API：

```yaml
spring:
  ai:
    openai:
      api-key: sk-xxx
      base-url: https://api.deepseek.com   # 关键：换 base-url
      chat:
        model: deepseek-chat
```

通义千问用 Alibaba 适配器：

```xml
<dependency>
    <groupId>com.alibaba.cloud.ai</groupId>
    <artifactId>spring-ai-alibaba-starter</artifactId>
</dependency>
```

```yaml
spring:
  ai:
    dashscope:
      api-key: sk-xxx
      chat:
        options:
          model: qwen-max
```

## 9. 决策的常见误区

### 9.1 "新框架比老框架好"

不一定。LangChain4j 早期版本（< 0.30）API 不稳定，Spring AI 1.0 也有不少 bug。**GA 后才考虑大规模用**。

### 9.2 "Star 多的就是好"

Star 多 ≠ 生产可用。要看 issues 的回复速度、breaking change 频率、核心团队稳定性。

### 9.3 "大厂在用就一定行"

大厂用≠对你的场景适合。大厂有专门的团队二次开发，你没有。

### 9.4 "性能基准测试说明一切"

很多基准测试是 toy benchmark。真实场景（100 万文档、1000 QPS）行为可能完全不同。

### 9.5 "等下次版本再决定"

技术债越积越多。先用最稳的版本（GA）开始，遇到瓶颈再优化，比"等完美版本"靠谱。

## 10. 实战避坑

### 10.1 "切换框架成本"

**症状**：项目跑了一年后想从 LangChain4j 切到 Spring AI。

**成本**：

- 所有抽象（ChatClient、Tool、Memory）API 不一致，全改。
- 测试要重写。
- 团队重新学习。

**建议**：选型慎重，能不切就不切。如果非要切，写一个适配层逐步替换。

### 10.2 "依赖版本冲突"

**症状**：Spring AI 用 OkHttp 4，业务用 OkHttp 3，启动报错。

**解决**：

- 用 Maven Dependency Plugin 查冲突。
- 在 `<dependencyManagement>` 锁版本。
- 必要时排除依赖。

### 10.3 "配置项被覆盖"

**症状**：application.yaml 配了 `spring.ai.openai.api-key`，启动报"找不到 key"。

**原因**：环境变量 / 命令行 / Nacos 配置中心优先级更高。

**解决**：用 Spring Boot 的配置优先级排查（命令行 > 环境变量 > application.yaml）。

### 10.4 "Bean 注入失败"

**症状**：`ChatClient.Builder` 注入不到。

**原因**：缺 starter 依赖（如 `spring-ai-starter-model-openai`，Spring AI 2.0 starter 命名规范已统一为 `spring-ai-starter-*`）或 Bean 配置冲突。注意：`ToolCallingAdvisor` 的自动注册**不**依赖 `@ConditionalOnMissingBean`，它由 `ChatClient.builder().build()` 兜底注入，与是否有 `@Bean ChatClient` 无关（见 21 篇 §1.3）。
