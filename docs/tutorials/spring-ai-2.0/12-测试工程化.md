# L2 测试工程化（Spring AI 2.0）

> 本文回答：**Java 工程师最擅长的测试，怎么迁移到 LLM 应用？AI 应用到底能不能测？**
>
> 答案：能测，但要分层。单测、集成测、契约测、评估测各有打法。
>
> 前置：[`./02-Tool与AgentLoop.md`](./02-Tool与AgentLoop.md) + [`./11-评估闭环与Prompt版本管理.md`](./11-评估闭环与Prompt版本管理.md)
> 预计：1.5 天

## 0. 认知地图

```
传统应用测试金字塔
    ├── 单元测试（80%）
    ├── 集成测试（15%）
    └── 端到端（5%）
    ↓
AI 应用测试金字塔
    ├── 单元测试（30%）：纯 Java 逻辑（工具方法、Advisor 内部逻辑）
    ├── 集成测试（30%）：Spring AI 装配、Memory 持久化、VectorStore
    ├── 契约测试（10%）：Mock LLM，验证调用方行为
    └── 评估测试（30%）：跑测试集，验证质量（覆盖 14 篇）
```

**核心原则**：**不要为 LLM 的输出做断言**（不稳定），而是**为 LLM 的调用行为做断言**（用了哪个模型、调了哪个工具、prompt 长什么样）。

## 1. 为什么传统单测在 AI 场景失败

### 1.1 反模式：断言 LLM 输出

```java
// ❌ 反模式
@Test
void testChat() {
    String answer = chatClient.prompt().user("今天几号").call().content();
    assertEquals("2026-07-17", answer);   // 失败概率 90%+
}
```

**为什么失败**：

- LLM 输出非确定（同 prompt 不同响应）。
- 时间相关的问题答案每天都在变。
- LLM 表述风格不一（"7月17日" / "2026年7月17日" / "July 17"）。
- 真的调 LLM 慢（秒级），CI 跑不动。

### 1.2 正模式：分层 + Mock

```
单元测试：纯 Java 方法（工具实现、辅助类）—— mock 一切
    ↓
集成测试：Spring AI + Memory + VectorStore —— mock LLM
    ↓
契约测试：验证调用方的 prompt 结构 —— mock LLM
    ↓
评估测试：跑离线集 —— 调真 LLM（手动触发，不进 CI）
```

CI 只跑前 3 层（秒级返回），评估测试手动触发或定时跑。

## 2. 第一层：单元测试（纯 Java 逻辑）

### 2.1 测试什么

- `@Tool` 注解方法的纯业务逻辑（不含 LLM）

- 自定义 splitter / retriever / augmenter 的纯算法部分

### 3.4 Mock LLM 的集成测试

```java
@SpringBootTest
class ChatServiceTest {

        }

    @Autowired
```

`MockChatModel` 实现：

```java
public class MockChatModel implements ChatModel, StreamingChatModel {

    private final String fixedResponse;

    public MockChatModel(String fixedResponse) {
        this.fixedResponse = fixedResponse;
    }

    @Override
    public ChatResponse call(Prompt prompt) {
        ChatMessage message = new AssistantMessage(fixedResponse);
        return new ChatResponse(List.of(new Generation(message)));
    }

    @Override
    public Flux<ChatResponse> stream(Prompt prompt) {
        return Flux.just(new ChatResponse(List.of(
                new Generation(new AssistantMessage(fixedResponse)))));
    }
}
```

## 4. 第三层：契约测试（验证调用方行为）

### 4.1 用 WireMock 录制 LLM 响应

Spring AI 通过 HTTP 调 LLM，可以用 WireMock 拦截：

```xml
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-contract-stub-runner</artifactId>
    <scope>test</scope>
</dependency>
<dependency>
    <groupId>com.github.tomakehurst</groupId>
    <artifactId>wiremock-jre8</artifactId>
    <scope>test</scope>
</dependency>
```

```java
@SpringBootTest
class ChatClientContractTest {

    @RegisterExtension
    static WireMockExtension wireMock = WireMockExtension.newInstance()
            .options(wireMockConfig().dynamicPort())
            .build();

    @TestConfiguration
    static class WireMockLlmConfig {
        @Bean
        @Primary
        public OpenAiChatModel chatModel() {
            OpenAiApi api = OpenAiApi.builder()
                    .baseUrl(wireMock.baseUrl())
                    .apiKey("test-key")
                    .build();
            return OpenAiChatModel.builder().openAiApi(api).build();
        }
    }

    @Test
    void shouldCallLlmWithCorrectPayload() {
        // 1. 录制 stub
        wireMock.stubFor(post(urlEqualTo("/v1/chat/completions"))
                .willReturn(okJson("""
                    {
                      "id": "chatcmpl-test",
                      "object": "chat.completion",
                      "created": 1718000000,
                      "model": "gpt-4o-mini",
                      "choices": [{
                        "index": 0,
                        "message": {"role": "assistant", "content": "测试回复"},
                        "finish_reason": "stop"
                      }],
                      "usage": {"prompt_tokens": 10, "completion_tokens": 5, "total_tokens": 15}
                    }
                    """)));

        // 2. 调用
        String response = chatClient.prompt()
                .system("你是助手")
                .user("你好")
                .call()
                .content();

        // 3. 断言行为
        assertThat(response).isEqualTo("测试回复");

        // 4. 验证请求 payload（契约）
        wireMock.verify(postRequestedFor(urlEqualTo("/v1/chat/completions"))
                .withRequestBody(matchingJsonPath("$.model", equalTo("gpt-4o-mini")))
                .withRequestBody(matchingJsonPath("$.messages[0].role", equalTo("system")))
                .withRequestBody(matchingJsonPath("$.messages[0].content", equalTo("你是助手")))
                .withRequestBody(matchingJsonPath("$.messages[1].role", equalTo("user")))
                .withRequestBody(matchingJsonPath("$.messages[1].content", equalTo("你好"))));
    }
}
```

**这是 AI 测试的关键技巧**：你**不**断言 LLM 的回复内容（不稳定），而是**断言你发给 LLM 的 payload 长什么样**（稳定，是契约）。

### 4.2 测试 Tool Calling 的循环

```java
@Test
void shouldLoopUntilNoToolCalls() {
    // 第一次 LLM 返回要调工具
    wireMock.stubFor(post(urlEqualTo("/v1/chat/completions"))
            .inScenario("ToolLoop")
            .whenScenarioStateIs("Started")
            .willReturn(okJson(toolCallResponse("get_weather", "{\"city\":\"北京\"}")))
            .willSetStateTo("ToolCalled"));

    // 第二次（工具调用后）返回最终答案
    wireMock.stubFor(post(urlEqualTo("/v1/chat/completions"))
            .inScenario("ToolLoop")
            .whenScenarioStateIs("ToolCalled")
            .willReturn(okJson(finalResponse("北京今天 25 度晴"))));

    String answer = chatClient.prompt()
            .user("北京天气")
            .tools(weatherTools)
            .call()
            .content();

    assertThat(answer).isEqualTo("北京今天 25 度晴");

    // 验证调了两次 LLM
    wireMock.verify(2, postRequestedFor(urlEqualTo("/v1/chat/completions")));
}
```

## 11. 测试覆盖率与质量门禁

### 11.1 不要强求高覆盖率

```xml
<plugin>
    <groupId>org.jacoco</groupId>
    <artifactId>jacoco-maven-plugin</artifactId>
    <configuration>
        <rules>
            <rule>
                <element>BUNDLE</element>
                <limits>
                    <limit>
                        <counter>LINE</counter>
                        <value>COVEREDRATIO</value>
                        <minimum>0.60</minimum>  <!-- 60% 就够，别追求 100% -->
                    </limit>
                </limits>
            </rule>
        </rules>
    </configuration>
</plugin>
```

**理由**：

- Advisor 链里的很多代码是"集成"而非"单元"，集成测覆盖率高但单测覆盖率低。
- LLM 调用本身没法单测，强行 mock 反而测试不可信。
- 强求 100% 会让团队写出无意义的"覆盖率测试"。

### 11.2 真正的质量门禁

| 指标 | 目标 |
|------|------|
| 工具方法单测覆盖率 | > 80% |
| Advisor 内部逻辑覆盖 | > 70% |
| 关键场景集成测试 | 100% 通过 |
| 评估测试（manual trigger） | 不退步于 baseline |
| 契约测试 | 关键 prompt 全部覆盖 |

## 12. CI/CD 集成

### 12.1 GitHub Actions 示例

```yaml
# .github/workflows/ci.yml
name: CI

jobs:
  unit-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-java@v4
        with: { java-version: 21, distribution: temurin }
      - run: ./mvnw test

  integration-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-java@v4
        with: { java-version: 21, distribution: temurin }
      - run: ./mvnw test -Dgroups=integration
      # Testcontainers 自动复用 GitHub Actions 的 Docker

  eval-test:
    runs-on: ubuntu-latest
    if: github.event_name == 'schedule'  # 只在 nightly 跑
    steps:
      - uses: actions/checkout@v4
      - run: RUN_EVAL=true ./mvnw test -Dgroups=eval
        env:
          OPENAI_API_KEY: ${{ secrets.OPENAI_API_KEY }}
```

### 12.2 分层执行

```xml
<!-- 单元：fast -->
 mvn test

<!-- 集成：慢但稳定 -->
mvn test -Dgroups=integration

<!-- 评估：慢且要 API key -->
mvn test -Dgroups=eval
```

## 13. 实战避坑

### 13.1 "测试集污染了生产知识库"

**症状**：测试跑完，生产 RAG 多了一堆测试文档。

**原因**：测试用的 VectorStore 和生产共享了 DB。

**解决**：

- 测试必须用独立的 DB（Testcontainers）。
- 每个测试方法用唯一 namespace / collection。
- `@AfterEach` 清理。

### 13.2 "WireMock 录制的响应不生效"

**症状**：stub 配了但请求没被拦截。

**原因**：

- `base-url` 没指向 wireMock。
- 用了 HTTPS 强制证书校验。

**解决**：

- 注入测试专用的 ChatModel Bean，强制 base-url。
- HTTPS 场景用 `httpDisabled` 或自签证书。

### 13.3 "Testcontainers 启动慢"

**症状**：本地跑测试要 30s+。

**解决**：

- 启用 Testcontainers reuse：`.withReuse(true)`。
- `~/.testcontainers.properties` 加 `testcontainers.reuse.enable=true`。
- 开发期本地常驻，CI 每次新启。

### 13.4 "Mock LLM 和真 LLM 行为不一致"

**症状**：mock 测试全过，上线还是挂。

**原因**：mock 不模拟流式、不模拟 tool call、不模拟 token 限制。

**解决**：

- 写一个更真实的 mock：支持 tool call 循环、流式 chunk。
- 关键场景 nightly 跑真 LLM 集成测（成本可控）。

### 13.5 "评估测试集被记忆"

**症状**：评估指标持续上涨但实际体验下降。

**原因**：你看着测试集改 prompt，prompt 记住了这 100 条 case 的具体形态。

**解决**：

- 测试集分 dev set 和 holdout set（14 篇）。
- 定期从生产流量抽样新增 case。
- 评估用 LLM-as-Judge，不要用 exact match。

### 13.6 "断言 LLM 输出导致测试 flaky"

**症状**：CI 时好时坏。

**解决**：

- 永远不断言 LLM 输出的具体内容。
- 改为断言：包含关键词、长度在范围、结构化字段类型正确、JSON 合法。

```java
// ❌ 反模式
assertThat(answer).isEqualTo("北京今天 25 度");

// ✅ 正模式
assertThat(answer).containsAnyOf("25", "25度", "25 度");
assertThat(answer.length()).isBetween(10, 100);
```
