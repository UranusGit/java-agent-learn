> 本文回答：**LLM 应用从"功能"到"系统"该怎么设计？怎么和传统领域驱动设计（DDD）结合？**

## 0. 认知地图

```
传统系统设计：CRUD + RPC + DDD
    +
LLM 特性：
    ├── 非确定性（同输入不同输出）
    ├── 长延迟（秒级 vs 毫秒级）
    ├── 高成本（每次调用烧钱）
    ├── 状态依赖（Memory / Context）
    └── 工具副作用（Agent 会改外部状态）
    ↓
AI 原生架构模式：
    ├── Event Sourcing（事件溯源）
    ├── CQRS（命令查询职责分离）
    ├── 领域 Agent 边界（Domain Agent）
    ├── 异步优先（Streaming + Queue）
    └── 可观测性优先（Trace + Eval）
```

## 1. LLM 应用与传统应用的本质差异

### 1.1 五个核心差异

| 维度 | 传统 Web 应用 | LLM 应用 |
|------|-------------|---------|
| **确定性** | 同输入必同输出 | 同输入可能差很多 |
| **延迟** | 100ms 内 | 1-30s |
| **成本** | 几乎免费 | 每次调用烧钱 |
| **状态** | 多数无状态 | 强 stateful（Memory） |
| **副作用** | 程序员控制 | LLM 自己决定调工具 |

### 1.2 架构含义

| 差异 | 架构调整 |
|------|---------|
| 非确定性 | 输出必须校验；用 schema 强约束；保留多版本输出 |
| 长延迟 | 流式优先；UI 必须支持加载状态；超时兜底 |
| 高成本 | 缓存是核心；配额；监控 |
| 状态依赖 | 持久化 Memory；session 管理；CQRS 分离读写 |
| 副作用 | 工具沙箱；审计；幂等设计 |

## 2. AI 应用的分层架构

### 2.1 经典四层

```
┌─────────────────────────────────────┐
│ 5. 接入层（API / Web / SDK）         │
├─────────────────────────────────────┤
│ 4. Agent 编排层（Graph / Router）    │
├─────────────────────────────────────┤
│ 3. AI 能力层（LLM / RAG / Tool）     │
├─────────────────────────────────────┤
│ 2. 数据层（Vector / Memory / SQL）   │
├─────────────────────────────────────┤
│ 1. 基础设施（Model Provider / GPU）   │
└─────────────────────────────────────┘
```

### 2.2 领域 Agent 的边界

借鉴 DDD 的 Bounded Context，给每个业务领域一个 Agent：

```
订单领域 → OrderAgent
  - 工具：getOrderStatus / submitRefund / trackShipment
  - 数据：order 表、shipment 表
  
客户领域 → CustomerAgent
  - 工具：getCustomer / updateProfile
  - 数据：customer 表
  
库存领域 → InventoryAgent
  - 工具：checkStock / reserveStock
  - 数据：inventory 表
```

**边界原则**：

1. 一个 Agent 只管一个领域，工具不跨界。
2. 跨领域操作由上层编排（如 Supervisor）。
3. 每个 Agent 独立评估、独立监控、独立演进。
4. Agent 之间通过 MCP 或消息通信，不直接共享数据库。

### 2.3 反模式：上帝 Agent

```java
// ❌ 反模式：一个 Agent 干所有事
@Component
public class GodAgent {
    @Tool public Order getOrder() {...}
    @Tool public Customer getCustomer() {...}
    @Tool public void updateInventory() {...}
    @Tool public void sendEmail() {...}
    @Tool public void generateReport() {...}
    @Tool public void authenticateUser() {...}
    // ... 50 个工具
}
```

**问题**：

- Prompt 容量爆炸（50 个工具 schema）。
- LLM 决策变差（选错工具）。
- 团队协作冲突（所有人改同一个文件）。
- 无法独立演进。

## 3. Event Sourcing：天然适合 LLM 应用

### 3.1 为什么 LLM 应用适合 Event Sourcing

LLM 应用的所有"事件"都是值得记录的：

- 用户输入
- LLM 接收的 prompt
- LLM 的输出
- 调用了哪个工具
- 工具的返回
- 用户的反馈（👍/👎）

**事件流（event stream）= 完整的对话生命周期**。出了问题能精确回放。

### 3.2 事件设计

```java
// org.demo02.ai.event.AiEvents
// 本代码仅作学习材料参考

public sealed interface AiEvent permits
        UserMessageReceived, PromptConstructed, LlmCallStarted,
        LlmCallCompleted, ToolCallRequested, ToolCallCompleted,
        ResponseReturned, UserFeedbackRecorded {

    String sessionId();
    Instant timestamp();
}

public record UserMessageReceived(
        String sessionId, Instant timestamp,
        String userId, String text, Map<String, Object> context
) implements AiEvent {}

public record PromptConstructed(
        String sessionId, Instant timestamp,
        String systemPrompt, String userPrompt, List<String> toolNames
) implements AiEvent {}

public record LlmCallStarted(
        String sessionId, Instant timestamp,
        String model, int inputTokens
) implements AiEvent {}

public record LlmCallCompleted(
        String sessionId, Instant timestamp,
        String model, int outputTokens, long latencyMs, BigDecimal cost
) implements AiEvent {}

public record ToolCallRequested(
        String sessionId, Instant timestamp,
        String toolName, String arguments
) implements AiEvent {}

public record ToolCallCompleted(
        String sessionId, Instant timestamp,
        String toolName, String result, long durationMs, boolean success
) implements AiEvent {}

public record ResponseReturned(
        String sessionId, Instant timestamp,
        String response
) implements AiEvent {}

public record UserFeedbackRecorded(
        String sessionId, Instant timestamp,
        String userId, FeedbackType feedback, String comment
) implements AiEvent {}
```

### 3.3 持久化

```java
// org.demo02.ai.event.AiEventStore
// 本代码仅作学习材料参考

@Repository
public class AiEventStore {

    private final JdbcTemplate jdbc;

    public void append(AiEvent event) {
        jdbc.update("""
            INSERT INTO ai_event (session_id, timestamp, event_type, payload)
            VALUES (?, ?, ?, ?::jsonb)
            """,
                event.sessionId(),
                event.timestamp(),
                event.getClass().getSimpleName(),
                toJson(event));
    }

    public List<AiEvent> replay(String sessionId) {
        return jdbc.query("""
            SELECT payload FROM ai_event 
            WHERE session_id = ? ORDER BY timestamp
            """,
                (rs, rowNum) -> fromJson(rs.getString("payload")),
                sessionId);
    }
}
```

### 3.4 派生视图（CQRS）

事件流是 source of truth，其他数据是派生视图：

```java
// 投影 1：对话视图（给 UI 用）
public class ConversationProjection {
    public ConversationView build(String sessionId) {
        List<AiEvent> events = store.replay(sessionId);
        // 从事件重建对话历史
        return ConversationView.fromEvents(events);
    }
}

// 投影 2：成本视图（给计费用）
public class CostProjection {
    public BigDecimal dailyCost(String tenantId, LocalDate day) {
        // 从 LlmCallCompleted 事件汇总
    }
}

// 投影 3：质量视图（给评估用）
public class QualityProjection {
    public double satisfactionRate(String tenantId) {
        // 从 UserFeedbackRecorded 事件算
    }
}
```

**好处**：

- 加新需求不用改数据库 schema，只加新投影。
- 出问题能从事件流完整重建任意时刻状态。

## 4. 异步优先：流式 + 消息队列

### 4.1 同步 vs 异步

| 模式 | 用户体验 | 实现复杂度 |
|------|---------|-----------|
| 同步 | 等几秒后一次给完整答案 | 简单 |
| 流式 | 立即开始打字 | 中等 |
| 异步队列 | "正在处理，结果稍后推送" | 复杂 |

### 4.2 何时用异步队列

- **任务超过 30s**：浏览器/网关超时。
- **批量任务**：一次跑 1000 个 query。
- **可重试任务**：失败可重跑。

### 4.3 实现示例

```java
// 接收请求，丢队列
@RestController
public class AsyncAgentController {

    private final RabbitTemplate rabbit;

    @PostMapping("/agent/async")
    public String submit(@RequestBody AgentTask task) {
        String taskId = UUID.randomUUID().toString();
        task.setTaskId(taskId);
        rabbit.convertAndSend("agent.tasks", task);
        return taskId;
    }

    @GetMapping("/agent/result/{taskId}")
    public TaskResult poll(@PathVariable String taskId) {
        return taskService.getResult(taskId);
    }
}

// 消费者
@RabbitListener(queues = "agent.tasks")
public class AgentTaskConsumer {

    public void handle(AgentTask task) {
        try {
            String result = agent.run(task.input());
            taskService.saveResult(task.taskId(), result, "success");
            notifyClient(task.taskId(), result);   // SSE / WebSocket 推送
        } catch (Exception e) {
            taskService.saveResult(task.taskId(), null, "error:" + e.getMessage());
        }
    }
}
```

## 6. 与 DDD 的融合

### 6.1 战术设计

| DDD 概念 | LLM 应用对应 |
|---------|------------|
| Aggregate | Agent + 它的工具 + 它的数据 |
| Domain Event | AiEvent（如 ToolCallRequested） |
| Repository | Event Store + 派生视图 |
| Value Object | ToolContext、ChatClientRequest |
| Domain Service | Agent 编排逻辑 |

### 6.2 战略设计

- **Bounded Context** = 一个 Agent。
- **Context Map** = Agent 之间的通信关系（MCP / 消息）。
- **Shared Kernel** = 共享的事件 schema（AiEvent 接口）。
- **Anti-Corruption Layer** = 把外部 LLM 输出转换成领域对象（避免被 LLM 的非确定性污染领域模型）。

### 6.3 一个具体例子：订单退款

**传统实现**：
```java
class RefundService {
    public void refund(String orderId, double amount) {
        Order order = repo.find(orderId);
        if (order.userId() != currentUser()) throw ...;
        if (order.amount() < amount) throw ...;
        paymentGateway.refund(order.paymentId(), amount);
        repo.save(order.markRefunded(amount));
    }
}
```

**AI 原生实现**：
```java
class RefundAgent {
    @Tool(description = "提交退款")
    public String submitRefund(String orderId, double amount, ToolContext ctx) {
        String userId = ctx.get("userId");
        RefundCommand cmd = new RefundCommand(orderId, amount, userId);
        // 领域校验（和传统一样）
        RefundEvent event = refundService.execute(cmd);
        // 发布事件
        eventBus.publish(event);
        return "退款已提交，金额 " + amount;
    }
}

// 用户说"帮我把昨天的订单退了" → Router → RefundAgent
// LLM 自动调 submitRefund，传入 LLM 解析出的 orderId 和 amount
```

**关键区别**：

- 传统：用户必须填表单，参数是死的。
- AI：用户用自然语言说，LLM 解析意图 + 调用工具。
- **但核心领域逻辑（校验、聚合、事务）不变**。

## 7. 可演进性的设计

### 7.1 演进的常见路径

```
v1：单 Agent + 5 工具 + 单模型
    ↓ 工具数膨胀
v2：单 Agent + ToolSearch（自动检索工具）+ 2 模型（路由）
    ↓ 业务领域分明
v3：多 Agent（Router 模式）+ MCP Server 拆分
    ↓ 需要复杂流程
v4：State Graph 编排 + 事件溯源
    ↓ 多租户、规模化
v5：完整平台（监控 / 计费 / 评估 / 自助）
```

### 7.2 保证可演进的几条原则

**1. 接口先行**：先定义 Agent / Tool 的契约，再写实现。换实现不影响调用方。

**2. 不可变事件**：所有事件追加只写（append-only），不修改不删除。需要"修改"就发新事件补偿。

**3. 派生视图分离**：读模型从事件流重建，可以随时加新视图不改写模型。

**4. 能力开关**：新功能用 feature flag 控制，逐步放量。

**5. 评估闭环**：每次演进都用评估集回归（见 14 篇），确保不退步。

### 7.3 反模式：技术债陷阱

```java
// ❌ 反模式：把业务逻辑写在 prompt 里
"""
你是退款助手。退款规则：
- 普通 VIP 用户每天最多退 3 次
- 黄金 VIP 用户每天最多退 10 次
- 大额退款（> 1000）需要经理审批
- 黑名单用户不能退款
"""
```

**问题**：业务规则散落在 prompt 里，改一处要改 prompt，无法单测，无法 audit。

**正模式**：prompt 只负责理解意图，业务规则在 Java 代码：

```java
@Tool(description = "提交退款")
public String submitRefund(String orderId, double amount, ToolContext ctx) {
    String userId = ctx.get("userId");
    // 业务规则在 Java 代码里，可单测
    refundPolicy.check(userId, amount);
    return refundService.execute(orderId, amount, userId);
}
```

## 8. 性能与可扩展性

### 8.1 性能优化方向

| 维度 | 优化手段 |
|------|---------|
| 延迟 | 流式、并行、缓存、便宜模型 |
| 吞吐 | 异步队列、限流、批处理 |
| 成本 | Prompt Cache、复杂度路由、语义缓存 |
| 可用性 | Fallback 链、健康检查、降级 |

### 8.2 横向扩展的难点

LLM 应用的横向扩展比传统应用难：

- **Memory 是有状态的**：sticky session 或集中式 Memory。
- **事件流是写入瓶颈**：append-only 但仍需分片。
- **MCP Server 是有状态的**：连接保持、限流计费。

### 8.3 推荐架构

```
                         ┌──────────────┐
                         │ Load Balancer│
                         └──────┬───────┘
                                ↓
                  ┌──────────────────────────┐
                  │ API Layer（无状态）       │
                  │ ┌─────┐ ┌─────┐ ┌─────┐ │
                  │ │  N1 │ │  N2 │ │  N3 │ │
                  │ └─────┘ └─────┘ └─────┘ │
                  └──────────────────────────┘
                                ↓
                  ┌──────────────────────────┐
                  │ Shared State              │
                  │ - Event Store (PostgreSQL)│
                  │ - Vector Store (pgvector) │
                  │ - Cache (Redis)           │
                  └──────────────────────────┘
                                ↓
                  ┌──────────────────────────┐
                  │ MCP Servers（按领域分片）│
                  └──────────────────────────┘
```

API 层无状态 → 横向扩展。
状态集中在 PostgreSQL + Redis + pgvector。
MCP Server 按领域拆分，独立扩缩。

## 9. 实战避坑

### 9.1 "Memory 用内存 Map，重启就丢"

**解决**：用持久化 ChatMemory（JDBC / Redis）。

### 9.2 "把 prompt 当配置文件随意改"

**问题**：prompt 改了没回归，上线就翻车。

**解决**：见 14 篇 —— Prompt 版本管理 + 评估闭环。

### 9.3 "Agent 直接调数据库表"

**问题**：Agent 知道数据库结构，schema 改了 Agent 就崩。

**解决**：Agent 通过 MCP Server 访问数据，MCP Server 是稳定契约。

### 9.4 "事件流爆炸"

**问题**：每次 LLM 调用都打 10+ 事件，存储成本高。

**解决**：

- 区分 hot event（实时查询）和 cold event（只归档）。
- 冷数据归档到对象存储。
- 大 payload（如完整 prompt）单独存对象，事件里只存引用。

### 9.5 "横切关注点散落"

**问题**：每个 Agent 各自实现日志、监控、限流。

**解决**：

- 用 Advisor 在框架层统一处理（见 12 篇）。
- 用 AOP / Spring Interceptor 在 Controller 层统一鉴权。
