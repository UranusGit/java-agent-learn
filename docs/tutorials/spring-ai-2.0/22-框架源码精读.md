### 1.2 build() 做了什么（理想模型）

下面是**便于理解的简化伪代码**。⚠️ 这只是教学模型，真实行为见 §1.3 勘误。

```java
// 理想模型（简化伪代码，便于记忆）
public ChatClient build() {
    List<Advisor> advisors = new ArrayList<>(this.defaultAdvisors);

    // 自动注册 ToolCallingAdvisor（如果未禁用且用户没自己放）
    if (autoRegisterToolCallingAdvisor && !hasToolAdvisor(advisors)) {
        advisors.add(ToolCallingAdvisor.builder()
                .toolCallingManager(this.toolCallingManager)
                .advisorOrder(BaseAdvisor.HIGHEST_PRECEDENCE + 300)
                .build());
    }

    advisors.sort(Comparator.comparingInt(Advisor::getOrder));
    return new DefaultChatClient(...);
}
```

⚠️ **重要**：上面**没有** "自动注册 ChatMemoryAdvisor" 这一步。Memory Advisor 不会在 build() 里自动注入，必须你自己 `defaultAdvisors(MessageChatMemoryAdvisor.builder(memory).build())`。早期版本的本文和某些社区博客说"`build()` 看到 chatMemory 自动包一层 MemoryAdvisor"是**错误**的。详见 §1.3。

### 1.3 自动注册的真相（已勘误，2026-07-17 复核官方文档）

**官方原文（Spring AI 2.0 Tool Calling 文档）**："`ToolCallingAdvisor`, which is **always auto-registered in the advisor chain (unless explicitly disabled)**"。

也就是说：

- 自动注册发生在 `DefaultChatClient.Builder.build()` 里，**不是** `ChatClientAutoConfiguration`。
- 是否有自定义 `@Bean ChatClient` 不影响自动注册——`build()` 一定会把 `ToolCallingAdvisor` 加进去。
- 全局关闭：`spring.ai.chat.client.tool-calling.enabled=false`。
- 单次关闭：`.advisors(AdvisorParams.toolCallingAdvisorAutoRegister(false))`。

**早期错误说法**："写 `@Bean ChatClient` 会触发 `@ConditionalOnMissingBean` 短路 ToolCallingAdvisor 自动注册"。这是错误归因，社区流传过，**已在 04、02、本文勘误**。

历史上观察到的"自定义 ChatClient 后工具不调"的真实原因请见 [`./04-流式响应与Reactor深度.md` §C.1](./04-流式响应与Reactor深度.md)。

### 1.3.1 修正后的 build() 伪代码

```java
public ChatClient build() {
    List<Advisor> advisors = new ArrayList<>(this.defaultAdvisors);

    // 自动注册 ToolCallingAdvisor（除非该次调用显式禁用）
    if (!toolCallingAdvisorDisabledGlobally && !hasToolAdvisor(advisors)) {
        advisors.add(ToolCallingAdvisor.builder()
                .toolCallingManager(this.toolCallingManager)
                .advisorOrder(BaseAdvisor.HIGHEST_PRECEDENCE + 300)
                .build());
    }

    // 按 order 排序
    advisors.sort(Comparator.comparingInt(Advisor::getOrder));

    return new DefaultChatClient(...);
}
```

注意 ChatMemoryAdvisor **不会**被 `build()` 自动注入（与 ToolCallingAdvisor 不同）——你必须自己 `defaultAdvisors(MessageChatMemoryAdvisor.builder(memory).build())`。

## 2. AdvisorChain：拦截链调度

### 2.1 类图

```
<<interface>> Advisor
    │
    +getName(): String
    +getOrder(): int

<<interface>> BaseAdvisor extends Advisor
    +before(ChatClientRequest, AdvisorChain): ChatClientRequest
    +after(ChatClientResponse, AdvisorChain): ChatClientResponse

<<interface>> CallAdvisor extends Advisor
    +adviseCall(ChatClientRequest, CallAdvisorChain): ChatClientResponse

<<interface>> StreamAdvisor extends Advisor
    +adviseStream(ChatClientRequest, StreamAdvisorChain): Flux<ChatClientResponse>

AdvisorChain（运行时容器）
    ├── advisors: List<Advisor>
    └── next(): Advisor（游标推进）

DefaultAdvisorChain implements AdvisorChain
    ├── nextIndex: int
    └── advisors: List<Advisor>
```

### 2.2 调度核心（简化）

```java
// DefaultAdvisorChain
public class DefaultAdvisorChain implements AdvisorChain {

    private final List<Advisor> advisors;
    private int currentIndex = 0;

    @Override
    public ChatClientRequest beforeNext(ChatClientRequest req) {
        if (currentIndex >= advisors.size()) {
            return req;  // 链走完了
        }
        Advisor current = advisors.get(currentIndex);
        currentIndex++;
        if (current instanceof BaseAdvisor ba) {
            return ba.before(req, this);
        }
        return beforeNext(req);  // 非 BaseAdvisor 跳过
    }

    @Override
    public ChatClientResponse afterNext(ChatClientResponse resp) {
        currentIndex--;
        if (currentIndex < 0) return resp;
        Advisor current = advisors.get(currentIndex);
        if (current instanceof BaseAdvisor ba) {
            return ba.after(resp, this);
        }
        return afterNext(resp);
    }
}
```

**关键设计**：

- `before` 是**正向**遍历（order 从小到大）。
- `after` 是**反向**遍历（order 从大到小）。
- 类似**洋葱模型**（Spring 代理 / Express middleware）。

### 2.3 ChatClient.prompt().call() 的完整链路

```
user call .call()
    ↓
1. 构造 ChatClientRequest（含 prompt、context）
    ↓
2. 遍历 BaseAdvisor 链，每个调 before（修改 request）
    ↓
3. 找到 CallAdvisor（如 ToolCallingAdvisor）
    ↓
4. CallAdvisor.adviseCall 执行：
    a. 调底层 ChatModel.call()
    b. 看返回是否有 tool_calls
    c. 有 → 调工具 → 拼新请求 → 递归 adviseCall
    d. 无 → 返回
    ↓
5. 反向遍历 BaseAdvisor，每个调 after（修改 response）
    ↓
6. 返回 ChatClientResponse
```

### 2.4 设计模式

| 模式 | 应用 |
|------|------|
| Chain of Responsibility | Advisor 链 |
| Strategy | BaseAdvisor / CallAdvisor / StreamAdvisor 不同接口 |
| Decorator | before/after 装饰 request/response |

### 2.5 扩展点

```java
// 实现 BaseAdvisor（最常见）
public class LoggingAdvisor implements BaseAdvisor {
    @Override
    public ChatClientRequest before(ChatClientRequest req, AdvisorChain chain) {
        log.info("before: {}", req.prompt().getUserMessage().getText());
        return req;
    }
    @Override
    public ChatClientResponse after(ChatClientResponse resp, AdvisorChain chain) {
        log.info("after: {}", resp.chatResponse().getResult().getOutput().getText());
        return resp;
    }
    @Override public String getName() { return "LoggingAdvisor"; }
    @Override public int getOrder() { return Integer.MIN_VALUE + 100; }
}

// 实现 CallAdvisor（高级，需要自己处理 ChatModel 调用）
public class MyCallAdvisor implements CallAdvisor {
    @Override
    public ChatClientResponse adviseCall(ChatClientRequest req, CallAdvisorChain chain) {
        // 自定义调用逻辑
        return chain.nextCall(req);
    }
    @Override public int getOrder() { return Integer.MIN_VALUE + 200; }
}
```

## 3. ToolCallingAdvisor：递归迭代的 Agent Loop

### 3.1 类图

```
ToolCallingAdvisor implements CallAdvisor
    │
    +adviseCall(ChatClientRequest, CallAdvisorChain): ChatClientResponse
    │
    ├── 内部状态：
    │   - maxIterations: int  （默认 20）
    │   - toolCallbacks: List<ToolCallback>
    │   - toolContext: ToolContext
    │
    └── 私有方法：
        - executeIteration(): 单次迭代
        - processToolCalls(): 处理 LLM 返回的 tool_calls
        - internalHistoryEnabled: boolean
```

### 3.2 核心源码（简化）

```java
public class ToolCallingAdvisor implements CallAdvisor {

    private static final Logger log = LoggerFactory.getLogger(ToolCallingAdvisor.class);

    @Override
    public ChatClientResponse adviseCall(ChatClientRequest req, CallAdvisorChain chain) {
        int iterations = 0;
        ChatClientRequest current = req;

        while (true) {
            // 1. 调下一层（最终到 ChatModel.call）
            ChatClientResponse response = chain.nextCall(current);

            // 2. 看是否有 tool_calls
            List<ToolCall> toolCalls = extractToolCalls(response);
            if (toolCalls.isEmpty()) {
                return response;  // 没有工具调用，结束
            }

            // 3. 检查 maxIterations
            if (++iterations > maxIterations) {
                log.warn("Max iterations exceeded, returning last response");
                return response;
            }

            // 4. 执行工具
            List<Message> toolResults = new ArrayList<>();
            for (ToolCall call : toolCalls) {
                try {
                    String result = executeToolCall(call);
                    toolResults.add(new ToolResponseMessage(call.id(), result));
                } catch (Exception e) {
                    // 关键设计：失败也返回，让 LLM 看到
                    toolResults.add(new ToolResponseMessage(call.id(),
                            "ERROR: " + e.getMessage()));
                }
            }

            // 5. 把工具结果拼到上下文，进入下一轮
            current = current.mutate()
                    .messages(toolResults)
                    .build();
        }
    }

    private String executeToolCall(ToolCall call) {
        ToolCallback callback = findCallback(call.name());
        if (callback == null) {
            return "ERROR: tool '" + call.name() + "' not found";
        }
        return callback.call(call.arguments(), this.toolContext);
    }
}
```

### 3.3 关键设计点

#### 失败要被 LLM 看到

```java
} catch (Exception e) {
    toolResults.add(new ToolResponseMessage(call.id(),
            "ERROR: " + e.getMessage()));
}
```

不是抛异常，而是**把错误作为 tool result 返回**。LLM 看到"ERROR: ..."会自我修复（改参数、换工具）。

#### 递归 vs 循环

2.0 改成**循环**而不是 1.0 的"用户手写 while"。**好处**：

- 用户不用写 Agent Loop。
- 框架可以加全局控制（maxIterations、budget、loop detection）。
- 与 Advisor 链无缝集成。

#### conversationId 的隐式管理

`ToolCallingAdvisor` 默认会**自己管理 conversation_id**（用于内部工具消息历史）。如果你同时用了 `ChatMemoryAdvisor` 也在管 history，会**重复**。

**坑**：自定义 ChatMemoryAdvisor 的 order 时，要同时调 `ToolCallingAdvisor.builder().disableInternalConversationHistory(true)`。详见 [`./04-流式响应与Reactor深度.md`](./04-流式响应与Reactor深度.md) §7。

### 3.4 设计模式

| 模式 | 应用 |
|------|------|
| Template Method | `adviseCall` 定义循环骨架，子类可改 `executeToolCall` |
| Strategy | `ToolCallback` 是工具执行策略 |
| Decorator | 装饰 ChatModel 调用，注入工具能力 |
| Iterator | 内部 iterations 计数 |

### 3.5 扩展点

```java
// 替换 ToolCallback 解析（高级）
ToolCallingAdvisor.builder()
    .toolCallbacks(customCallbacks)
    .maxIterations(10)
    .disableInternalConversationHistory(true)
    .build();

// 拦截每次工具调用
public class ToolLoggingAdvisor implements BaseAdvisor {
    @Override
    public ChatClientRequest before(ChatClientRequest req, AdvisorChain chain) {
        // 在 ToolCallingAdvisor 之前记录 prompt
        return req;
    }
    // after 时拿到完整工具调用历史
}
```

## 4. Streaming Flux：流式背压

### 4.1 类图

```
<<interface>> StreamingChatModel
    +stream(Prompt): Flux<ChatResponse>

OpenAiStreamingChatModel implements StreamingChatModel
    +stream(Prompt): Flux<ChatResponse>
        └── 内部：WebClient.post().retrieve().bodyToFlux(String.class)
                   .map(sse -> parse(sse))
                   .takeUntil(sse -> "[DONE]".equals(sse))

ChatClient.stream()
    ↓
StreamRequest
    ↓
Flux<ChatClientResponse>
```

### 4.2 核心源码（简化）

```java
// OpenAiStreamingChatModel
public Flux<ChatResponse> stream(Prompt prompt) {
    return Mono.fromCallable(() -> buildRequestBody(prompt))
            .flatMapMany(request -> webClient.post()
                    .uri("/chat/completions")
                    .header("Authorization", "Bearer " + apiKey)
                    .bodyValue(request)
                    .retrieve()
                    .bodyToFlux(String.class)
                    .takeUntil("[DONE]"::equals)
                    .filter(s -> !"[DONE]".equals(s))
                    .map(this::parseChunk)
                    .map(this::toChatResponse))
            // 错误恢复：网络抖动重试 1 次
            .retryWhen(Retry.backoff(1, Duration.ofMillis(500))
                    .filter(this::isTransientError))
            // 超时：整个流最多 60s
            .timeout(Duration.ofSeconds(60));
}
```

### 4.3 背压模型

Reactor Flux 是**冷流**：

- 每个订阅者独立拉取（不是广播）。
- `request(n)` 控制拉多少（默认 unlimited，可能压垮下游）。
- `onBackpressureBuffer(maxSize)` / `onBackpressureDrop()` 控制溢出。

```java
// 控制下游拉取速度
chatClient.stream()
    .chatResponse()
    .onBackpressureBuffer(100, BufferOverflowStrategy.ERROR)
    .subscribeOn(Schedulers.boundedElastic())  // 阻塞操作走弹性线程池
    .publishOn(Schedulers.parallel())           // 后续走并行池
    .subscribe(...);
```

### 4.4 ChatClientMessageAggregator 的精妙

LLM 流式返回的 chunk 不完整（每个 chunk 可能就一两个字）。**问题**：工具调用 JSON 被切到多个 chunk，怎么拼？

```java
// 框架内部（简化）
public class ChatClientMessageAggregator {

    public Flux<ChatClientResponse> aggregate(Flux<ChatClientResponse> rawFlux) {
        // 状态机
        StringBuilder toolCallBuffer = new StringBuilder();
        StringBuilder contentBuffer = new StringBuilder();

        return rawFlux.map(resp -> {
            // 拼文本
            contentBuffer.append(extractDelta(resp));

            // 拼工具调用
            List<ToolCall> partialCalls = extractToolCallDelta(resp);
            for (ToolCall tc : partialCalls) {
                toolCallBuffer.append(tc.arguments());
            }

            return resp;
        }).windowUntil(this::isStreamDone)  // 等到流结束
          .flatMap(window -> window.reduce(this::merge));  // 合并窗口
    }
}
```

详见 [`./04-流式响应与Reactor深度.md`](./04-流式响应与Reactor深度.md) §E。

### 4.5 设计模式

| 模式 | 应用 |
|------|------|
| Observer | Flux 是观察者模式 |
| Strategy | 不同 StreamingChatModel 是不同流源 |
| Decorator | `.map().filter().retry()` 装饰流 |
| State | Aggregator 的拼接状态机 |

### 4.6 扩展点


## 5. ChatMemory：事件溯源的精妙

### 5.1 类图

```
<<interface>> ChatMemory
    +add(String conversationId, List<Message>)
    +get(String conversationId): List<Message>
    +clear(String conversationId)

<<interface>> ChatMemoryRepository
    +saveAll(String conversationId, List<Message>)
    +findByConversationId(String conversationId): List<Message>
    +deleteByConversationId(String conversationId)

JdbcChatMemoryRepository implements ChatMemoryRepository
    ├── jdbcTemplate: JdbcTemplate
    └── dialect: Dialect（PostgreSQL / MySQL / H2）

MessageWindowChatMemory implements ChatMemory
    ├── repository: ChatMemoryRepository
    ├── maxMessages: int
    └── systemPrompt: String  （永远保留，不计入窗口）

ChatMemoryAdvisor implements BaseAdvisor
    ├── chatMemory: ChatMemory
    └── conversationId: String
```

### 5.2 MessageWindowChatMemory 的窗口逻辑

```java
public class MessageWindowChatMemory implements ChatMemory {

    private final ChatMemoryRepository repository;
    private final int maxMessages;
    private final Message systemMessage;

    @Override
    public void add(String conversationId, List<Message> newMessages) {
        List<Message> existing = repository.findByConversationId(conversationId);

        List<Message> all = new ArrayList<>(existing);
        all.addAll(newMessages);

        // 保留 system message + 最近 N 条
        List<Message> kept = new ArrayList<>();
        if (systemMessage != null) kept.add(systemMessage);

        int start = Math.max(0, all.size() - maxMessages);
        for (int i = start; i < all.size(); i++) {
            Message m = all.get(i);
            if (m.getType() != MessageType.SYSTEM) {
                kept.add(m);
            }
        }

        repository.saveAll(conversationId, kept);
    }

    @Override
    public List<Message> get(String conversationId) {
        return repository.findByConversationId(conversationId);
    }
}
```

**关键设计**：

- `systemMessage` 永远保留（不计入窗口）—— 保证人格稳定。
- 窗口外的消息**直接丢弃**（不归档、不摘要）。**这是 2.0 的限制**：长会话需要自己实现摘要。

### 5.3 ChatMemoryAdvisor 的 order 坑

```java
public class ChatMemoryAdvisor implements BaseAdvisor {

    @Override
    public ChatClientRequest before(ChatClientRequest req, AdvisorChain chain) {
        // 读历史
        String convId = (String) req.context().get("conversationId");
        List<Message> history = chatMemory.get(convId);

        // 拼到 prompt
        req.prompt().messages().addAll(0, history);
        return req;
    }

    @Override
    public ChatClientResponse after(ChatClientResponse resp, AdvisorChain chain) {
        // 把这次对话存进 memory
        String convId = (String) resp.context().get("conversationId");
        chatMemory.add(convId, List.of(
                resp.chatClientRequest().prompt().getUserMessage(),
                resp.chatResponse().getResult().getOutput()));
        return resp;
    }
}
```

**order 设计**：

- `ChatMemoryAdvisor` order 一般靠**内层**（接近 ChatModel）。
- 这样 `before` 时其他 Advisor 已经把 prompt 改完了，存进 memory 的才是最终 prompt。
- `after` 时其他 Advisor 已经处理过 response，存进 memory 的也是最终 response。

### 5.4 事件溯源思路（自定义高级）

Spring AI 内置 `ChatMemory` 是**状态存储**（覆盖式）。如果你要做事件溯源：

```java
// 自定义 EventSourcedChatMemory
public class EventSourcedChatMemory implements ChatMemory {

    private final EventStore eventStore;

    @Override
    public void add(String conversationId, List<Message> messages) {
        for (Message m : messages) {
            eventStore.append(new SessionEvent.MessageAdded(
                    conversationId, m, Instant.now()));
        }
    }

    @Override
    public List<Message> get(String conversationId) {
        // 重放所有 MessageAdded 事件
        return eventStore.replay(conversationId).stream()
                .filter(e -> e instanceof SessionEvent.MessageAdded)
                .map(e -> ((SessionEvent.MessageAdded) e).message())
                .toList();
    }
}
```

详见 [`./17-AI原生系统设计.md`](./17-AI原生系统设计.md) Event Sourcing。

### 5.5 设计模式

| 模式 | 应用 |
|------|------|
| Repository | ChatMemoryRepository 抽象存储 |
| Strategy | JDBC / Redis / Cassandra 不同存储 |
| Decorator | MessageWindowChatMemory 装饰 Repository 加窗口 |
| Event Sourcing | 自定义事件溯源版本 |

### 6.2 自动发现流程

```java
// Spring Boot 启动时（简化）
@PostConstruct
public void discoverTools() {
    List<ToolCallback> all = new ArrayList<>();

    for (McpSyncClient client : mcpClients) {
        // 1. 每个 MCP Server 调 listTools()
        ListMcpToolsResult result = client.listTools();

        // 2. 把每个远程 tool 包装成 ToolCallback（2.0 没有 McpToolCallback，用 SyncMcpToolCallback）
        for (Tool def : result.tools()) {
            all.add(SyncMcpToolCallback.builder()
                    .mcpClient(client)
                    .toolDefinition(toToolDefinition(def))
                    .build());
        }
    }

    this.toolCallbacks = all.toArray(new ToolCallback[0]);
}

// SyncMcpToolCallback.call 真正调用远程
@Override
public String call(String args, ToolContext context) {
    Map<String, Object> arguments = parseArgs(args);
    CallToolResult result = mcpClient.callTool(toolDefinition.name(), arguments);
    return serializeResult(result);
}
```

### 6.3 三种传输的源码差异

| 传输 | 类 | 特点 |
|------|----|----|
| stdio | `StdioClientTransport` | 启动子进程，stdin/stdout 通信，JSON-RPC over pipe |
| SSE | `HttpClientSseClientTransport`（client 侧） / `HttpServletSseServerTransportProvider`（server 侧） | 长连接，单向流，需要心跳保活 |
| Streamable HTTP | `HttpClientStreamableHttpTransport`（client） / `HttpServletStreamableServerTransportProvider`（server） | 双向、无状态、可扩缩容（2.0 推荐） |

**关键设计**：上层 `McpSyncClient` 完全屏蔽传输差异。换传输不用改业务代码。

### 6.4 设计模式

| 模式 | 应用 |
|------|------|
| Adapter | SyncMcpToolCallback 把 MCP 协议适配成 ToolCallback |
| Proxy | McpSyncClient 代理远程 Server |
| Factory | Spring Boot 自动装配 McpSyncClient |
| Strategy | 不同 Transport 是策略 |

### 6.5 扩展点

## 7. 横向对比：6 个模块的设计模式总结

| 模块 | 主要模式 | 关键设计 |
|------|---------|---------|
| ChatClient.builder | Builder + Fluent | 自动注册 Advisor（与是否有 `@Bean ChatClient` 无关，见 §1.3） |
| AdvisorChain | Chain of Resp. + Onion | before 正向 / after 反向 |
| ToolCallingAdvisor | Template Method | 失败返回而不是抛，让 LLM 自修复 |
| Streaming Flux | Observer + Decorator | 冷流 + 背压 + Aggregator 状态机 |
| ChatMemory | Repository + Decorator | 窗口裁剪 + system 永远保留 |
| MCP Client | Adapter + Proxy | 传输层抽象，自动发现 |

## 9. 实战避坑

### 9.1 "Advisor 没生效"

**原因**：order 设错了，被另一个 Advisor 跳过了。

**调试**：在 BaseAdvisor.before 第一行打日志，看实际有没有进。

### 9.2 "Tool 调用了一次就结束"

**原因**：maxIterations 默认 20，但你可能用 stream 模式，流被提前关闭。

**调试**：检查 `Flux.timeout` / `take` 配置。

### 9.3 "ChatMemory 只记最后 N 条"

**这是设计**，不是 bug。长会话需要自己实现摘要。

### 9.4 "MCP Server 改了工具但 Client 没刷新"

**原因**：MCP Client 启动时一次性 listTools，运行时不会重新拉。

**解决**：调用方提供 refresh 接口，或重启 Client。

### 9.5 "Streaming chunk 里工具调用 JSON 不完整"

**这是设计**，框架用 Aggregator 拼。**你自己不要再拼一遍**。

详见 [`./04-流式响应与Reactor深度.md`](./04-流式响应与Reactor深度.md) §E。
