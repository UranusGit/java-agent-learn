# L1 初级 - Spring AI 2.0 基础重塑

> 把你脑子里 1.0 的所有 API 假设先放一边，本文逐项对照 1.0 → 2.0 的迁移。
>
> 前置：已学完 [`../../tutorials/spring-ai/`](../spring-ai/) 1.0 系列全部 10 篇
> 预计：半天

---

## 0. 你为什么必须读这一篇

Spring AI 2.0.0 GA（2026-06-12）不是版本号 +1，是**范式迁移**：

| 范式 | 1.0 | 2.0 |
|------|-----|-----|
| Agent 循环 | 自己写 `while (hasToolCalls)` | `ToolCallingAdvisor` 原生递归 |
| 工具自动调用 | 半自动（要写循环） | 全自动（`.call().content()` 直接拿到） |
| 结构化输出 | `BeanOutputConverter` 一次性 | + `StructuredOutputValidationAdvisor` 校验 + 重试 |
| 拦截器 | `CallAdvisor` / `StreamAdvisor` | 同名但签名收敛，引入 `ChatClientRequest/Response` |
| 会话 | `MessageWindowChatMemory` 内存版 | + `spring-ai-session` event-sourced 可重放 |
| 工具生态 | `@Tool` 单进程 | + `@McpTool` 跨进程跨语言 |
| 可观测 | Micrometer | + OTel GenAI Semantic Conventions |

> 1.0 的"自研 AgentLoop"在 2.0 里**整段被原生取代**，直接删掉。

## 1. 版本与依赖切换

### 1.1 pom.xml 关键变更

```xml
<!-- 1.0 旧版 -->
<parent>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-parent</artifactId>
    <version>3.5.x</version>
</parent>
<properties>
    <spring-ai.version>1.0.0</spring-ai.version>
</properties>

<!-- 2.0 新版 -->
<parent>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-parent</artifactId>
    <version>4.0.0</version>
</parent>
<properties>
    <spring-ai.version>2.0.0</spring-ai.version>
    <java.version>21</java.version>
</properties>
```

**强制前置条件**：
1. JDK ≥ 17（推荐 21）
2. Spring Boot 4.x（带 Spring Framework 7）
3. Jackson 3（Spring Boot 4 自带，不需要你手动升级）
4. JSpecify（Spring Boot 4 自带）

### 1.2 starter 名称变化

| 1.0 | 2.0 |
|-----|-----|
| `spring-ai-openai-spring-boot-starter` | **`spring-ai-starter-model-openai`** |
| `spring-ai-ollama-spring-boot-starter` | **`spring-ai-starter-model-ollama`** |
| `spring-ai-anthropic-spring-boot-starter` | **`spring-ai-starter-model-anthropic`** |

> 注意"starter"位置从后缀挪到中间。这是 Spring 官方新约定。

### 1.3 流式必须 webflux

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-webflux</artifactId>
</dependency>
```

不要再用 servlet 写 SSE，2.0 的 `Flux<...>` 必须有 webflux 才能正确序列化。

## 2. ChatClient 的核心 API 变更

### 2.1 调用方式对比

```java
// 1.0：参数 + 返回值有点散
String answer = chatClient.prompt("...")
        .user("你好")
        .call()
        .content();

// 2.0：API 收敛，参数风格统一
String answer = chatClient.prompt()
        .system("...")
        .user("你好")
        .tools(myToolBean)
        .advisors(memoryAdvisor)
        .call()
        .content();
```

表面看差不多，**底层**变了：
- `ChatClientRequest` / `ChatClientResponse` 两个值对象贯穿 Advisor 链
- `AdvisorParams` 提供静态工厂方法（`toolCallingAdvisorAutoRegister(false)` 等）
- `chatClientResponse()` 拿到的是"完整响应 + Advisor 上下文"，不是裸字符串

### 2.2 流式 API（基本不变，但返回类型更丰富）

```java
// 1.0
Flux<String> stream = chatClient.prompt().user(q).stream().content();

// 2.0：三种粒度
Flux<String> tokenStream = chatClient.prompt().user(q).stream().content();           // 纯 token
Flux<ChatResponse> respStream = chatClient.prompt().user(q).stream().chatResponse(); // 带元数据
Flux<ChatClientResponse> fullStream = chatClient.prompt().user(q).stream().chatClientResponse(); // 带 Advisor 上下文
```

**为什么三种**： Advisor 链路要在 `ChatClientResponse` 层生效，token 层只是 LLM 的字面输出。

## 3. ToolCallingAdvisor：删掉你的自研 while

### 3.1 1.0 怎么做的（回忆）

```java
// 1.0：自己写循环
while (true) {
    ChatResponse resp = model.call(prompt);
    if (!resp.hasToolCalls()) return resp;
    ToolExecutionResult r = toolManager.executeToolCalls(prompt, resp);
    prompt = new Prompt(r.conversationHistory());
}
```

### 3.2 2.0 怎么做

```java
// 2.0：自动
String answer = chatClient.prompt()
        .user(q)
        .tools(myToolBean)
        .call()
        .content();
// 内部 ToolCallingAdvisor 自己处理循环
```

**背后发生了什么**：
1. `ChatClient` 构造时**默认自动注册** `ToolCallingAdvisor`，优先级 `Ordered.HIGHEST_PRECEDENCE + 300`
2. 这个 Advisor 在 ChatModel 拿到响应后检查 `hasToolCalls()`
3. 有的话调 `ToolCallingManager.executeToolCalls(...)` 执行
4. 把工具结果拼回 conversation history，**重新发起一次 model call**
5. 循环直到 LLM 不再要工具

### 3.3 关掉自动注册（用于手动模式）

```java
chatClient.prompt()
        .user(q)
        .advisors(AdvisorParams.toolCallingAdvisorAutoRegister(false))
        .call();
```

或全局关闭：
```yaml
spring:
  ai:
    chat:
      client:
        tool-calling:
          enabled: false
```

> 99% 场景用默认自动模式即可。手动模式见 [`./04-流式响应与Reactor深度.md`](./04-流式响应与Reactor深度.md)。

## 4. Advisor 链：API 收敛

### 4.1 1.0 的痛点

```java
// 1.0：CallAdvisor 和 StreamAdvisor 是两个独立接口，签名不一样
public class MyAdvisor implements CallAdvisor, StreamAdvisor {
    @Override public ChatClientResponse adviseCall(ChatClientRequest req, CallAdvisorChain chain) { ... }
    @Override public Flux<ChatClientResponse> adviseStream(ChatClientRequest req, StreamAdvisorChain chain) { ... }
}
```

两个方法逻辑可能重复，但参数类型不一样，没法直接复用。

### 4.2 2.0 的统一

```java
public class MyAdvisor implements CallAdvisor, StreamAdvisor {

    @Override
    public String getName() { return "MyAdvisor"; }

    @Override
    public int getOrder() { return HIGHEST_PRECEDENCE; }

    @Override
    public ChatClientResponse adviseCall(ChatClientRequest request, CallAdvisorChain chain) {
        // 前置逻辑
        ChatClientResponse response = chain.nextCall(request);
        // 后置逻辑
        return response;
    }

    @Override
    public Flux<ChatClientResponse> adviseStream(ChatClientRequest request, StreamAdvisorChain chain) {
        return chain.nextStream(request)
                .doOnNext(resp -> { /* 后置 */ });
    }
}
```

**核心变化**：
- 输入统一是 `ChatClientRequest`
- 输出统一是 `ChatClientResponse`（call）或 `Flux<ChatClientResponse>`（stream）
- `chain.nextCall(req)` / `chain.nextStream(req)` 显式调用下一环
- `getName()` 和 `getOrder()` 是新接口要求

### 4.3 内置 Advisor 的优先级表（重要，决定执行顺序）

| Advisor | Order | 位置 |
|---------|-------|------|
| `MessageChatMemoryAdvisor`（默认） | `HIGHEST_PRECEDENCE + 200` | 最外层（先读 / 最后写 memory） |
| `ToolCallingAdvisor` | `HIGHEST_PRECEDENCE + 300` | 内层 |
| `StructuredOutputValidationAdvisor` | `HIGHEST_PRECEDENCE + 500` | 最内层 |
| `QuestionAnswerAdvisor` / `RetrievalAugmentationAdvisor` | `HIGHEST_PRECEDENCE + 700` | 最内层 |

数字小 = 外层 = 先执行前置、后执行后置。

> ⚠️ 改 Memory 的 order 时务必同时改 `disableInternalConversationHistory()`（见 [`./04-流式响应与Reactor深度.md`](./04-流式响应与Reactor深度.md) §7）。

## 6. 从 1.0 项目升级的实战步骤

### 6.1 顺序

```
1. 升级 JDK 到 21
   ↓
2. 升级 Spring Boot 到 4.x
   ↓
3. 升级 Spring AI BOM 到 2.0.0
   ↓
4. 改 starter 名称（spring-ai-openai-spring-boot-starter → spring-ai-starter-model-openai）
   ↓
5. 删掉所有自研 AgentLoop（while + hasToolCalls）
   ↓
6. 修改自定义 Advisor 的签名（用 ChatClientRequest / Response）
   ↓
7. 测试流式接口（确认 Flux 序列化正常）
   ↓
8. 测试工具调用（确认 ToolCallingAdvisor 自动工作）
```

### 6.2 高频 breaking change 清单

| 你 1.0 的代码 | 2.0 要改成 |
|--------------|-----------|
| `OpenAiApi.builder().baseUrl(...)` | `spring.ai.openai.base-url: ...` 配置即可 |
| `MessageWindowChatMemory.builder().build()`（默认 InMemory） | 同左，API 未变；新增 `sequence_id` 列按 turn 裁剪 |
| 自研 `while (resp.hasToolCalls())` | 直接删掉，依赖 `ToolCallingAdvisor` |
| `Advisor.adviseCall(Prompt, CallAdvisorChain)` | `Advisor.adviseCall(ChatClientRequest, CallAdvisorChain)` |
| `entity(MyType.class)` 单次校验 | + `StructuredOutputValidationAdvisor` 做重试 |

### 6.3 一个常见错误：Jackson 3 反序列化行为变化

Spring Boot 4 强制 Jackson 3，部分字段行为变了：
- `LocalDateTime` 默认序列化为数组（不是 ISO 字符串）
- 未知字段默认抛错（不是忽略）

如果你 Tool 返回 `record` 包含 `LocalDateTime`，给字段加：
```java
@JsonFormat(shape = JsonFormat.Shape.STRING, pattern = "yyyy-MM-dd'T'HH:mm:ss")
LocalDateTime createdAt;
```

或全局：
```java
@Bean
public Jackson2ObjectMapperBuilderCustomizer customizer() {
    return builder -> builder.featuresToDisable(
            com.fasterxml.jackson.databind.DeserializationFeature.FAIL_ON_UNKNOWN_PROPERTIES
    );
}
```
