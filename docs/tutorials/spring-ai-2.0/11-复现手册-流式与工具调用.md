# 复现手册 - Spring AI 2.0 流式 + 工具调用

> 本文目标：照着做就能跑通。不想象，不编造。每一行代码都来自 Spring AI 2.0 官方文档（Tool Calling + ChatClient + Streaming 三页）的 API。
>
> **调研日期**：2026-07-16
> **官方文档版本**：Spring AI 2.0.0 GA（2026-06-12 发布，基于 Spring Boot 4 + Jackson 3 + JSpecify）
> **依赖**：Spring Boot 4.x、Spring AI 2.0.0、JDK 17+（推荐 21）

---

## 0. 三句话先讲清楚原理

1. **自动模式**：`ChatClient` 默认会自动注册一个 `ToolCallingAdvisor`（优先级 `Ordered.HIGHEST_PRECEDENCE + 300`），它帮你处理"LLM 要调工具 → 执行 → 把结果送回去 → 让 LLM 继续"的整个循环，无论 `.call()` 还是 `.stream()` 都透明支持。
2. **流式中的关键坑**：LLM 的一次 tool call 在流式响应里会被切成很多 chunk，单看一个 chunk 是看不到完整 tool call 的。**必须先聚合再判断**，用 `ChatClientMessageAggregator`。
3. **手动模式**：当你想完全控制"几轮 tool 调用就停"或"每轮换不同 model options"时，关掉自动注册（`AdvisorParams.toolCallingAdvisorAutoRegister(false)`），自己写 `while` 循环。

下面分四节落地：依赖 → 自动模式 → 手动模式 → 限制与坑。

## 2. 准备一个 @Tool Bean

```java
package com.example.tool;

import org.springframework.ai.tool.annotation.Tool;
import org.springframework.ai.tool.annotation.ToolParam;
import org.springframework.stereotype.Component;

import java.time.LocalDateTime;
import java.time.format.DateTimeFormatter;

@Component
public class TimeTools {

    @Tool(description = "获取当前系统时间，返回 ISO-8601 格式字符串")
    public String getCurrentTime() {
        return LocalDateTime.now().format(DateTimeFormatter.ISO_LOCAL_DATE_TIME);
    }

    @Tool(description = "计算两个数字相加的结果")
    public double add(
            @ToolParam(description = "第一个加数") double a,
            @ToolParam(description = "第二个加数") double b
    ) {
        return a + b;
    }
}
```

**关键检查项**：
- 包名是 `org.springframework.ai.tool.annotation.Tool`（不是 LangChain4j 的 `dev.langchain4j.agent.tool.Tool`）
- 参数描述用 `@ToolParam`（不是 LangChain4j 的 `@P`）
- 类必须是 Spring Bean（`@Component`），否则 `@Autowired` 拿不到

## 4. 自动模式：`.call()` 同步 + 自动 tool 循环

```java
package com.example.controller;

import com.example.tool.TimeTools;
import org.springframework.ai.chat.client.ChatClient;
import org.springframework.web.bind.annotation.*;

@RestController
@RequestMapping("/auto")
public class AutoController {

    private final ChatClient chatClient;
    private final TimeTools timeTools;

    public AutoController(ChatClient chatClient, TimeTools timeTools) {
        this.chatClient = chatClient;
        this.timeTools = timeTools;
    }

    // 1. 同步调用：LLM 自动调用工具直到拿到最终答案
    @GetMapping("/call")
    public String call(@RequestParam String q) {
        return chatClient.prompt()
                .user(q)
                .tools(timeTools)
                .call()
                .content();
    }
}
```

**curl 测试**：
```bash
curl "http://localhost:8080/auto/call?q=现在几点？然后帮我算一下 12.5 加 7.3 等于多少"
```

**预期**：
1. LLM 返回第一个 chunk 时声明要调用 `getCurrentTime`
2. `ToolCallingAdvisor` 自动执行 `getCurrentTime()`，把结果（"2026-07-16T..."）回传给 LLM
3. LLM 再声明要调用 `add(12.5, 7.3)`
4. Advisor 再次执行，回传 `19.8`
5. LLM 用自然语言总结输出

**日志里能看到的关键证据**：
```
DEBUG o.s.ai.chat.model.ToolCallingManager - Executing tool: getCurrentTime
DEBUG o.s.ai.chat.model.ToolCallingManager - Tool getCurrentTime returned: 2026-07-16T...
DEBUG o.s.ai.chat.model.ToolCallingManager - Executing tool: add
DEBUG o.s.ai.chat.model.ToolCallingManager - Tool add returned: 19.8
```

## 5. 自动模式：`.stream()` 流式 + 自动 tool 循环（最常用）

```java
import org.springframework.http.MediaType;
import reactor.core.publisher.Flux;

@GetMapping(value = "/stream", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
public Flux<String> stream(@RequestParam String q) {
    return chatClient.prompt()
            .user(q)
            .tools(timeTools)
            .stream()
            .content();
}
```

**curl 测试**：
```bash
curl -N "http://localhost:8080/auto/stream?q=讲一个包含当前时间的故事"
```

`-N` = 关闭缓冲，立刻看到 token 逐个输出。

**预期行为**：
- 如果 LLM 先调工具，**流式响应会暂时停顿**（因为 ToolCallingAdvisor 在执行工具）
- 工具执行完后，LLM 继续，后续内容逐字流出
- 你看不到工具调用过程（被 Advisor 吞了），只能看到最终自然语言输出

**如果你也想流式地观察中间过程**，加 `SimpleLoggerAdvisor`（spring-ai 自带）：

```java
import org.springframework.ai.chat.client.advisor.SimpleLoggerAdvisor;

@Bean
ChatClient chatClient(ChatClient.Builder builder, TimeTools timeTools) {
    return builder
            .defaultSystem("...")
            .defaultTools(timeTools)
            .defaultAdvisors(new SimpleLoggerAdvisor())
            .build();
}
```

`SimpleLoggerAdvisor` 同时实现 `CallAdvisor` 和 `StreamAdvisor`，对流式也生效。

## 6. 手动模式：完全控制 tool 调用循环（生产推荐）

**为什么需要手动**：
- 想限制最多调 3 轮 tool（防失控）
- 每轮想换不同的 model options（温度 / 模型名）
- 想在工具调用之间插入业务逻辑（审计 / 限流 / 缓存）

### 6.1 核心循环代码

```java
package com.example.controller;

import com.example.tool.TimeTools;
import org.springframework.ai.chat.client.ChatClient;
import org.springframework.ai.chat.client.ChatClientResponse;
import org.springframework.ai.chat.client.advisor.api.AdvisorParams;
import org.springframework.ai.chat.messages.Message;
import org.springframework.ai.chat.messages.UserMessage;
import org.springframework.ai.chat.model.ChatResponse;
import org.springframework.ai.chat.model.ToolCallingManager;
import org.springframework.ai.chat.prompt.Prompt;
import org.springframework.ai.chat.model.ToolContext;
import org.springframework.ai.tool.ToolCallback;
import org.springframework.ai.tool.definition.ToolDefinition;
import org.springframework.web.bind.annotation.*;
import reactor.core.publisher.Flux;

import java.util.ArrayList;
import java.util.List;
import java.util.concurrent.atomic.AtomicReference;

@RestController
@RequestMapping("/manual")
public class ManualController {

    private final ChatClient chatClient;
    private final ToolCallingManager toolCallingManager;
    private final TimeTools timeTools;

    public ManualController(
            ChatClient chatClient,
            ToolCallingManager toolCallingManager,
            TimeTools timeTools
    ) {
        this.chatClient = chatClient;
        this.toolCallingManager = toolCallingManager;
        this.timeTools = timeTools;
    }

    @GetMapping(value = "/stream", produces = org.springframework.http.MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<String> stream(@RequestParam String q) {

        final int MAX_TOOL_TURNS = 3;   // 三重保护之一：maxTurns
        final List<ToolCallback> tools = List.copyOf(
                org.springframework.ai.tool.ToolCallbacks.from(timeTools)
        );

        return Flux.defer(() -> {
            // 用一个可变容器跨流式 chunk 传递"聚合后的最终响应"
            AtomicReference<ChatClientResponse> lastResponse = new AtomicReference<>();

            List<Message> messages = new ArrayList<>();
            messages.add(new UserMessage(q));

            // 关键 1：关闭自动注册 ToolCallingAdvisor，否则会双重执行
            // 关键 2：关闭内部 conversation history（我们自己管理）
            Flux<ChatClientResponse> firstRound = chatClient.prompt()
                    .messages(messages)
                    .toolContext(java.util.Map.of())
                    .advisors(AdvisorParams.toolCallingAdvisorAutoRegister(false))
                    .stream()
                    .chatClientResponse();

            return streamRecursive(firstRound, messages, lastResponse, tools, MAX_TOOL_TURNS, 0);
        });
    }

    private Flux<String> streamRecursive(
            Flux<ChatClientResponse> currentStream,
            List<Message> messages,
            AtomicReference<ChatClientResponse> lastResponse,
            List<ToolCallback> tools,
            int maxTurns,
            int turn
    ) {
        if (turn >= maxTurns) {
            return Flux.just("[ABORT] 超过最大 tool 调用轮数 " + maxTurns);
        }

        // 关键 3：流式响应中，一次 tool call 被切成多个 chunk
        //         必须用 ChatClientMessageAggregator 聚合后再判断 hasToolCalls
        AtomicReference<ChatClientResponse> aggregated = new AtomicReference<>();

        return new org.springframework.ai.chat.client.ChatClientMessageAggregator()
                .aggregateChatClientResponse(currentStream, aggregated::set)
                .map(resp -> resp.chatResponse() == null ? "" : resp.chatResponse().getResult().getOutput().getText())
                .concatWith(Flux.defer(() -> {
                    ChatClientResponse finalResp = aggregated.get();
                    if (finalResp == null
                            || finalResp.chatResponse() == null
                            || !finalResp.chatResponse().hasToolCalls()) {
                        return Flux.empty();   // 没有工具调用，结束
                    }

                    // 关键 4：用 ToolCallingManager 执行所有 tool calls，得到新的 conversation history
                    ToolCallingManager.ToolExecutionResult execResult =
                            toolCallingManager.executeToolCalls(
                                    new Prompt(messages),
                                    finalResp.chatResponse()
                            );

                    List<Message> updatedMessages = new ArrayList<>(execResult.conversationHistory());

                    // 关键 5：拿更新后的历史，开始下一轮流式
                    Flux<ChatClientResponse> nextRound = chatClient.prompt()
                            .messages(updatedMessages)
                            .toolContext(java.util.Map.of())
                            .advisors(AdvisorParams.toolCallingAdvisorAutoRegister(false))
                            .stream()
                            .chatClientResponse();

                    return streamRecursive(nextRound, updatedMessages, lastResponse, tools, maxTurns, turn + 1);
                }));
    }
}
```

### 6.2 ToolCallingManager Bean

```java
package com.example.config;

import org.springframework.ai.chat.model.ToolCallingManager;
import org.springframework.ai.chat.model.DefaultToolCallingManager;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;

@Configuration
public class ToolManagerConfig {

    @Bean
    ToolCallingManager toolCallingManager() {
        return DefaultToolCallingManager.builder().build();
    }
}
```

> Spring AI 2.0 的 starter 默认会自动注册一个 `DefaultToolCallingManager`，所以这个 Bean 不一定要手写。手写的好处：可以替换 `ToolExecutionExceptionProcessor` 等。

### 6.3 手动模式的核心要点（必须理解）

| 点 | 说明 |
|----|------|
| `AdvisorParams.toolCallingAdvisorAutoRegister(false)` | **必须**关闭自动注册，否则会执行两遍工具 |
| `ChatClientMessageAggregator.aggregateChatClientResponse(...)` | 流式响应被切 chunk，必须先聚合才能 `hasToolCalls()` |
| `toolCallingManager.executeToolCalls(prompt, chatResponse)` | 执行所有工具调用，返回包含完整 conversation history 的结果 |
| `ToolExecutionResult.conversationHistory()` | 包含原始消息 + LLM 的 tool call + 工具结果，下轮直接喂回去 |
| `MAX_TOOL_TURNS` 限制 | 防止 LLM 反复调工具导致成本爆炸 |

## 7. 流式 + 记忆（Memory Advisor）的正确顺序

这是**最容易踩坑**的地方。Spring AI 2.0 的 `MessageChatMemoryAdvisor` 有两种放法：

### 7.1 方案 A：Memory 在循环外（默认，推荐）

`MessageChatMemoryAdvisor` 默认 `Ordered.HIGHEST_PRECEDENCE + 200`，**比 `ToolCallingAdvisor`（`+300`）外层**。

效果：每轮的 conversation history 只写一次 memory，循环结束后写入。

```java
@Bean
ChatClient chatClient(ChatClient.Builder builder, TimeTools timeTools) {
    return builder
            .defaultTools(timeTools)
            .defaultAdvisors(
                    MessageChatMemoryAdvisor.builder(chatMemory).build()
                    // 默认顺序就是 HIGHEST_PRECEDENCE + 200，外层
            )
            .build();
}
```

### 7.2 方案 B：Memory 在循环内

需要把 order 调到 `HIGHEST_PRECEDENCE + 400`（比 ToolCallingAdvisor 内层），并且**关闭内部 conversation history**：

```java
MessageChatMemoryAdvisor.builder(chatMemory)
        .order(Ordered.HIGHEST_PRECEDENCE + 400)
        .build()

// 构造 ChatClient 时
chatClient.prompt()
        ...
        .advisors(
                AdvisorParams.toolCallingAdvisorAutoRegister(false),
                AdvisorParams.toolCallingAdvisorInternalConversationHistory(false)  // 关闭内部历史
        )
        ...
```

**Why**：ToolCallingAdvisor 默认会在循环内自己拼一份 conversation history，如果 Memory Advisor 也在循环内，两边都写历史，结果会重复。关掉 `disableInternalConversationHistory()` 就能避免。

> 95% 的场景用方案 A（默认）。方案 B 适用于"我必须每轮都把上下文写进 memory"的特殊需求。

## 8. 流式 + 工具调用的"看起来卡住了"现象

用户感知：发请求 → 几秒空白 → 突然一次性输出全部内容。

**根本原因**：
1. LLM 第一轮决定调工具 → 流式输出此时是 tool call chunks，**不是文字 chunks**
2. ToolCallingAdvisor 在等流结束 + 执行工具 → 期间没有文字 token
3. 工具结果回传 → LLM 第二轮生成文字 → 开始流式输出

**这是正常的**，不是 bug。

### 8.1 如果想给用户更好的体感

**方案 1：先告诉用户"我正在查工具"**

```java
@GetMapping(value = "/stream", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
public Flux<String> stream(@RequestParam String q) {
    Flux<String> thinkingHint = Flux.just("正在思考...\n\n");
    Flux<String> realStream = chatClient.prompt()
            .user(q)
            .tools(timeTools)
            .stream()
            .content();
    return thinkingHint.concatWith(realStream);
}
```

**方案 2：开 SSE comment 帧**（标准 SSE，浏览器 EventSource 不显示 comment）

```java
return Flux.interval(Duration.ofMillis(500))
        .map(i -> ": keepalive\n\n")
        .takeUntilOther(realStream)
        .concatWith(realStream);
```

**方案 3：直接用 `Flux<ServerSentEvent<String>>`**，每条都带 `event` 类型，前端可按事件类型渲染：

```java
import org.springframework.http.codec.ServerSentEvent;

@GetMapping(value = "/sse", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
public Flux<ServerSentEvent<String>> sse(@RequestParam String q) {
    return chatClient.prompt()
            .user(q)
            .tools(timeTools)
            .stream()
            .content()
            .map(token -> ServerSentEvent.<String>builder()
                    .event("token")
                    .data(token)
                    .build());
}
```

前端 `EventSource.addEventListener("token", ...)` 单独处理。

## 9. 已知限制（Spring AI 2.0 当前版本）

### 9.1 流式不支持 `validateSchema()` 校验

`StructuredOutputValidationAdvisor.validateSchema()` 是同步 + 重试机制（解析失败 → 改 prompt → 重新请求），跟流式语义冲突，**不能同时使用**。

**正确做法**：流式接口里只做"输出文本"，结构化校验在另一个非流式 endpoint 做：

```java
// 流式：纯文本输出
@GetMapping(value = "/stream", produces = TEXT_EVENT_STREAM_VALUE)
public Flux<String> stream(@RequestParam String q) {
    return chatClient.prompt().user(q).tools(timeTools).stream().content();
}

// 结构化：同步调用，可叠加 validateSchema
@GetMapping("/analyze")
public Sentiment analyze(@RequestParam String text) {
    return chatClient.prompt()
            .user(text)
            .call()
            .entity(Sentiment.class);
}
```

### 9.2 流式 + ChatClient span 的链路追踪断裂

当前版本（截至 2026-07-16）下，`ChatClient` 的 observability span 和 tool calling 的 span 在流式模式下**不会自动连成一条链**（同步模式是连的）。GitHub Issue [#4945](https://github.com/spring-projects/spring-ai/issues/4945) 正在处理。

**临时方案**：自己在 Advisor 里补 `MDC.put("traceId", ...)`，日志至少能串起来。

### 9.3 流式过程中 ToolContext 只能传一次

`ToolContext` 在第一次 `prompt()` 设置后，**整个 tool 调用循环都能拿到**（不会丢）。但如果中途想换 context（比如根据用户回答换租户），需要重新构造 prompt。

## 10. 常见错误排查

### 10.1 工具根本没被调用

**症状**：LLM 用自然语言回答"现在时间大概是..."，没调 `getCurrentTime`。

**排查步骤**：
1. 检查日志有没有 `Executing tool: ...` 这行
2. 没有的话，看请求体里 `tools` 数组是不是空的
   - 开 `logging.level.org.springframework.ai: DEBUG`
   - 设置 `spring.ai.openai.chat.observations.include-prompt-extensions: true`
3. `tools` 数组为空 → 检查 `@Tool` 包名是不是 `org.springframework.ai.tool.annotation.Tool`
4. `tools` 数组不为空但模型不调 → 模型可能不支持 function calling（DeepSeek chat 支持，部分小模型不支持）

### 10.2 同一个工具被调用两次

**症状**：日志里看到 `Executing tool: getCurrentTime` 出现两次。

**原因**：你既用了 `.defaultTools(timeTools)`，又没关闭 `ToolCallingAdvisor` 自动注册。

**解决**：手动模式下**必须**加 `AdvisorParams.toolCallingAdvisorAutoRegister(false)`。

### 10.3 流式响应一直不返回

**症状**：curl 一直挂着，没输出。

**排查**：
1. 检查 `produces = MediaType.TEXT_EVENT_STREAM_VALUE` 加了没（不加默认是 `application/json`，浏览器会等所有数据再渲染）
2. 检查 `base-url` 配置，能否 ping 通（DeepSeek：`curl https://api.deepseek.com/v1/models -H "Authorization: Bearer $DEEPSEEK_API_KEY"`）
3. 检查 API key 是否过期 / 余额是否用尽

### 10.4 `NoUniqueBeanDefinitionException: ToolCallingManager`

**原因**：你手写了 `DefaultToolCallingManager` Bean，但 starter 也注册了一个。

**解决**：
- 方案 1：删掉手写的 Bean，用默认的
- 方案 2：手写的 Bean 加 `@Primary`

### 10.5 `ChatClientMessageAggregator` 找不到

**症状**：编译期 `cannot find symbol`。

**原因**：Spring AI 2.0 把这个类从某个内部包挪到了 `org.springframework.ai.chat.client.ChatClientMessageAggregator`。

**解决**：升级到 2.0.0 GA，import 用全限定名。

### 10.6 工具调用结果在第二轮被 LLM 忽略

**症状**：LLM 调了 `getCurrentTime` 拿到 `"2026-07-16T..."`，但下一轮回答还是说"我不知道时间"。

**原因**：conversation history 没拼对，把 ToolResponseMessage 丢了。

**排查**：在递归里打印 `execResult.conversationHistory()`，看里面有没有 `AssistantMessage`（带 tool call）+ `ToolResponseMessage`（带结果）这两条。

```java
execResult.conversationHistory().forEach(m ->
    System.out.println(m.getClass().getSimpleName() + " => " + m.getText())
);
```

# 15. 实战复现：自定义 ChatClient 下流式 + 工具调用的两个真实坑（2026-07-17）

> 本节是 **2026-07-17 在真实 Spring Boot 4.1.0 + Spring AI 2.0.0 GA 项目上的复现记录**，不是抄官方文档，而是按本文 §1-§10 跑了一遍后实际撞墙的坑和最终解。建议把本节当成 §7（Memory 顺序）的实证补充。
>
> 复现环境：`spring-ai-starter-model-openai:2.0.0`、`spring-boot-starter-web:4.1.0`、DeepSeek `deepseek-v4-flash` 模型、`MessageWindowChatMemory`（内存版）。
> 目标接口：`/demo02/devisor/muti-chat-stream?prompt=现在几点了&sessionId=aaaa`，要求 SSE 流式输出 + 自动调用 `getTime` 工具 + 按 `sessionId` 隔离历史。

## 15.1 起点代码（按 §7 方案 A 写的"看起来正确"的版本）

```java
@Configuration
public class ChatConfig {
    @Autowired
    private TimeTools timeTools;

    @Bean
    public ChatMemory chatMemory() {
        return MessageWindowChatMemory.builder().build();
    }

    @Bean
    public ChatClient chatClient(ChatClient.Builder builder, ChatMemory memory) {
        return builder.defaultSystem("你是一个友好的助手")
                .defaultTools(timeTools)
                .defaultAdvisors(
                        MessageChatMemoryAdvisor.builder(memory).build(),
                        new SimpleLogAdvisor()
                ).build();
    }
}
```

按本文 §7.1 的描述，「`MessageChatMemoryAdvisor` 默认 `HIGHEST_PRECEDENCE + 200`，会自动排在 `ToolCallingAdvisor`（`+300`）外层」，看起来不需要任何额外操作。

**实际跑：工具根本没被调用**。

```bash
$ curl -N "http://127.0.0.1:8080/demo02/devisor/muti-chat-stream?prompt=现在几点了&sessionId=aaaa"
data:好的
data:，
data:我来
data:查询
data:一下
data:当前
data:时间
data:。
```

模型只输出了"好的，我来查询一下当前时间。"就结束了。`SimpleLogAdvisor` 日志里没有 `Executing tool call: getTime`。

## 15.2 第一个坑：自定义 `@Bean ChatClient` 会把 `ToolCallingAdvisor` 自动注册吃掉

### 15.2.1 验证模型本身没问题

先用原生 HTTP 直打 DeepSeek，确认 `deepseek-v4-flash` 是支持 function calling 的：

```bash
$ curl https://api.deepseek.com/v1/chat/completions \
  -H "Authorization: Bearer $DEEPSEEK_KEY" \
  -d '{
    "model":"deepseek-v4-flash",
    "messages":[{"role":"user","content":"现在几点了？"}],
    "tools":[{"type":"function","function":{"name":"getTime","description":"获取当前时间","parameters":{"type":"object","properties":{}}}}]
  }'

# 响应（节选）
"tool_calls":[{"index":0,"id":"call_...","type":"function","function":{"name":"getTime","arguments":"{}"}}],
"finish_reason":"tool_calls"
```

模型原生返回 `tool_calls`，`finish_reason=tool_calls`。**问题不在模型**。

### 15.2.2 反编译验证：`ChatClientAutoConfiguration` 的 `@ConditionalOnMissingBean`

把 `spring-ai-autoconfigure-model-chat-client-2.0.0.jar` 解开看：

```
ChatClientAutoConfiguration.chatClientBuilder(
    ...,
    ObjectProvider<ToolCallingAdvisor.Builder<?>> toolCallingAdvisorBuilder
)
```

这个 `@Bean ChatClient.Builder` 是 **`@ConditionalOnMissingBean(ChatClient.class)`** 触发的。一旦你在项目里写了 `@Bean ChatClient chatClient(...)`，自动配置就完全退出，连带着：

- `ToolCallingAdvisor.Builder` 不再创建
- `ToolCallingAdvisor` 不再注入到 builder
- 你的 `defaultTools(timeTools)` **只把 ToolCallback 注册进去，但没人在 advisor 链里执行 tool-calling loop**

这就是工具不被调用的真正原因。本文 §1 "ChatClient 默认会自动注册一个 ToolCallingAdvisor" 这句话**仅在你没有自定义 `ChatClient` Bean 时成立**。

### 15.2.3 修复一：在 ChatConfig 里显式构造 `ToolCallingAdvisor`

```java
@Bean
public ToolCallingAdvisor toolCallingAdvisor(ToolCallingManager toolCallingManager) {
    return ToolCallingAdvisor.builder()
            .toolCallingManager(toolCallingManager)
            .build();
}

@Bean
public ChatClient chatClient(ChatClient.Builder builder, ChatMemory memory,
                              ToolCallingAdvisor toolCallingAdvisor) {
    return builder.defaultSystem("你是一个友好的助手")
            .defaultTools(timeTools)
            .defaultAdvisors(
                    toolCallingAdvisor,                                   // 显式加进去
                    MessageChatMemoryAdvisor.builder(memory).build(),
                    new SimpleLogAdvisor()
            ).build();
}
```

`ToolCallingManager` 由 `spring-ai-autoconfigure-model-tool` 的 `ToolCallingAutoConfiguration` 自动注入，不需要自己声明。

## 15.3 第二个坑：流式下 `conversationId cannot be null`（500 错误）

加完 `ToolCallingAdvisor` 后，**同步接口 `/muti-chat` 工具终于被调用了**（日志出现 `Executing tool call: getTime`），但流式接口 `/muti-chat-stream` 直接 500：

```
java.lang.IllegalArgumentException: conversationId cannot be null
    at org.springframework.util.Assert.notNull(Assert.java:182)
    at org.springframework.ai.chat.client.advisor.api.BaseChatMemoryAdvisor.getConversationId(BaseChatMemoryAdvisor.java:41)
    at org.springframework.ai.chat.client.advisor.MessageChatMemoryAdvisor.after(MessageChatMemoryAdvisor.java:146)
    at org.springframework.ai.chat.client.advisor.MessageChatMemoryAdvisor.lambda$adviseStream$2(MessageChatMemoryAdvisor.java:162)
    at org.springframework.ai.chat.client.ChatClientMessageAggregator.lambda$aggregateChatClientResponse$1(ChatClientMessageAggregator.java:53)
```

### 15.3.1 调用链追溯

`ToolCallingAdvisor` 默认 `advisorOrder = -2147483348`（接近 `Integer.MIN_VALUE`，最外层），`MessageChatMemoryAdvisor` 默认 order=0（内层）。这是 **§7.2 的"方案 B"放法**（Tool 在外、Memory 在内），不是 §7.1 推荐的"方案 A"。

实测的执行链是：

1. `ToolCallingAdvisor.adviseStream` 用 `ChatClientMessageAggregator` 把第一轮流式聚合
2. 检测到 `hasToolCalls()`，执行工具
3. **触发第二轮 stream**（带 tool 结果再次调 ChatModel）
4. 第二轮流式仍然要走完下游 advisor chain，于是进入 `MessageChatMemoryAdvisor.adviseStream`
5. `MessageChatMemoryAdvisor` 也内部聚合一次，在 `after()` 里调 `getConversationId(context)` 读 `chat_memory_conversation_id`
6. **第二轮的 context 是 ToolCallingAdvisor 自己构造的，没有把第一轮的 `adviseContext` 透传**，于是 NPE

这对应 GitHub Issue [#5167](https://github.com/spring-projects/spring-ai/issues/5167) 描述的 "Stream mode loses toolCall information / context" 一类问题。**Spring AI 2.0.0 GA 下，`ToolCallingAdvisor` 排在 `MessageChatMemoryAdvisor` 外层 + 流式 = 必爆**。

### 15.3.2 修复二：调整 advisor order，让 Memory 在 Tool 外层

把顺序翻过来，回到 §7.1 的"方案 A"：

```java
@Bean
public ToolCallingAdvisor toolCallingAdvisor(ToolCallingManager toolCallingManager) {
    return ToolCallingAdvisor.builder()
            .toolCallingManager(toolCallingManager)
            .advisorOrder(100)                    // 内层
            .build();
}

@Bean
public ChatClient chatClient(ChatClient.Builder builder, ChatMemory memory,
                              ToolCallingAdvisor toolCallingAdvisor) {
    return builder.defaultSystem("你是一个友好的助手")
            .defaultTools(timeTools)
            .defaultAdvisors(
                    MessageChatMemoryAdvisor.builder(memory).order(0).build(),  // 外层
                    toolCallingAdvisor,
                    new SimpleLogAdvisor()
            ).build();
}
```

关键差异：

| Advisor | 改前 order | 改后 order | 相对位置 |
|---------|-----------|-----------|---------|
| `MessageChatMemoryAdvisor` | 0（默认） | 0（显式） | 外层 |
| `ToolCallingAdvisor` | -2147483348（默认） | 100（显式） | 内层 |
| `SimpleLogAdvisor` | 0 | 0 | 最外层日志 |

这样 `ToolCallingAdvisor` 的循环完全在 `MessageChatMemoryAdvisor` 的 `before`/`after` 之间发生，`after` 时 context 还是第一轮的（含 `CONVERSATION_ID`），不再 NPE。

## 15.5 最终代码（可直接抄）

```java
package org.demo02.config;

import org.demo02.advisor.SimpleLogAdvisor;
import org.demo02.tools.TimeTools;
import org.springframework.ai.chat.client.ChatClient;
import org.springframework.ai.chat.client.advisor.MessageChatMemoryAdvisor;
import org.springframework.ai.chat.client.advisor.ToolCallingAdvisor;
import org.springframework.ai.chat.memory.ChatMemory;
import org.springframework.ai.chat.memory.MessageWindowChatMemory;
import org.springframework.ai.model.tool.ToolCallingManager;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;

@Configuration
public class ChatConfig {
    @Autowired
    private TimeTools timeTools;

    @Bean
    public ChatMemory chatMemory() {
        return MessageWindowChatMemory.builder().build();
    }

    @Bean
    public ToolCallingAdvisor toolCallingAdvisor(ToolCallingManager toolCallingManager) {
        return ToolCallingAdvisor.builder()
                .toolCallingManager(toolCallingManager)
                .advisorOrder(100)
                .build();
    }

    @Bean
    public ChatClient chatClient(ChatClient.Builder builder, ChatMemory memory,
                                  ToolCallingAdvisor toolCallingAdvisor) {
        return builder.defaultSystem("你是一个友好的助手")
                .defaultTools(timeTools)
                .defaultAdvisors(
                        MessageChatMemoryAdvisor.builder(memory).order(0).build(),
                        toolCallingAdvisor,
                        new SimpleLogAdvisor()
                ).build();
    }
}
```

`TimeTools` / `TestController02` 不动，`mutiChat` / `mutiChatStream` 的历史记录调用方式（`.advisors(spec -> spec.param(ChatMemory.CONVERSATION_ID, sessionId))`）不动。

## 15.6 同时考虑过的备选方案（没采用，记录决策过程）

用户最初提了一个降级思路：「检测到要调工具就走 `call()` 非流式，没工具再走 `stream()`」。这个方案能 work，但被否定：

| 方案 | 工具调用 | 流式体验 | 历史记录 | 代码复杂度 |
|------|---------|---------|---------|-----------|
| 降级方案（有工具用 call，无工具用 stream） | ✅ | ⚠️ 工具时假流式（一次性返回） | ✅ | 高（需两次请求探测） |
| **最终方案**（advisor order 调整） | ✅ | ✅ 真流式 | ✅ | 低（纯配置，零分支） |

降级方案的额外问题：要么先发一个探测请求判断模型是否要调工具（增加一次 LLM 调用成本），要么用启发式（容易误判）。**最终方案是纯框架原生能力，零运行时分支**，所以选择最终方案。

## 15.7 反模式速查（对本文 §7 的实战补充）

| ❌ 反模式 | 后果 | ✅ 正确做法 |
|----------|------|-----------|
| 自定义 `@Bean ChatClient` 但不显式加 `ToolCallingAdvisor` | 工具永远不被调用，`defaultTools` 形同虚设 | 显式 `@Bean ToolCallingAdvisor` 并加进 `defaultAdvisors` |
| `ToolCallingAdvisor` 默认 order（最外层）+ `MessageChatMemoryAdvisor` 内层 + stream | `IllegalArgumentException: conversationId cannot be null` 500 | `ToolCallingAdvisor.advisorOrder(100)`、`MessageChatMemoryAdvisor.order(0)`，让 Memory 在外 |
| `.conversationId("default")` 写在 `MessageChatMemoryAdvisor.Builder` 上 | 编译失败，2.0 已移除该方法 | 调用时 `.advisors(spec -> spec.param(ChatMemory.CONVERSATION_ID, sessionId))` |
