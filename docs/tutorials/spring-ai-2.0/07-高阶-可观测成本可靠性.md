## 2. 可观测性

### 2.1 OTel GenAI Semantic Conventions

OpenTelemetry 2025-06 推出的 GenAI 标准语义约定，所有 LLM 厂商 / 框架统一字段：

| 字段 | 含义 |
|------|------|
| `gen_ai.system` | 模型厂商（openai / anthropic / deepseek） |
| `gen_ai.request.model` | 模型名 |
| `gen_ai.request.max_tokens` | 请求 max tokens |
| `gen_ai.response.finish_reason` | 结束原因（stop / tool_calls / length） |
| `gen_ai.usage.input_tokens` | 输入 token |
| `gen_ai.usage.output_tokens` | 输出 token |
| `gen_ai.tool.name` | 调用的工具名 |

Spring AI 2.0 已内置支持。

### 2.5 自定义业务指标

```java
@Component
@RequiredArgsConstructor
public class ChatMetrics {

    private final MeterRegistry registry;

    public void recordTokenUsage(String model, int inputTokens, int outputTokens) {
        registry.counter("ai.tokens.total",
                "model", model,
                "type", "input"
        ).increment(inputTokens);

        registry.counter("ai.tokens.total",
                "model", model,
                "type", "output"
        ).increment(outputTokens);
    }

    public void recordCost(String model, double cost) {
        registry.counter("ai.cost.usd", "model", model).increment(cost);
    }
}
```

## 3. 成本工程

### 3.1 三大省钱手段

| 手段 | 节省幅度 | 实施难度 |
|------|---------|---------|
| Prompt Cache | 30-90% | 低 |
| 模型路由（小模型分流） | 50-70% | 中 |
| 结果缓存（重复请求） | 20-50% | 低 |

### 3.2 Prompt Cache

参考 [`../../reference/生产化与运营/15-成本工程与PromptCache.md`](../../reference/生产化与运营/15-成本工程与PromptCache.md)。

DeepSeek 和 OpenAI 都支持 prompt cache：**相同的 prompt 前缀**会被缓存，下次命中省钱。

**实现方式**：把 system prompt 放在前面，user message 放在后面：

```java
// ✅ 好的做法：固定前缀
chatClient.prompt()
        .system("固定的长 system prompt...")   // 会被缓存
        .user("变化的用户问题")                  // 不缓存
        .call();

// ❌ 坏的做法：每次拼不一样的 prompt
String prompt = "用户" + userId + "问：" + q;   // 永远 cache 不到
chatClient.prompt().user(prompt).call();
```

**指标**：DeepSeek 返回的 `prompt_cache_hit_tokens` 字段告诉你命中了多少 token。

### 3.3 模型路由

用便宜模型处理简单问题，复杂问题升级：

```java
@Service
public class SmartRouter {

    private final ChatClient cheapModel;   // deepseek-chat
    private final ChatClient strongModel;  // deepseek-reasoner

    public String chat(String q) {
        String category = cheapModel.prompt()
                .system("分类问题：SIMPLE / COMPLEX，只输出分类词")
                .user(q)
                .call()
                .content()
                .trim();

        if ("COMPLEX".equalsIgnoreCase(category)) {
            return strongModel.prompt().user(q).call().content();
        }
        return cheapModel.prompt().user(q).call().content();
    }
}
```

**收益**：80% 问题是 SIMPLE → 走便宜模型 → 整体成本降 50%+。

### 3.4 结果缓存

```java
@Bean
ChatClient chatClient(ChatClient.Builder builder, CacheManager cacheManager) {
    Cache cache = cacheManager.getCache("aiResponses");

    return builder
            .defaultAdvisors(new CachingAdvisor(cache))
            .build();
}

public class CachingAdvisor implements CallAdvisor {
    private final Cache cache;

    @Override
    public ChatClientResponse adviseCall(ChatClientRequest req, CallAdvisorChain chain) {
        String key = sha256(req.prompt().getInstructions().toString());
        ChatClientResponse cached = cache.get(key, ChatClientResponse.class);
        if (cached != null) return cached;

        ChatClientResponse fresh = chain.nextCall(req);
        cache.put(key, fresh);
        return fresh;
    }
}
```

适合：FAQ / 翻译 / 文档摘要等重复性高的场景。

### 3.5 预算守卫

```java
public class BudgetGuardAdvisor implements CallAdvisor {
    private final AtomicLong dailyBudget = new AtomicLong(10_000);   // 日预算 1 万 token
    private final AtomicLong used = new AtomicLong(0);

    @Override
    public ChatClientResponse adviseCall(ChatClientRequest req, CallAdvisorChain chain) {
        if (used.get() >= dailyBudget.get()) {
            throw new BudgetExceededException("今日 AI 预算用尽");
        }
        ChatClientResponse resp = chain.nextCall(req);
        Usage usage = resp.chatResponse().getMetadata().getUsage();
        used.addAndGet(usage.getTotalTokens());
        return resp;
    }
}
```

## 4. 可靠性：Resilience4j

### 4.1 LLM 调用的三类失败

| 失败类型 | 表现 | 应对 |
|---------|------|------|
| 瞬时网络错误 | timeout / connection reset | 重试 |
| 限流 | 429 | 退避重试 |
| 模型异常 | 500 / 输出乱码 | 熔断 + 降级 |

### 4.2 引入 Resilience4j

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-aop</artifactId>
</dependency>
<dependency>
    <groupId>io.github.resilience4j</groupId>
    <artifactId>resilience4j-spring-boot3</artifactId>
    <version>2.2.0</version>
</dependency>
```

### 4.3 三层防护配置

```yaml
resilience4j:
  retry:
    instances:
      aiCall:
        max-attempts: 3
        wait-duration: 500ms
        exponential-backoff-multiplier: 2
        retry-exceptions:
          - java.io.IOException
          - java.net.SocketTimeoutException
          - org.springframework.web.client.ResourceAccessException

  circuitbreaker:
    instances:
      aiCall:
        sliding-window-size: 20
        failure-rate-threshold: 50
        wait-duration-in-open-state: 30s
        permitted-number-of-calls-in-half-open-state: 5

  ratelimiter:
    instances:
      aiCall:
        limit-for-period: 10
        limit-refresh-period: 1s
        timeout-duration: 5s
```

### 4.4 应用到 LLM 调用

```java
@Service
public class RobustChatService {

    private final ChatClient chatClient;

    @CircuitBreaker(name = "aiCall", fallbackMethod = "fallback")
    @Retry(name = "aiCall")
    @RateLimiter(name = "aiCall")
    public String chat(String q) {
        return chatClient.prompt().user(q).call().content();
    }

    public String fallback(String q, Throwable t) {
        log.warn("AI 调用失败，降级到本地缓存", t);
        return "抱歉，AI 服务暂时不可用。已记录你的问题，稍后回复。";
    }
}
```

### 4.5 熔断状态机

```
CLOSED (正常)
    │ 失败率 > 50%
    ▼
OPEN (拒绝所有请求，直接 fallback)
    │ 等 30s
    ▼
HALF_OPEN (放 5 个请求探测)
    │ 5 个都成功 → CLOSED
    │ 有失败 → OPEN（再等 30s）
```

## 5. 失败要被 LLM 看到（Claude Code 启示）

参考 [`../../reference/生产化与运营/12-ClaudeCode源码启示录.md`](../../reference/生产化与运营/12-ClaudeCode源码启示录.md)。

### 5.1 反模式：抛异常

```java
@Tool(description = "查询用户订单")
public Order getOrder(String id) {
    return orderService.findById(id)
        .orElseThrow(() -> new RuntimeException("订单不存在"));
    // 抛异常 → ToolCallingAdvisor 默认会把异常塞回 LLM，但消息不友好
}
```

### 5.2 推荐：返回错误信息

```java
@Tool(description = "查询用户订单")
public ToolResult getOrder(String id) {
    try {
        Order order = orderService.findById(id)
            .orElse(null);
        if (order == null) {
            return ToolResult.error("订单 " + id + " 不存在，请检查订单号");
        }
        return ToolResult.success(order);
    } catch (Exception e) {
        return ToolResult.error("数据库查询失败：" + e.getMessage() + "，请稍后重试");
    }
}
```

LLM 看到 `ToolResult.error(...)` 后会**自我修复**：换一种问法、改订单号、向用户解释。

### 5.3 配置

```java
@Bean
ToolCallingManager toolCallingManager() {
    return DefaultToolCallingManager.builder()
            .toolExecutionExceptionProcessor(
                    new DefaultToolExecutionExceptionProcessor(false)   // false = 不抛，作为 result 返回
            )
            .build();
}
```

或全局：
```yaml
spring:
  ai:
    tools:
      throw-exception-on-error: false
```

## 7. 实战：可观测 + 可靠的 ChatService

```java
@Service
@RequiredArgsConstructor
public class ProductionChatService {

    private final ChatClient chatClient;
    private final ChatMetrics metrics;
    private final CacheManager cacheManager;

    @CircuitBreaker(name = "aiCall", fallbackMethod = "fallback")
    @Retry(name = "aiCall")
    @RateLimiter(name = "aiCall")
    @Timed(value = "ai.chat.duration", description = "Chat call duration")
    public String chat(String userId, String q) {
        // 1. 缓存检查
        String cacheKey = userId + ":" + sha256(q);
        String cached = cacheManager.getCache("aiResponses").get(cacheKey, String.class);
        if (cached != null) {
            metrics.recordCacheHit();
            return cached;
        }

        // 2. 调用 LLM
        String answer = chatClient.prompt()
                .user(q)
                .advisors(spec -> spec.param(ChatMemory.CONVERSATION_ID, userId))
                .call()
                .content();

        // 3. 缓存写入
        cacheManager.getCache("aiResponses").put(cacheKey, answer);

        return answer;
    }

    public String fallback(String userId, String q, Throwable t) {
        metrics.recordFallback();
        return "AI 服务暂时不可用，请稍后重试。";
    }
}
```

## 8. 常见错误

### 8.1 OTel 看不到 token usage

**原因**：没配置 `spring.ai.chat.client.observations.include-prompt-extensions: true`。

**解决**：开启观察扩展（生产注意 prompt 可能含敏感数据）。

### 8.2 Prompt Cache 不生效

**症状**：DeepSeek 返回的 `prompt_cache_hit_tokens` 一直是 0。

**原因**：prompt 前缀每次变（如插了时间戳 / 用户 ID）。

**解决**：把变量挪到 prompt 末尾，固定前缀。

### 8.3 熔断一直不恢复

**原因**：`wait-duration-in-open-state` 太长。

**解决**：调到 30s 以下，配合 `HALF_OPEN` 试探。

### 8.4 重试导致成本爆炸

**原因**：重试没限制次数 / 没退避。

**解决**：`max-attempts: 3` + 指数退避。
