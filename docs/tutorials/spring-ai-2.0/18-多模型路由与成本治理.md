# L2 多模型路由与成本治理（Spring AI 2.0）

> 本文回答：**怎么让简单问题用便宜模型、复杂问题用贵模型？怎么算清一笔 AI 服务的成本？**
>
> 真实生产中 80% 的请求其实是简单问题，但用 GPT-4 烧的钱是 DeepSeek-V3 的 20 倍。多模型路由是企业级 AI 服务的标配。

## 0. 认知地图

```
L1：固定一个模型跑业务
    ↓ 问题：成本爆炸、延迟不均、能力错配
L2：
    ├── 复杂度路由（本文）       ← 按问题难度选模型
    ├── LLM Gateway              ← 统一入口
    ├── Token 配额 + 计费         ← 算清成本
    └── Prompt Cache             ← 省钱利器
    ↓
L3：自适应路由（强化学习）+ 国产/海外双活
```

## 1. 为什么要做多模型路由

### 1.1 成本差异数量级

| 模型 | 输入 ($/1M token) | 输出 ($/1M token) | 备注 |
|------|-------------------|-------------------|------|
| Claude Opus 4 | 15 | 75 | 最强，最贵 |
| GPT-4 Turbo | 10 | 30 | 综合强 |
| Claude Sonnet | 3 | 15 | 性价比 |
| DeepSeek-V3 | 0.27 | 1.1 | 国产之光，便宜 |
| Qwen2.5-Max | 1.4 | 5.6 | 国产 |
| GPT-4o mini | 0.15 | 0.6 | 便宜 |
| Llama 3.1 70B（自托管） | 0.05（电费） | 0.05 | 自有 GPU |

**结论**：同一个问题，不同模型的成本可以差 100 倍。

### 1.2 业务请求的复杂度分布

实测某客服系统 100 万条 query：

| 复杂度 | 占比 | 例子 |
|--------|------|------|
| 简单（FAQ） | 65% | "营业时间"、"退货政策" |
| 中等（多轮） | 25% | "我的订单到哪了" |
| 复杂（推理） | 8% | "对比 A 和 B 产品的差异" |
| 极复杂（多步推理） | 2% | "根据我的使用历史推荐定制方案" |

如果全用 Claude Opus：100% 钱花在最贵模型。
如果 65% 用便宜模型：成本下降约 60%，质量几乎不变。

### 1.3 延迟差异

| 模型 | 首 token 延迟 (P50) | 总延迟 (P95) |
|------|---------------------|--------------|
| GPT-4o mini | 200ms | 1.2s |
| Claude Sonnet | 500ms | 2.5s |
| Claude Opus | 800ms | 5s |
| DeepSeek-V3 | 1.2s（国内访问） | 4s |

**结论**：简单 query 用小模型不只是省钱，更是用户体验。

## 2. 三种路由策略

### 2.1 规则路由（最简单）

用 if/else 根据业务标签选模型。

```java
public ChatModel pickModel(RequestContext ctx) {
    if (ctx.hasTag("code_generation")) return claudeOpus;   // 代码用强模型
    if (ctx.hasTag("simple_faq"))      return gpt4oMini;    // FAQ 用小模型
    if (ctx.hasTag("chinese"))         return deepseekV3;   // 中文用国产
    return defaultModel;
}
```

**优点**：可控、可解释。
**缺点**：业务规则多了维护困难。

### 2.2 复杂度路由（推荐）

用 LLM（或专门的小分类器）判断 query 复杂度，按复杂度选模型。

```
用户 query → 复杂度分类器 →
  ├── simple → GPT-4o mini
  ├── medium → Claude Sonnet
  └── complex → Claude Opus
```

### 2.3 级联路由（Cascading）

先用便宜模型跑，如果模型说"我答不了"（置信度低 / 输出含"我不知道"）再升级到大模型。

```
query → 便宜模型 →
  ├── 输出 OK → 返回
  └── 输出"我不知道" → 升级到贵模型 → 返回
```

**优点**：自适应、最大化省钱。
**缺点**：升级路径增加延迟。

## 3. 复杂度路由的实现

### 3.1 复杂度分类器

用便宜模型（如 GPT-4o mini）做分类，输出三个等级：

```java
// org.demo02.routing.ComplexityClassifier
// 本代码仅作学习材料参考

@Component
public class ComplexityClassifier {

    private final ChatClient classifierClient;   // 用最便宜的模型

    public enum Complexity { SIMPLE, MEDIUM, COMPLEX }

    public Complexity classify(String query) {
        String prompt = """
            判断以下用户问题的复杂度，只输出一个词：simple / medium / complex
            
            判断标准：
            - simple：单步、明确、能用一句话回答（如"营业时间"、"退货政策"）
            - medium：需要解释、对比、推理 1-2 步（如"对比 A 和 B"、"解释 X 是什么"）
            - complex：多步推理、需要规划、代码生成、深度分析
            
            用户问题：%s
            """.formatted(query);

        String result = classifierClient.prompt()
                .user(prompt)
                .call()
                .content()
                .trim()
                .toLowerCase();

        return switch (result) {
            case "simple" -> Complexity.SIMPLE;
            case "medium" -> Complexity.MEDIUM;
            default -> Complexity.COMPLEX;   // 默认走强模型
        };
    }
}
```

### 3.2 路由表

```java
// org.demo02.routing.ModelRouter
// 本代码仅作学习材料参考

@Component
public class ModelRouter {

    private final ComplexityClassifier classifier;

    // 每个复杂度对应的模型
    private final Map<Complexity, ChatModel> models;

    @Value("${demo02.routing.fallback-to-complex:true}")
    private boolean fallbackToComplex;

    public ChatModel route(String query) {
        Complexity c = classifier.classify(query);

        // 防御：如果分类失败默认走 COMPLEX
        ChatModel model = models.getOrDefault(c, models.get(Complexity.COMPLEX));

        log.info("Query routed: {} -> {} -> {}",
                truncate(query, 50), c, getModelName(model));
        return model;
    }
}
```

```java
@Configuration
public class MultiModelConfig {

    @Bean

        OpenAiApi api = OpenAiApi.builder()

    @Bean
    @Qualifier("balanced")
    public ChatModel balancedModel() {
        OpenAiApi api = OpenAiApi.builder()
                .apiKey("xxx")
                .baseUrl("https://api.anthropic.com")
                .build();
        return OpenAiChatModel.builder()
                .openAiApi(api)
                .defaultOptions(OpenAiChatOptions.builder()
                        .model("claude-sonnet-4-5")
                        .build())
                .build();
    }

    @Bean
    @Qualifier("powerful")
    public ChatModel powerfulModel() {
        // ... Claude Opus
    }

    @Bean
    public ChatModelRouter router(
            @Qualifier("cheap") ChatModel cheap,
            @Qualifier("balanced") ChatModel balanced,
            @Qualifier("powerful") ChatModel powerful) {
        return new ChatModelRouter(Map.of(
                ComplexityClassifier.Complexity.SIMPLE, cheap,
                ComplexityClassifier.Complexity.MEDIUM, balanced,
                ComplexityClassifier.Complexity.COMPLEX, powerful
        ));
    }
}
```

## 4. LLM Gateway：统一入口

### 4.1 为什么需要 Gateway

直接每个模型 client 各自调用会有问题：

- 每个模型 SDK 不同（OpenAI / Anthropic / DeepSeek）。
- 切换供应商要改代码。
- 没有 fallback（一个挂了全挂）。
- 没有统一计费、监控。

**Gateway 模式**：所有请求走统一入口，Gateway 内部路由到不同模型。

### 4.2 自建 Gateway 的最小实现

```java
// org.demo02.gateway.LlmGateway
// 本代码仅作学习材料参考

@RestController
@RequestMapping("/demo02/gateway")
public class LlmGateway {

    private final ModelRouter router;
    private final MeterRegistry meters;
    private final QuotaService quota;

    @PostMapping("/chat")
    public ChatResponse chat(@RequestBody ChatRequest req,
                              @RequestHeader("X-Tenant") String tenant,
                              @RequestHeader("X-User") String user) {
        // 1. 配额检查
        if (!quota.tryConsume(tenant, user)) {
            throw new ResponseStatusException(TOO_MANY_REQUESTS, "Quota exceeded");
        }

        // 2. 路由
        ChatModel model = router.route(req.query());

        // 3. 调用（带 fallback）
        try {
            ChatResponse resp = callWithFallback(model, req);
            
            // 4. 计费
            int tokens = resp.getMetadata().getUsage().getTotalTokens();
            BigDecimal cost = pricingCalculator.compute(model, tokens);
            billingService.charge(tenant, user, cost);
            
            // 5. 指标
            meters.timer("ai.gateway.latency", "model", modelName(model))
                  .record(Duration.ofMillis(req.elapsed()));
            meters.counter("ai.gateway.tokens", "model", modelName(model))
                  .increment(tokens);
            
            return resp;
        } catch (Exception e) {
            meters.counter("ai.gateway.errors", "model", modelName(model))
                  .increment();
            throw e;
        }
    }

    private ChatResponse callWithFallback(ChatModel primary, ChatRequest req) {
        try {
            return callOnce(primary, req);
        } catch (Exception e) {
            log.warn("Primary model failed, falling back: {}", e.getMessage());
            ChatModel fallback = router.fallbackFor(primary);
            return callOnce(fallback, req);
        }
    }
}
```

### 4.3 现成的 LLM Gateway

不想自建可以用开源 / 商业：

| 产品 | 类型 | 特点 |
|------|------|------|
| **LiteLLM** | 开源 | Python，支持 100+ 模型供应商 |
| **Portkey** | 商业 | 全功能、可视化 |
| **OpenRouter** | 商业 SaaS | 单一 API 调多家模型 |
| **Cloudflare AI Gateway** | 商业 | 边缘缓存 + 限流 |

**推荐**：Java 项目自建一个简单的 Gateway（如上），复杂场景再上 LiteLLM。

## 5. Token 配额与计费

### 5.1 计费模型

| 模式 | 计费单位 | 适合 |
|------|---------|------|
| 按 Token | $X per 1M token | 直接复用上游 |
| 按请求 | $X per call | 简单但易被超长 prompt 套利 |
| 按订阅 | 月费 N 次 | SaaS |
| 按 Token 加价 | 上游价 × 1.2 | 平台抽成模式 |

### 5.2 配额实现

```sql
CREATE TABLE tenant_quota (
    tenant_id VARCHAR(64),
    day DATE,
    tokens_used BIGINT DEFAULT 0,
    tokens_limit BIGINT NOT NULL,
    PRIMARY KEY (tenant_id, day)
);

CREATE TABLE token_usage_log (
    id BIGSERIAL,
    tenant_id VARCHAR(64),
    user_id VARCHAR(64),
    model VARCHAR(64),
    input_tokens INT,
    output_tokens INT,
    cost_usd DECIMAL(10, 6),
    created_at TIMESTAMP,
    INDEX idx_tenant_day (tenant_id, created_at)
);
```

```java
// org.demo02.billing.QuotaService
// 本代码仅作学习材料参考

@Service
public class QuotaService {

    private final JdbcTemplate jdbc;

    @Transactional
    public boolean tryConsume(String tenant, String user, int estimatedTokens) {
        LocalDate today = LocalDate.now();
        int updated = jdbc.update("""
            UPDATE tenant_quota 
            SET tokens_used = tokens_used + ? 
            WHERE tenant_id = ? AND day = ? AND tokens_used + ? <= tokens_limit
            """, estimatedTokens, tenant, today, estimatedTokens);
        return updated > 0;
    }

    public void recordUsage(String tenant, String user, String model,
                             int inputTokens, int outputTokens, BigDecimal cost) {
        jdbc.update("""
            INSERT INTO token_usage_log 
            (tenant_id, user_id, model, input_tokens, output_tokens, cost_usd, created_at)
            VALUES (?, ?, ?, ?, ?, ?, NOW())
            """, tenant, user, model, inputTokens, outputTokens, cost);
    }
}
```

### 5.3 实时算钱

```java
@Component
public class PricingCalculator {

    // (model, per_1m_input, per_1m_output)
    private static final Map<String, Pricing> PRICING = Map.of(
            "gpt-4o-mini", new Pricing(0.15, 0.60),
            "claude-sonnet-4-5", new Pricing(3.0, 15.0),
            "claude-opus-4", new Pricing(15.0, 75.0),
            "deepseek-v3", new Pricing(0.27, 1.10)
    );

    public BigDecimal compute(String model, int inputTokens, int outputTokens) {
        Pricing p = PRICING.getOrDefault(model, new Pricing(1.0, 2.0));
        return BigDecimal.valueOf(inputTokens)
                .multiply(BigDecimal.valueOf(p.inputPerMillion()))
                .divide(BigDecimal.valueOf(1_000_000), 6, RoundingMode.HALF_UP)
                .add(BigDecimal.valueOf(outputTokens)
                        .multiply(BigDecimal.valueOf(p.outputPerMillion()))
                        .divide(BigDecimal.valueOf(1_000_000), 6, RoundingMode.HALF_UP));
    }

    record Pricing(double inputPerMillion, double outputPerMillion) {}
}
```

## 6. Prompt Cache：省钱的核武器

### 6.1 原理

很多 query 的前缀是相同的（system prompt + few-shot 例子）。**重复部分缓存**，下次只算新部分的 token。

| Provider | Cache 能力 | 缓存命中价格 |
|---------|-----------|------------|
| Anthropic | Prompt Caching（自动） | 输入价 10% |
| OpenAI | Automatic Prompt Caching | 输入价 50% |
| DeepSeek | Context Caching | 输入价 1% |

**收益**：长期运行的 Agent，system prompt 几千 token，缓存命中可省 50~90%。

### 6.2 在 Spring AI 用 Anthropic Prompt Cache

Anthropic 是自动的 —— 只要前缀稳定就自动命中。要点：

- system prompt 放最前面（不要每次重排）。
- few-shot 例子紧跟 system。
- 用户输入放最后。

```java
ChatClient.builder(anthropicModel)
        .defaultSystem("""
            你是公司助手。公司信息：
            ... (大量静态内容 5000 token)
            
            操作规范：
            ... (大量规范 3000 token)
            """)   // 这部分被缓存
        .build();

// 用户实际 prompt = 8000 token（缓存） + 用户问的 50 token
// 计费：8000 × 0.1（缓存价） + 50 × 1
```

### 6.3 自建语义缓存

对完全相同的 query 重复请求，可以自建缓存：

```java
// org.demo02.cache.SemanticCacheAdvisor
// 本代码仅作学习材料参考

public class SemanticCacheAdvisor implements BaseAdvisor {

    private final VectorStore cacheStore;
    private final double similarityThreshold;  // 例如 0.95

    @Override
    public ChatClientRequest before(ChatClientRequest req, AdvisorChain chain) {
        String query = req.prompt().getUserMessage().getText();
        // 用 embedding 找相似 query
        List<Document> hits = cacheStore.similaritySearch(
                SearchRequest.builder().query(query).topK(1).build());
        if (!hits.isEmpty() && hits.get(0).getMetadata().getScore() > similarityThreshold) {
            // 命中缓存，跳过 LLM
            return req.mutate()
                    .prompt(req.prompt().mutate()
                            .userMessage(new UserMessage("[CACHE HIT]"))
                            .build())
                    .cache("cached_response", hits.get(0).getText())
                    .build();
        }
        return req;
    }

    @Override
    public ChatClientResponse after(ChatClientResponse resp, AdvisorChain chain) {
        // 把这次的 query + 答案存到缓存
        String cached = resp.context().cache("cached_response");
        if (cached != null) {
            return resp.mutate().chatResponse(/* 用 cached 替换 */).build();
        }
        cacheStore.add(List.of(new Document(resp.chatResponse().getText())));
        return resp;
    }
}
```

**警告**：语义缓存要慎用，可能把"上次给 A 用户的个性化答案"返回给 B 用户。生产用要带 userId/tenantId 过滤。

## 7. 实战避坑

### 7.1 "路由分类器自己烧钱"

**症状**：每次请求都调一次分类器，分类器的成本占了 10%。

**解决**：

- 分类器用最便宜的模型（gpt-4o-mini $0.15/1M）。
- 或者训练一个本地 BERT 小模型做分类（成本几乎为零）。
- 或者用规则路由覆盖高频 case，分类器只兜底。

### 7.2 "级联路由延迟翻倍"

**症状**：升级路径让 P95 延迟从 3s 涨到 8s。

**解决**：

- 设短超时：便宜模型 2s 不返回就升级。
- 用流式：便宜模型边输出边判断，发现答不上来立即 cut off 升级。

### 7.3 "成本超预算"

**症状**：月底一算，账单是预算的 3 倍。

**解决**：

- 每日预算硬上限（超了直接拒绝请求）。
- 每用户每月配额。
- 异常流量告警（某用户突然 10x 调用）。

### 7.4 "供应商挂了全挂"

**症状**：OpenAI 服务降级，整个应用不可用。

**解决**：

- 多供应商（OpenAI 挂走 Anthropic，都挂走 DeepSeek）。
- 自托管兜底模型（如 Llama）。
- Fallback 链：primary → secondary → tertiary → cached_response。

### 7.5 "Prompt Cache 命中率低"

**症状**：开了 Prompt Caching 但命中率 < 10%。

**原因**：每次 prompt 前缀都不一样（如把 timestamp、userId 拼在 system prompt 里）。

**解决**：

- 静态内容（system + few-shot）和动态内容（用户输入）严格分离。
- 静态部分保持稳定，不掺入运行时变量。

## 8. 监控看板

### 8.1 关键指标

| 指标 | 含义 | 告警阈值 |
|------|------|---------|
| 日总成本 | 当天累计 USD | > 日预算 80% |
| 每千请求成本 | 单位经济模型 | 周环比 +20% |
| 各模型调用量 | 路由分布 | 异常倾斜 |
| Cache 命中率 | 缓存效果 | <30%（异常） |
| 错误率 | 5xx + 超时 | >1% |
| P95 延迟 | 用户体验 | >5s |

### 8.2 Grafana SQL 示例

```sql
-- 日成本趋势
SELECT 
    DATE(created_at) AS day,
    SUM(cost_usd) AS total_cost,
    COUNT(*) AS requests
FROM token_usage_log
WHERE created_at > NOW() - INTERVAL '30 days'
GROUP BY day
ORDER BY day;

-- 各模型用量
SELECT 
    model,
    SUM(input_tokens + output_tokens) AS total_tokens,
    SUM(cost_usd) AS total_cost
FROM token_usage_log
WHERE created_at > NOW() - INTERVAL '1 day'
GROUP BY model;
```
