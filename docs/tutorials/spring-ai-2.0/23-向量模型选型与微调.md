## 0. 认知地图

```
不需要训练 → 直接用通用 embedding（bge-large / jina / Cohere）
        ↓ 业务术语严重 / 跨语言 / 长文档
需要微调 → 用 SentenceTransformers + LoRA / 对比学习
        ↓ 仍不够
需要训练 → 自监督预训练（成本极高，99% 项目用不到）
```

```
数据敏感不能出域？
├── 是 → 开源：bge-m3 / bge-large-zh / jina-v3
└── 否 →
    预算充足，要最快上效果？
    ├── 是 → 商业：OpenAI 3-large / Cohere v3 / Voyage 3
    └── 否 → 开源自托管：bge-m3
```

## 2. 评估 embedding 模型

### 2.1 MTEB / C-MTEB

- **MTEB**（Massive Text Embedding Benchmark）：58 个英文任务的 leaderboard。
- **C-MTEB**：中文版，35 个任务。

✅ 适合**初筛**。❌ **不等于**你的业务表现——MTEB 跑的是通用任务，你的可能是法律 / 医疗 / 电商。

### 2.2 业务评估三件套

#### A. 检索 Recall@K

```python
# 伪代码
queries = load_test_queries()                # 业务真实 query
relevant_docs = load_test_relevant()         # 人工标注的相关文档

for model_name in ['bge-m3', 'jina-v3', 'openai-3']:
    embeddings = embed(model_name, docs)
    index = build_faiss(embeddings)
    hit = 0
    for q, gt in zip(queries, relevant_docs):
        retrieved = index.search(embed(model_name, [q]), k=10)
        if set(retrieved) & set(gt):
            hit += 1
    recall_at_10 = hit / len(queries)
```

#### B. 端到端 RAG 指标

把 embedding 嵌入完整 RAG，跑 [`./11-评估闭环与Prompt版本管理.md`](./11-评估闭环与Prompt版本管理.md) 的 eval 集，看 `answer correctness` / `faithfulness`。

#### C. 困难负样本对比

随机负样本容易过拟合，主动挖"看着像其实无关"的负样本（hard negatives）：

- 同类别不同文档（同 FAQ 库不同条目）
- 同字段不同值（同一商品不同颜色）
- 时间错位（去年的政策 vs 今年的）

### 3.1 用商业 API

```yaml

    openai:
      api-key: ${OPENAI_API_KEY}
      embedding:
        options:
          model: text-embedding-3-large
```

### 3.2 用本地 ONNX 模型（Spring AI 内置 ONNX SPI）

```yaml
spring:
  ai:
    embedding:
      transformer:
        onnx:
          model-uri: classpath:/bge-large-zh.onnx
        tokenizer:
          uri: classpath:/tokenizer.json
```

### 3.3 用 Ollama 自托管

```yaml
spring:
  ai:
    ollama:
      base-url: http://localhost:11434
      embedding:
        model: bge-m3
```

### 3.4 多 embedding 模型同时使用

```java
// 本代码仅作学习材料参考
@Bean
EmbeddingModel shortTextEmbedding(...) { /* OpenAI */ }

@Bean
EmbeddingModel longDocEmbedding(...) { /* bge-m3 local */ }

// 业务里按 doc.length 路由
public float[] embed(String text) {
    return (text.length() > 2000 ? longDocEmbedding : shortTextEmbedding)
            .embed(text);
}
```

## 4. 什么时候该微调

### 4.1 信号清单（任一满足即考虑微调）

| 信号 | 解释 |
|------|------|
| 业务术语严重（金融衍生品、医疗 ICD-10） | 通用模型把"看涨期权"和"看跌期权"embedding 太近 |
| 同义词多（电商 SKU、品牌昵称） | "iPhone 15 Pro Max" / "苹果 15PM" 通用模型区分不开 |
| 跨语言混合（中英夹杂的技术文档） | 通用模型在中英混合句上召回下降 |
| 长文档切分后语义丢失 | chunk 切分导致 chunk 内语义不完整 |
| MTEB 高但业务 recall@10 < 0.6 | 通用 benchmark 不能代表业务 |

### 4.2 不该微调的信号

- recall@10 已经 ≥ 0.85：**问题在 prompt / chunk 策略**，不在 embedding。
- 数据 < 1000 条正样本：训练不充分，过拟合风险大。
- 没有 eval set：微调后没法验证收益。

## 5. 微调方法论

### 5.1 主流范式：对比学习 + 困难负样本

目标是让 `(query, positive_doc)` 距离小，`(query, negative_doc)` 距离大。

```
loss = -log( exp(sim(q, p+) / τ) / Σ exp(sim(q, p_i) / τ) )
```

### 5.2 数据准备三件套

每个 query 准备：

- 1 个 **正样本**（人工标注的相关文档）
- 1 个 **强负样本**（hard negative）
- N 个 **随机负样本**（in-batch）

数据量参考：

| 数据规模 | 效果 |
|---------|------|
| 1000 query | 略有提升 |
| 1 万 query | 明显提升（+5~10pt recall） |
| 10 万 query | 接近上限 |

### 5.3 主流工具

| 工具 | 语言 | 适合 |
|------|------|------|
| **SentenceTransformers** | Python | 入门首选，文档好 |
| **FlagEmbedding**（BGE 团队） | Python | bge 系列微调官方工具 |
| **LLM2Vec** | Python | 把 LLM 改造成 embedding |
| **TEI**（HuggingFace Text Embeddings Inference） | Go/Rust | 部署推理服务 |

### 5.4 LoRA 微调（轻量）

```python
# 伪代码（实际见 FlagEmbedding 文档）
from FlagEmbedding import FlagModel
from peft import LoraConfig

model = FlagModel('bge-large-zh-v1.5', use_fp16=True)
lora_config = LoraConfig(
    r=16, lora_alpha=32,
    target_modules=["query", "value"],
    task_type="FEATURE_EXTRACTION"
)
model.finetune(
    train_data='train.jsonl',
    loss='InfoNCE',
    neg_data='hard_neg.jsonl',
    lora_config=lora_config,
    epochs=3, batch_size=64
)
```

**LoRA 优势**：

- 训练快（全量微调 1/5 时间）
- 显存省（1×A100 训 7B 模型够用）
- 可热插拔（一个 base + N 个 LoRA 适配不同业务）

```text

    ↓
```

## 8. 工程化细节

### 8.1 重新 embedding 的成本

切 embedding 模型 = 全量重建向量库。**100 万 chunk × 1024 dim × 4 byte ≈ 4GB**，重建耗时数小时。

**建议**：

- 新旧 schema 共存（`embedding_v1`, `embedding_v2`），双写 1-2 周后切换。
- 用 [`./17-大规模Agent平台与数据基础设施.md`](./17-大规模Agent平台与数据基础设施.md) 的"双管道"模式。

### 8.2 维度压缩

OpenAI text-embedding-3 支持 `dimensions` 参数（3072 → 256），用 Matryoshka 表示学习，掉点很小但存储省 12 倍。

### 8.3 量化（Quantization）

- **FP32 → FP16**：存储减半，精度几乎不损。
- **FP16 → INT8**：存储 1/4，掉点 1-3%。
- **Binary quantization**：存储 1/32，掉点 5-10%。适合"粗召回 + 精排"架构。

### 8.4 多向量（ColBERT 风格）

bge-m3 同时输出 dense + sparse + colbert 三种向量。混合检索 recall 显著高于单一 dense（见 [`./19-RAG高级篇.md`](./19-RAG高级篇.md) Hybrid Search）。

## 9. 反模式速查

| 反模式 | 后果 | 修复 |
|--------|------|------|
| 用 ChatGPT 的 embedding 做 RAG | 中文场景不如 bge | 中文场景用 bge-m3 |
| 用 cosine 相似度前不 normalize | 性能慢 10x | normalize 后用 dot product |
| chunk 切太大（>1000 token） | embedding 平均化 | 256-512 token，加 Parent-Child |
| 微调后没在 holdout 上验证 | 看似变好实际过拟合 | 留 20% 数据做 holdout |
| 切模型不重建索引 | 检索结果错乱 | 全量 reindex 或双写 |
| 微调用 LLM 自动生成的正样本 | 噪声大 | 人工标注 1k 条胜过 LLM 生成 10k |
| 没有 hard negative | 模型学到表面特征 | 主动挖掘 hard neg |
