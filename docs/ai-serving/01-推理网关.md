```
│  ┌──────────┐   ┌──────────┐   ┌──────────────┐   │
│  │ Provider │   │ Router   │   │ Resilience   │   │

│  │ 语义缓存 │   │ Token    │   │ OTel + Metrics│  │

└─────────────────────────────────────────────────────┘
                    ↓
   OpenAI / Anthropic / 通义 / 智谱 / DeepSeek / ...
```

**心法**：网关不是包一层 HTTP Client 那么简单。它的核心价值是**让上层业务不感知底层模型差异**，同时**让运维能治理所有流量**。

## 1. 为什么需要推理网关

### 1.1 没有网关的痛点

```java
// 业务代码里直接调
ChatClient.builder().build()
    .prompt().user("你好").call().content();
```

跑通了。然后呢？

- **换模型**：要改业务代码（OpenAI → 国产模型）
- **降级**：OpenAI 挂了，业务全瘫
- **算账**：哪个业务烧了多少钱？不知道
- **限流**：一个业务把 quota 用光，其他业务都用不了
- **审计**：合规要求留 prompt 日志，要改 N 处
- **缓存**：同样的 prompt 重复调，浪费钱

### 1.2 有网关之后

业务代码只调网关：

```java
// 本代码仅作学习材料参考
gateway.chat(GatewayRequest.builder()
    .tenantId("tenantA")
    .businessCode("order-assistant")
    .messages(messages)
    .preference(COST)        // 偏好：成本优先
    .build());
```

网关自己处理：路由到哪个模型、缓存命中没、限额超了没、要不要审计、失败了降级到哪个备份模型。

### 1.3 业界参考

- **云厂商**：阿里百炼、火山方舟、腾讯混元接入平台都是这个定位
- **开源**：LiteLLM（Python）、One-API、Portkey、Helicone
- **Spring AI 自身**：2.0 已经有 `ChatModel` 抽象 + `ModelManagementOptions`，但只是地基，**真正的网关要自己搭**

### 3.1 Spring AI 已经做了什么

### 3.2 网关层的 Provider 注册

```java
// 本代码仅作学习材料参考
```

## 4. Layer 3：Router 路由策略

路由是网关的"大脑"。同一个请求，按业务偏好和实时状态选不同模型。

### 4.1 四种路由策略

| 策略 | 决策依据 | 典型场景 |
|------|---------|---------|
| **COST** | 选最便宜的能干活的 | 高频内部工具 |
| **LATENCY** | 选最快的 | 实时对话、客服 |
| **QUALITY** | 选最强的 | 法律 / 医疗 / 代码生成 |
| **TENANT** | 按租户订阅等级 | SaaS 产品 |

### 4.2 路由器接口

```java
// 本代码仅作学习材料参考
public interface ModelRouter {

    ModelDescriptor route(GatewayRequest req, List<ModelDescriptor> candidates);
}

public record GatewayRequest(
    String tenantId,
    String businessCode,
    List<Message> messages,
    List<ToolDescriptor> tools,
    ChatPreference preference,    // COST / LATENCY / QUALITY / TENANT
    int maxTokens,
    boolean stream,
    Map<String, String> metadata
) {}
```

### 4.3 复合路由器（生产推荐）

实际生产用单一策略不够，要组合：

```java
// 本代码仅作学习材料参考
public class CompositeRouter implements ModelRouter {

    private final List<RouterFilter> filters;

    @Override
    public ModelDescriptor route(GatewayRequest req, List<ModelDescriptor> candidates) {
        List<ModelDescriptor> pool = new ArrayList<>(candidates);

        // 1. 过滤：能力匹配（vision / function-calling）
        pool = capabilityFilter(req, pool);

        // 2. 过滤：上下文窗口够不够
        pool = contextWindowFilter(req, pool);

        // 3. 过滤：租户订阅允许的模型
        pool = tenantQuotaFilter(req, pool);

        // 4. 过滤：当前熔断 / 限流状态的模型剔除
        pool = healthFilter(pool);

        if (pool.isEmpty()) {
            throw new NoModelAvailableException(req);
        }

        // 5. 评分：按 preference 选最优
        return scoreByPreference(req, pool);
    }

    private ModelDescriptor scoreByPreference(GatewayRequest req, List<ModelDescriptor> pool) {
        return switch (req.preference()) {
            case COST     -> cheapest(pool);
            case LATENCY  -> fastest(pool);
            case QUALITY  -> highestTier(pool);
            case TENANT   -> tenantSubscription(req, pool);
        };
    }
}
```

### 4.4 实时延迟感知

LATENCY 策略不能只看模型标称延迟，要看**实时滑动窗口**：

```java
// 本代码仅作学习材料参考
@Component
public class LatencyTracker {

    // 每个模型的最近 100 次调用延迟
    private final Map<String, LatencyStats> stats = new ConcurrentHashMap<>();

    public void record(String modelId, long latencyMs) {
        stats.computeIfAbsent(modelId, k -> new LatencyStats(100))
             .record(latencyMs);
    }

    public Duration p50Latency(String modelId) {
        return stats.getOrDefault(modelId, LatencyStats.empty()).p50();
    }

    public Duration p95Latency(String modelId) {
        return stats.getOrDefault(modelId, LatencyStats.empty()).p95();
    }
}
```

`fastest()` 用 p50 排序，比硬编码"哪个模型快"靠谱得多。

### 4.5 路由决策日志

每次路由都记下来，便于复盘：

## 5. Layer 3：Cache 语义缓存

### 5.1 为什么需要缓存

同一个 prompt 调两次是浪费钱。常见场景：

- 用户重复点提交
- RAG 系统的 query 改写阶段，同样的 query 重复出现
- 测试环境的回归请求

### 5.2 两层缓存

```
请求 → [精确缓存] → 命中？
        ↓ 未命中
      [语义缓存] → 命中？
        ↓ 未命中
      调模型 → 写回两层缓存
```

**精确缓存**：prompt 完全相同（哈希匹配）。命中率高、零误差。

**语义缓存**：prompt 语义相似（embedding 距离 < 阈值）。命中率高但有误差，**只适合可容忍误差的场景**（如 RAG 改写、闲聊），不适合写操作 / 代码生成 / 法律咨询。

### 5.3 精确缓存实现

```java
// 本代码仅作学习材料参考
@Component
@RequiredArgsConstructor
public class ExactCache {

    private final Cache<String, CachedResponse> cache;   // Caffeine / Redis

    public Optional<ChatResponse> lookup(GatewayRequest req) {
        String key = hashKey(req);
        CachedResponse hit = cache.getIfPresent(key);
        if (hit == null) return Optional.empty();
        if (hit.isExpired()) {
            cache.invalidate(key);
            return Optional.empty();
        }
        return Optional.of(hit.response());
    }

    private String hashKey(GatewayRequest req) {
        // 注意：messages + tools + modelPreference + maxTokens 都参与 hash
        // 但 tenantId / businessCode 不参与（缓存是跨租户的）
        return DigestUtils.sha256Hex(
            req.messages() + "|" + req.tools() + "|" +
            req.preference() + "|" + req.maxTokens()
        );
    }
}
```

**关键决策**：缓存要不要跨租户？

- 跨租户：命中率高，但有数据泄露风险（租户 A 的私有上下文被租户 B 命中）
- 不跨租户：把 tenantId 加入 hash key，安全但命中率低

**生产推荐**：默认不跨租户，只在"无敏感数据"的场景（如通用知识问答）打开跨租户开关。

### 5.4 语义缓存实现

```java
// 本代码仅作学习材料参考
@Component
@RequiredArgsConstructor
public class SemanticCache {

    private final EmbeddingModel embedding;
    private final VectorStore vectorStore;
    private final double threshold = 0.92;   // 余弦相似度阈值

    public Optional<ChatResponse> lookup(GatewayRequest req) {
        String queryText = lastUserMessage(req);
        float[] queryVec = embedding.embed(queryText);

        SearchRequest sr = SearchRequest.builder()
            .query(queryText)
            .topK(1)
            .similarityThreshold(threshold)
            .filterExpression("businessCode == '" + req.businessCode() + "'")
            .build();

        List<Document> hits = vectorStore.similaritySearch(sr);
        if (hits.isEmpty()) return Optional.empty();

        CachedEntry entry = CachedEntry.fromJson(hits.get(0).getText());
        return Optional.of(entry.response());
    }

    public void put(GatewayRequest req, ChatResponse response) {
        // 只缓存稳定答案（流式 / 工具调用不缓存）
        if (req.stream() || !req.tools().isEmpty()) return;

        CachedEntry entry = new CachedEntry(req, response, Instant.now());
        vectorStore.add(List.of(new Document(
            JsonUtils.toJson(entry),
            Map.of("businessCode", req.businessCode(),
                   "ts", System.currentTimeMillis())
        )));
    }
}
```

**避坑**：

- 语义缓存的过滤一定要带 `businessCode`（不同业务场景的相似 prompt 语义可能截然不同）
- 阈值要按业务调（建议 0.85-0.95，越高越保守）
- 写操作 / 工具调用 / 流式响应**不要进语义缓存**

## 6. Layer 3：Resilience 熔断 / 重试 / 降级

### 6.1 三层防护

```
请求 → Timeout（单次调用上限）
        ↓
      CircuitBreaker（连续失败快速失败）
        ↓
      Retry（瞬时错误重试）
        ↓
      Fallback（失败降级到备份模型）
```

### 6.2 配置

```java
// 本代码仅作学习材料参考
@Configuration
public class GatewayResilienceConfig {

    @Bean
    public CircuitBreaker openaiCb() {
        return CircuitBreaker.of("openai", CircuitBreakerConfig.custom()
            .failureRateThreshold(50)             // 50% 失败开熔断
            .slowCallRateThreshold(80)            // 80% 慢调用开熔断
            .slowCallDurationThreshold(Duration.ofSeconds(30))
            .waitDurationInOpenState(Duration.ofSeconds(60))
            .slidingWindowSize(20)
            .minimumNumberOfCalls(10)
            .build());
    }

    @Bean
    public Retry retry() {
        return Retry.of("llm", RetryConfig.custom()
            .maxAttempts(3)
            .waitDuration(Duration.ofMillis(500))
            .retryOnException(this::isRetryable)
            .build());
    }

    private boolean isRetryable(Throwable e) {
        // 4xx 不重试（请求本身有问题）
        // 5xx / 超时 / 限流 才重试
        if (e instanceof HttpClientErrorException) return false;
        return true;
    }

    @Bean
    public TimeLimiter timeLimiter() {
        return TimeLimiter.of(Duration.ofSeconds(60));
    }
}
```

### 6.3 Fallback 链

```java
// 本代码仅作学习材料参考
@Component
public class FallbackChain {

    private final Map<String, List<String>> fallbackMap = Map.of(
        "gpt-4o",           List.of("claude-sonnet-4-6", "qwen-plus"),
        "claude-sonnet-4-6", List.of("gpt-4o", "qwen-plus"),
        "qwen-plus",         List.of("deepseek-chat", "gpt-4o-mini")
    );

    public List<String> fallbackChain(String primary) {
        return fallbackMap.getOrDefault(primary, List.of());
    }
}

// 使用
public ChatResponse callWithFallback(GatewayRequest req, String modelId) {
    try {
        return callModel(modelId, req);
    } catch (Exception e) {
        for (String fb : fallbackChain.fallbackChain(modelId)) {
            try {
                log.warn("Fallback {} → {}", modelId, fb);
                return callModel(fb, req);
            } catch (Exception ignored) {}
        }
        throw new AllModelsFailedException(modelId);
    }
}
```

**Fallback 链设计原则**：

- 备份模型的能力要 ≥ 主模型（否则降级后效果崩）
- 备份模型尽量跨厂商（OpenAI 挂了用 Anthropic，不要再用 Azure OpenAI）
- Fallback 要在审计里显式标记，便于事后追账（"为什么这次用了更贵的 Claude"）

### 7.2 实时计费

```java
// 本代码仅作学习材料参考
@Component
@RequiredArgsConstructor
public class BillingService {

    private final BillingRepository repo;
    private final ModelDescriptorRegistry registry;

    public BillingRecord record(GatewayRequest req, ChatResponse resp,
                                 String modelId, long latencyMs, boolean cached) {
        Usage usage = resp.getMetadata().getUsage();

        ModelDescriptor desc = registry.get(modelId);
        BigDecimal cost = BigDecimal.valueOf(
            usage.getPromptTokens() * desc.costPer1kInput() / 1000.0
            + usage.getCompletionTokens() * desc.costPer1kOutput() / 1000.0
        );

        BillingRecord rec = BillingRecord.builder()
            .requestId(req.requestId())
            .tenantId(req.tenantId())
            .businessCode(req.businessCode())
            .modelId(modelId)
            .promptTokens(usage.getPromptTokens())
            .completionTokens(usage.getCompletionTokens())
            .costUsd(cost)
            .cached(cached)
            .latencyMs(latencyMs)
            .calledAt(Instant.now())
            .build();

        return repo.save(rec);
    }
}
```

### 7.3 配额预扣

每次调用前**预扣**配额，避免超额：

```java
// 本代码仅作学习材料参考
@Service
@RequiredArgsConstructor
public class QuotaService {

    private final QuotaRepository quotaRepo;
    private final ModelDescriptorRegistry registry;

    public QuotaToken reserve(GatewayRequest req, String modelId, int estInputTokens) {
        BigDecimal cost = registry.estimateCost(modelId, estInputTokens, 1000);
        boolean ok = quotaRepo.tryDeduct(req.tenantId(), req.businessCode(), cost);
        if (!ok) {
            throw new QuotaExceededException(req.tenantId(), req.businessCode());
        }
        return new QuotaToken(req.requestId(), cost);
    }

    public void settle(QuotaToken token, BigDecimal actualCost) {
        // 实际成本 < 预扣：退差额
        // 实际成本 > 预扣：补扣（但要防止单次超额）
        quotaRepo.adjust(token, actualCost);
    }
}
```

**为什么预扣**：流式响应可能调很久，期间用户可能并发发起多个请求，不预扣容易超额。

## 8. Layer 1：Audit Prompt 日志

### 8.1 合规要求

某些行业（金融 / 医疗 / 政企）的合规要求：

- 所有 prompt 留档（脱敏后）
- 保留期 ≥ 6 个月
- 可按用户 / 时间 / 业务追溯
- 不可篡改（写入后只读 / 加签名）

### 8.2 分级日志

```java
// 本代码仅作学习材料参考
public enum AuditLevel {
    NONE,          // 不记
    METADATA,      // 只记元数据（不记 prompt 内容）
    PROMPT_HASH,   // 记 prompt 哈希（不记明文）
    FULL           // 记完整 prompt + response（脱敏）
}

@Component
public class AuditLogger {

    public void log(GatewayRequest req, ChatResponse resp, AuditLevel level) {
        AuditEntry.AuditEntryBuilder b = AuditEntry.builder()
            .requestId(req.requestId())
            .tenantId(req.tenantId())
            .businessCode(req.businessCode())
            .timestamp(Instant.now());

        switch (level) {
            case FULL -> b.prompt(sanitize(req.messages()))
                         .response(sanitize(resp));
            case PROMPT_HASH -> b.promptHash(sha256(req.messages()));
            case METADATA -> {} // 只元数据
            case NONE -> { return; }
        }

        auditRepo.save(b.build());
    }
}
```

**关键**：脱敏要在日志写入前做（phone / idCard / 邮箱 / 银行卡）。详见 [`../tutorials/spring-ai-2.0/14-安全工程与红队.md`](../tutorials/spring-ai-2.0/14-安全工程与红队.md)。

## 10. 完整调用链路

把所有层串起来：

```java
// 本代码仅作学习材料参考
@Component
@RequiredArgsConstructor
public class GatewayFacade {

    private final AuthChecker auth;
    private final QuotaService quota;
    private final ExactCache exactCache;
    private final SemanticCache semanticCache;
    private final ModelRouter router;
    private final ResilienceExecutor resilience;
    private final BillingService billing;
    private final AuditLogger audit;
    private final GatewayMetrics metrics;
    private final ObservationRegistry observations;

    @Observed(name = "gateway.chat")
    public ChatResponse chat(GatewayRequest req) {
        // 1. 鉴权
        auth.check(req.tenantId(), req.businessCode());

        // 2. 配额预扣
        QuotaToken token = quota.reserve(req, estimateModel(req), estTokens(req));

        try {
            // 3. 精确缓存
            Optional<ChatResponse> hit = exactCache.lookup(req);
            if (hit.isPresent()) {
                metrics.recordCacheHit("exact");
                return hit.get();
            }

            // 4. 语义缓存
            hit = semanticCache.lookup(req);
            if (hit.isPresent()) {
                metrics.recordCacheHit("semantic");
                return hit.get();
            }

            // 5. 路由选模型
            ModelDescriptor selected = router.route(req, registry.all());
            metrics.recordRoute(req, selected);

            // 6. 调用（带 resilience）
            ChatResponse resp = resilience.execute(selected, req);
            long latency = System.currentTimeMillis() - req.startMs();

            // 7. 写缓存（非流式 / 非工具调用）
            if (!req.stream() && req.tools().isEmpty()) {
                exactCache.put(req, resp);
                semanticCache.put(req, resp);
            }

            // 8. 计费
            billing.record(req, resp, selected.id(), latency, false);

            // 9. 审计
            audit.log(req, resp, auditLevelFor(req));

            // 10. 指标
            metrics.recordSuccess(selected.id(), req.tenantId(), latency,
                                   resp.totalTokens(), resp.cost());

            quota.settle(token, resp.cost());
            return resp;

        } catch (Exception e) {
            metrics.recordError(req, e);
            audit.logError(req, e);
            throw e;
        }
    }
}
```

## 11. 性能与容量规划

### 11.1 网关自身的容量

网关是无状态服务，按 QPS 横向扩容即可。

| 资源 | 建议配置 |
|------|---------|
| CPU | 4C 起步（OTel + JSON 序列化有 CPU 开销） |
| 内存 | 8G（精确缓存 + 业务对象） |
| 实例数 | 按峰值 QPS / 100 估算 |

### 11.2 关键性能优化

1. **语义缓存的 embedding 调用要异步**：不要阻塞主链路。
2. **审计日志异步写**：用 Disruptor 或 Kafka 解耦。
3. **计费批量写库**：单条 insert 太慢，用 batch + 定时 flush。
4. **TTl 配置**：缓存、日志、billing 都要有 TTL，否则存储爆炸。

### 11.3 容灾

- **DB 主从**：billing / quota / audit 不能丢，主从切换要演练
- **缓存降级**：Redis 挂了能降级到本地缓存（命中率会降，但服务不挂）
- **多机房**：网关无状态，可多机房部署 + DNS 切换

## 12. 实战避坑

### 12.1 "路由结果不稳定"

**原因**：依赖了不稳定的信号（实时延迟、健康度），导致同一请求选不同模型。

**解决**：

- 加冷却期（5 分钟内同一业务尽量用同一模型）
- 路由决策记录到上下文，链路里可追溯
- 实时延迟窗口不要太短（30 秒太敏感，5 分钟更稳）

### 12.2 "语义缓存命中错了"

**原因**：阈值设太低，或者跨业务缓存。

**解决**：

- 阈值 ≥ 0.90
- 强制带 `businessCode` filter
- 对"事实类"问题（如"今天天气"）关语义缓存（时效性问题不能缓存）

### 12.3 "Fallback 把账单打爆"

**原因**：主模型 GPT-4o，fallback 是 Claude Opus（更贵），主模型挂了几分钟内所有请求都用了更贵的备份。

**解决**：

- Fallback 链设计时要考虑成本（备份可以能力差不多但**不更贵**）
- Fallback 触发阈值告警（连续 N 次就报警）
- Fallback 期间临时降级（如关掉 QUALITY preference 强制走 COST）

### 12.4 "Token 用量对不上账单"

**原因**：

- 没记 cached 命中（cached 也可能产生部分费用）
- 流式响应的最后一个 chunk 没处理
- Fallback 链中失败模型的消耗没记

**解决**：

- 流式响应聚合 token 用法（`onComplete` 时记一次）
- Fallback 链每段都记 billing（成功 / 失败都记，失败也要算成本）
- 定时对账（每月和云厂商账单 diff，差异 > 5% 报警）

### 12.5 "审计日志把磁盘打爆"

**原因**：FULL 级别审计所有请求，prompt + response 巨大。

**解决**：

- 按业务分级（金融业务 FULL，闲聊业务 METADATA）
- 老数据归档（> 30 天转对象存储）
- 压缩 + 冷存储
