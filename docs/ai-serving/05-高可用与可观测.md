## 0. 认知地图

```
AI 系统的 SRE
├── 传统 SRE 继承
│   ├── 服务可用性
│   ├── 资源监控
│   └── 限流熔断
│
```

**心法**：AI 系统的故障模式比传统系统**多一层"语义故障"**——服务在线但答非所问。监控不能只看 QPS / 延迟，要看**生成质量**。

## 1. AI 系统的特有故障

### 1.1 故障分类

| 类型 | 表现 | 检测方式 |
|------|------|---------|
| **可用性故障** | 服务挂了 / 超时 | 健康检查 / 探活 |
| **性能故障** | 延迟飙升 / QPS 跌 | Prometheus |
| **成本故障** | 单次调用烧钱飙升 | 成本监控 |
| **质量故障** | 回答变差 / 幻觉增多 | 评估闭环 |
| **安全故障** | 越权 / 数据泄露 | 审计 / 红队 |
| **语义故障** | 业务正确但表达突变 | 评估闭环 |

### 1.2 模型漂移

**典型场景**：OpenAI 在某天升级了 GPT-4，你的系统输出风格突变（更啰嗦 / 更简洁），用户投诉。

**关键问题**：

- 厂商**不会主动通知**升级
- 同一 model id 在升级前后**输出可能不同**
- 没有评估闭环就发现不了

### 1.3 Prompt 回归

**典型场景**：

```
团队改了 prompt（觉得更优雅）
  → 单测全过（单测只测基础功能）
  → 上线
  → 3 天后才发现 RAG 召回率掉 20%（prompt 改写变了）
```

Prompt 是代码，但比代码脆弱——任何细微改动都可能影响模型行为。

## 2. 评估闭环：AI 的"自动化测试"

### 2.1 三层评估

```
L1 单元评估     —— 单个 prompt → 单个输出
L2 集成评估     —— 完整 Agent 流程
L3 在线评估     —— 生产流量的实时抽样
```

### 2.2 Golden Set（黄金集）

每条业务维护一个评估集：

```sql
CREATE TABLE eval_case (
    id              SERIAL PRIMARY KEY,
    business_code   VARCHAR(64),
    input           TEXT,           -- 用户输入
    expected        JSONB,          -- 期望输出（多维度）
    category        VARCHAR(32),    -- 分类（normal / edge / adversarial）
    priority        VARCHAR(16),    -- P0 / P1 / P2
    created_at      TIMESTAMP,
    last_updated    TIMESTAMP
);

CREATE TABLE eval_run (
    id              SERIAL PRIMARY KEY,
    business_code   VARCHAR(64),
    version         VARCHAR(64),    -- prompt 版本 + 模型版本
    started_at      TIMESTAMP,
    total           INT,
    passed          INT,
    score           DECIMAL(4, 3)
);
```

### 2.3 评估维度

不只看"对错"，多维度：

```java
// 本代码仅作学习材料参考
public record EvalResult(
    String caseId,
    String actualOutput,
    Map<String, Double> scores   // 多维度评分
) {}

// 维度示例：
Map.of(
    "accuracy",       0.95,    // 准确性
    "completeness",   0.85,    // 完整性
    "tone",           0.90,    // 语气符合
    "safety",         1.0,     // 安全（不输出敏感）
    "latency_ms",     1200.0,  // 延迟
    "cost_usd",       0.002    // 成本
);
```

### 2.4 评估方法

| 方法 | 用法 | 适合 |
|------|------|------|
| **精确匹配** | output == expected | 结构化输出（JSON / SQL） |
| **关键词匹配** | output contains X | FAQ / 知识库 |
| **LLM-as-Judge** | 用强模型评分生成 | 开放式问答 |
| **Embedding 相似度** | output vs expected 的余弦 | 长文本 |
| **规则校验** | JSON schema / SQL 解析 | 工具调用 |
| **人工抽查** | 抽 1% 流量人工标 | 兜底 |

### 2.5 LLM-as-Judge 实现

```java
// 本代码仅作学习材料参考
@Component
public class LlmJudge {

    private final ChatModel judgeModel;   // 用最强的模型（Claude Opus / GPT-4）

    public double score(String input, String expected, String actual) {
        String prompt = """
            请评估以下回答的质量。
            
            用户问题: %s
            参考答案: %s
            待评估答案: %s
            
            评分维度（每项 0-1）：
            1. 准确性（事实是否正确）
            2. 完整性（是否覆盖参考答案要点）
            3. 表达（是否清晰）
            
            返回 JSON: {"accuracy": 0.x, "completeness": 0.x, "expression": 0.x, "overall": 0.x}
            """.formatted(input, expected, actual);

        String resp = judgeModel.call(new Prompt(prompt)).getResult().getOutput().getText();
        return parseScore(resp).overall();
    }
}
```

**注意**：

- LLM-as-Judge 不是 100% 准，但比人工快 1000 倍
- 用**不同模型**做 judge（如用 Claude 评估 GPT 输出，避免同模型偏向）
- judge 成本要控制（只用 P0 / P1 case 做）

### 2.6 持续评估

```java
// 本代码仅作学习材料参考
@Scheduled(cron = "0 0 * * * *")    // 每小时跑一次
public void continuousEval() {
    for (String business : criticalBusinesses()) {
        List<EvalCase> cases = evalRepo.findByBusiness(business);
        EvalRunResult result = runEval(cases);

        // 和上次对比
        EvalRun lastRun = evalRunRepo.findLast(business);
        if (lastRun != null && result.score() < lastRun.score() * 0.95) {
            // 质量下降 5%+，告警
            alert(business, result.score(), lastRun.score());
        }

        evalRunRepo.save(business, currentVersion(), result);
    }
}
```

## 3. Prompt 版本管理

### 3.1 Prompt 是代码

把 prompt 当代码管：

```sql
CREATE TABLE prompt_template (
    id              VARCHAR(64),
    business_code   VARCHAR(64),
    version         VARCHAR(32),     -- semver
    content         TEXT,
    variables       JSONB,
    status          VARCHAR(16),     -- DRAFT / STAGED / PUBLISHED / DEPRECATED
    author          VARCHAR(64),
    created_at      TIMESTAMP,
    PRIMARY KEY (id, version)
);

CREATE TABLE prompt_ab_test (
    test_id         VARCHAR(64),
    business_code   VARCHAR(64),
    variant_a       VARCHAR(64),     -- prompt id
    variant_a_ver   VARCHAR(32),
    variant_b       VARCHAR(64),
    variant_b_ver   VARCHAR(32),
    traffic_split   DECIMAL(3, 2),   -- A:B 比例
    start_time      TIMESTAMP,
    end_time        TIMESTAMP
);
```

### 3.2 版本流转

```
DRAFT  →  STAGED  →  PUBLISHED  →  DEPRECATED
              ↓
        （评估通过）
```

- **DRAFT**：草稿，可以改
- **STAGED**：跑过评估，待发布
- **PUBLISHED**：生产用
- **DEPRECATED**：废弃但保留

### 3.3 Prompt 路由（A/B 测试）

```java
// 本代码仅作学习材料参考
@Service
public class PromptResolver {

    public PromptTemplate resolve(String promptId, String userId) {
        PromptABTest test = abTestRepo.findActive(promptId);
        if (test == null) {
            return repo.findPublished(promptId);
        }

        // 按 userId 哈希分流（同一用户始终同一版本，避免体验跳变）
        double bucket = Math.abs(userId.hashCode() % 100) / 100.0;
        return bucket < test.trafficSplit()
            ? repo.findVersion(test.variantA(), test.variantAVer())
            : repo.findVersion(test.variantB(), test.variantBVer());
    }
}
```

### 3.4 一键回滚

```java
// 本代码仅作学习材料参考
@PostMapping("/api/prompts/{id}/rollback/{version}")
public void rollback(@PathVariable String id, @PathVariable String version) {
    PromptTemplate target = repo.findVersion(id, version);

    PromptTemplate current = repo.findPublished(id);
    repo.updateStatus(id, current.version(), DEPRECATED);
    repo.updateStatus(id, target.version(), PUBLISHED);

    audit.log("Rollback prompt " + id + " from " + current.version() + " to " + version);
}
```

## 5. 监控告警

### 5.1 SLO 定义

| 指标 | SLO |
|------|-----|
| 可用性 | 99.9% (单日 down < 1.5min) |
| 单次响应延迟 p95 | < 3s |
| 单次响应延迟 p99 | < 8s |
| 错误率 | < 1% |
| 评估通过率 | > 90% |
| 缓存命中率 | > 30% |

### 5.2 告警分级

| 级别 | 触发 | 动作 |
|------|------|------|
| P0 | 服务完全不可用 | 立即电话值班 |
| P1 | SLO 大幅破坏（p95 > 10s） | 短信告警 + 30 分钟响应 |
| P2 | SLO 轻微破坏 | 邮件告警 + 2 小时响应 |
| P3 | 异常但不影响用户 | 钉钉群周报 |

### 5.3 告警规则（Prometheus）

```yaml
groups:
  - name: ai-gateway
    rules:
      - alert: HighErrorRate
        expr: |
          rate(gateway_requests{result="error"}[5m])
          / rate(gateway_requests[5m]) > 0.05
        for: 2m
        labels:
          severity: P1
        annotations:
          summary: "Gateway error rate > 5%"

      - alert: HighLatencyP95
        expr: |
          histogram_quantile(0.95, gateway_latency_bucket) > 5
        for: 5m
        labels:
          severity: P1

      - alert: ModelDrift
        expr: |
          eval_score < 0.85
        for: 1h
        labels:
          severity: P1
        annotations:
          summary: "Eval score drop detected"

      - alert: CostAnomaly
        expr: |
          rate(gateway_cost_usd[1h]) > 2 * rate(gateway_cost_usd[1h] offset 1d)
        for: 10m
        labels:
          severity: P2
```

每个告警配套 Runbook：

## 7. Prompt 工程的 SRE

### 7.1 CI 流程

```
开发者改 prompt
  ↓
提交 PR
  ↓
CI 自动跑：
  - 单元评估（Golden Set P0/P1）
  - 安全评估（红队 prompt）
  - 成本评估（token 数变化）
  ↓
评估通过 → 自动 STAGED
  ↓
人工 review → PUBLISHED
  ↓
灰度发布（10% → 50% → 100%）
```

### 7.2 灰度发布

```java
// 本代码仅作学习材料参考
@Service
public class PromptReleaseService {

    public void release(String promptId, String version, double percentage) {
        // 1. 创建 A/B 测试
        PromptABTest test = PromptABTest.builder()
            .testId(UUID.randomUUID().toString())
            .businessCode(promptBusiness(promptId))
            .variantA(promptId)
            .variantAVer(currentPublishedVersion(promptId))
            .variantB(promptId)
            .variantBVer(version)
            .trafficSplit(percentage / 100.0)   // 新版本流量占比
            .build();
        abTestRepo.save(test);

        // 2. 监控新版本的指标
        scheduleMonitoring(test);
    }

    private void scheduleMonitoring(PromptABTest test) {
        // 24 小时后自动评估
        scheduler.schedule(() -> evaluateAB(test), Duration.ofHours(24));
    }

    private void evaluateAB(PromptABTest test) {
        EvalResult a = collectResults(test.variantA(), test.variantAVer());
        EvalResult b = collectResults(test.variantB(), test.variantBVer());

        if (b.score() > a.score() * 1.02) {
            // 新版本更好，全量
            promote(test);
        } else if (b.score() < a.score() * 0.95) {
            // 新版本更差，回滚
            rollback(test);
        } else {
            // 持续观察 7 天
            extendTest(test, Duration.ofDays(7));
        }
    }
}
```

prompt 本身没变，但模型升级了导致效果变化。这时要靠**评估闭环**自动发现（§2.6）。

### 8.3 内容安全

```java
// 本代码仅作学习材料参考
@Component
public class ContentSafetyFilter {

    private final List<ContentModerationStrategy> strategies;

    public ChatResponse filter(ChatResponse resp) {
        String text = resp.getResult().getOutput().getText();
        for (ContentModerationStrategy s : strategies) {
            ContentCheck check = s.check(text);
            if (!check.passed()) {
                auditLog(resp, check);
                return safeFallback(resp, check);
            }
        }
        return resp;
    }
}
```

**策略**：

- 关键词过滤（最简单）
- 第三方审核 API（阿里云 / Azure Content Safety）
- LLM-as-Judge（用专门模型判安全）

## 9. 实战避坑

### 9.1 "评估集本身过时"

**原因**：业务在变，eval case 没更新。

**解决**：

- 定期（每月）从生产流量抽样补充 eval case
- 废弃的 eval case 标记但不删（历史追溯）
- eval case 的覆盖率也要监控（核心场景 100% 覆盖）

### 9.2 "LLM-as-Judge 不靠谱"

**原因**：judge 模型本身有偏向。

**解决**：

- 用最强模型做 judge
- 多 judge 投票
- 人工校准：每月抽 100 条人工复核 judge 评分

### 9.3 "告警太多变成噪音"

**原因**：阈值太敏感，天天响。

**解决**：

- 多窗口检查（5 分钟连续触发才告警）
- 分级（P0/P1/P2/P3）只让 P0/P1 推短信
- 每月复盘告警准确率，调阈值

### 9.4 "灰度发布回滚不及时"

**原因**：发现质量问题但流程慢。

**解决**：

- 自动化评估：发布后 24 小时自动跑评估，不达标自动回滚
- 紧急回滚按钮：值班人能一键回滚，不需要审批

### 9.5 "trace 数据量爆炸"

**原因**：所有请求都埋 span，QPS 高时存储爆。

**解决**：

- 采样：tail-based sampling（异常请求 100% 采样，正常请求 1%）
- TTL：trace 数据 7 天 / 30 天后归档
- 只在异常场景详细埋点
