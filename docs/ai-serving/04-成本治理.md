## 1. 为什么 AI 项目最容易失控

传统 SaaS 的成本主要是固定服务器开销，可预测。AI 项目不同：

- **按 token 计费**：一个用户一次对话可能花 $0.001 也可能 $1
- **token 难以预估**：依赖 prompt 长度 + 模型 + 工具调用次数
- **业务方不感知成本**：写 prompt 的人不付钱
- **质量与成本强相关**：便宜模型回答质量差，业务会自动往贵的迁移

**结果**：很多团队上线 3 个月账单翻 10 倍，老板才发现问题。

### 2.2 业务码体系

业务码（businessCode）是**最重要**的标签，决定成本归因到哪。

```
businessCode 设计建议：

业务线.子业务.功能

示例：
  cs.chatbot                客服机器人
```

### 2.4 聚合任务

```java
// 本代码仅作学习材料参考
@Scheduled(cron = "0 5 0 * * *")   // 每天 00:05 跑
public void aggregateDaily() {
    LocalDate yesterday = LocalDate.now().minusDays(1);

    jdbc.update("""
        INSERT INTO cost_daily
        SELECT DATE(called_at), tenant_id, business_code, agent_id,
               COUNT(*), SUM(prompt_tokens), SUM(completion_tokens),
               SUM(cost_usd),
               AVG(CASE WHEN cache_hit THEN 1.0 ELSE 0.0 END)
        FROM cost_record
        WHERE called_at >= ? AND called_at < ?
        GROUP BY DATE(called_at), tenant_id, business_code, agent_id
        ON CONFLICT (date, tenant_id, business_code, agent_id) DO UPDATE SET
            call_count = EXCLUDED.call_count,
            cost_usd = EXCLUDED.cost_usd
        """, yesterday, yesterday.plusDays(1));
}
```

### 3.3 Redis 实时计数

DB 计数有延迟，配额检查要用 Redis：

```java
// 本代码仅作学习材料参考
@Component
@RequiredArgsConstructor
public class QuotaGuard {

    private final StringRedisTemplate redis;
    private final QuotaRepository quotaRepo;

    public void checkAndReserve(CostTag tag, int estTokens) {
        String dayKey = "quota:" + tag.tenantId() + ":" + tag.businessCode() + ":" + LocalDate.now();
        String monthKey = "quota:" + tag.tenantId() + ":" + tag.businessCode() + ":" + YearMonth.now();

        Long dailyUsed = redis.opsForValue().increment(dayKey, estTokens);
        Long monthlyUsed = redis.opsForValue().increment(monthKey, estTokens);

        Quota quota = quotaRepo.find(tag.tenantId(), tag.businessCode());
        if (quota.dailyTokens() > 0 && dailyUsed > quota.dailyTokens()) {
            redis.opsForValue().decrement(dayKey, estTokens);   // 回滚
            throw new QuotaExceededException("Daily quota exceeded for " + tag.businessCode());
        }
        if (quota.monthlyTokens() > 0 && monthlyUsed > quota.monthlyTokens()) {
            redis.opsForValue().decrement(monthKey, estTokens);
            throw new QuotaExceededException("Monthly quota exceeded");
        }
    }
}
```

```java
public ModelPreference decidePreference(CostTag tag, ModelPreference requested) {
    Quota q = quotaRepo.find(tag.tenantId(), tag.businessCode());
    double usageRate = currentUsage(q) / q.monthlyLimit();
```

## 4. 优化层：5 大杠杆

### 4.1 杠杆 1：Prompt 瘦身

#### 4.1.1 现状

很多团队 system prompt 写到 3000 token，里面 80% 是废话。

```
❌ 冗长 prompt（1500 tokens）：
"你是一个非常有礼貌、专业、友善的客服代表。在回答用户问题时，
 请始终使用敬语，称呼用户为'尊敬的客户'。回答前先问好，
 回答后再次询问是否还有其他需要帮助的。每个回答不少于 200 字..."
```

```
✅ 精简 prompt（200 tokens）：
"客服。规则：1) 用敬语 2) 称呼'尊敬的客户'
 3) 回答前后寒暄 4) 详细回答（200字+）"
```

效果几乎一样，**成本差 7 倍**。

#### 4.1.2 自动检测

```java
// 本代码仅作学习材料参考
@Component
public class PromptAuditService {

    public PromptReport audit(String systemPrompt) {
        int tokens = tokenCount(systemPrompt);
        return new PromptReport(
            tokens,
            tokens > 500 ? "过长" : "合理",
            detectDuplicates(systemPrompt),     // 重复内容检测
            detectFillerWords(systemPrompt)     // 废话词检测
        );
    }
}
```

CI 阶段拦截 prompt 巨大变更（如突然从 500 → 2000 tokens）。

### 4.2 杠杆 2：模型路由（详见 01 篇）

按场景选合适 Tier：

| 场景 | 推荐 Tier | 理由 |
|------|----------|------|
| FAQ 机器人 | LIGHTWEIGHT | 简单问答用便宜模型 |
| 客服转人工判断 | MAINSTREAM | 平衡 |
| 法律 / 医疗 | FRONTIER | 不能错 |
| 内部工具 | LIGHTWEIGHT | 错得起 |
| 代码生成 | FRONTIER | 错不起 |
| 改写 / 摘要 | MAINSTREAM | 一般任务 |

**常见反模式**：所有任务都用 GPT-4。每月多花 10x。

### 4.3 杠杆 3：缓存（详见 01 篇）

- 精确缓存：50-90% 命中（高频请求）
- 语义缓存：30-60% 命中（容错场景）

10x 缓存命中 = 10x 成本节省。

### 4.4 杠杆 4：上下文裁剪

#### 4.4.1 RAG 场景

RAG 检索 topK=10 vs topK=3：成本差 3 倍，但召回率差不多。

```java
// 本代码仅作学习材料参考
SearchRequest.builder()
    .topK(3)                                  // 不用 10
    .similarityThreshold(0.75)
    .build();
```

#### 4.4.2 长对话场景

ChatMemory 不裁剪会让对话越拉越长：

```java
// 本代码仅作学习材料参考
ChatMemory memory = MessageWindowChatMemory.builder()
    .repository(repo)
    .maxMessages(20)             // 只保留最近 20 条
    .build();
```

或者用 SummarizingMemory：每隔 N 条把前面的总结成 1 条（成本 vs 质量的权衡）。

#### 4.4.3 工具调用结果

工具返回的数据常常巨大（如 1000 行 SQL 结果）。**裁剪后再喂给 LLM**：


批量 embedding（详见 02 篇 §7.2.1）：

### 5.1 业务 ROI 分析

每个业务的成本 vs 收益：

```sql
SELECT
    b.business_code,
    b.cost_usd,
    b.call_count,
    b.cost_usd / b.call_count AS per_call_cost,
    -- 业务指标（自定义）
    m.conversion_rate,
    m.satisfaction_score,
    -- ROI
    m.revenue_usd / b.cost_usd AS roi
FROM cost_daily_aggregate b
LEFT JOIN business_metric m ON b.business_code = m.business_code
WHERE b.date >= '2026-06-01'
GROUP BY b.business_code;
```

```java
// 用过去 30 天的数据预测未来 30 天
public CostForecast forecast(String tenantId, int daysAhead) {
    List<Double> history = past30DaysCost(tenantId);

    // 简单线性回归（够用了，不用上 ML）
    double[] params = linearRegression(history);
    double slope = params[0];
    double intercept = params[1];

    List<Double> forecast = new ArrayList<>();
    for (int i = 0; i < daysAhead; i++) {
        forecast.add(slope * (history.size() + i) + intercept);
    }
    return new CostForecast(forecast, slope > 0 ? "上升" : "下降");
}
```

## 7. 实战避坑

### 7.1 "成本核算对不上账单"

**原因**：

- 流式响应最后一个 chunk 没记
- Fallback 链失败调用没记
- 缓存命中没扣减（应该记 0 成本但有调用记录）

**解决**：

- 流式响应在 `onComplete` 时统一记一次
- Fallback 每段都记 billing（含失败的）
- 缓存命中也记 cost=0 的 record，便于分析命中率
- 每月对账：和云厂商账单 diff > 5% 报警

### 7.2 "配额被并发请求穿透"

**原因**：Redis incr 是异步的，高并发下可能 N 个请求同时通过检查。

**解决**：

- 用 Redis Lua 脚本原子化"检查+扣减"
- 设置硬上限低于真实配额（如真实 100 万，硬上限 95 万），留缓冲

### 7.3 "优化后效果反而变差"

**原因**：把客服切到便宜模型，回答质量下降，用户重复问，总调用次数 +50%，成本反而更高。

**解决**：

- 切换前做 A/B 测试
- 监控用户满意度 / 任务完成率，不只看 token 数
- 留有"切回去"的开关

### 7.4 "测试环境的成本混入生产统计"

**原因**：dev 调用没打标签或打错标签。

**解决**：

- 测试环境强制 `businessCode` 前缀 `dev.`
- 聚合时按 prefix 过滤
- 测试环境用便宜模型 / 本地模型

## 9. 理解检查
