## 0. 认知地图

```
向量库选型决策树
├── 数据规模
│   ├── < 10 万 → pgvector / Redis / 进程内（够用）
│   ├── 10 万 - 1000 万 → Milvus / Qdrant / Weaviate（专用）
│   └── > 1000 万 → Milvus 集群 / Vespa（分布式）
│
├── 调用模式
│   ├── 纯 ANN 查询 → 任何库都行
│   ├── 需要结构化 filter → pgvector / Qdrant（hybrid 表现好）
│   └── 需要事务 + 业务数据 → pgvector（关系库原生）
│
├── 部署偏好
│   ├── 已有 PostgreSQL → pgvector（零成本接入）
│   ├── 已有 Redis Stack → Redis Vector
│   ├── 想要云托管 → Pinecone / Zilliz Cloud
│   └── 想要自建 → Milvus / Qdrant / Weaviate
│
└── 性能需求
    ├── 写多读少 → Milvus（写性能强）
    ├── 读多写少 → Qdrant（读优化好）
    └── 都重要 → pgvector（最均衡的"够用"选项）
```

**心法**：选型不是选"最强"，是选"在你这个规模 + 这个调用模式下最合适"。pgvector 满足 80% 场景，先用它跑起来，遇到瓶颈再迁移。

## 2. 索引算法基础

向量检索本质是 ANN（Approximate Nearest Neighbor）问题。精确找最近邻要 O(N)，不可行，所以都用近似算法。

### 2.1 三大算法

| 算法 | 原理 | 召回率 | 内存 | 写入速度 | 查询速度 |
|------|------|--------|------|---------|---------|
| **Flat**（暴力） | 全扫描 | 100% | 中 | 快 | 慢 |
| **IVF**（倒排） | 聚类后查近邻簇 | 90-95% | 低 | 中 | 中 |
| **HNSW**（图） | 多层小世界图 | 95-99% | 高 | 慢 | 快 |
| **PQ/SQ**（量化） | 压缩向量 | 85-92% | 极低 | 中 | 中 |
| **DiskANN** | 磁盘索引 | 95%+ | 极低 | 慢 | 中（依赖 SSD） |

### 2.2 选索引的快速判断

```
数据 < 10 万 → Flat（够用，准确率 100%）
数据 10 万 - 100 万 → HNSW（内存够就用）
数据 > 100 万 + 内存紧张 → IVF + PQ
数据 > 1000 万 + 频繁更新 → HNSW（仍是首选）
数据极大 + 冷数据多 → DiskANN
```

### 2.3 HNSW 调参（最常用）

HNSW 两个关键参数：

| 参数 | 含义 | 默认 | 影响 |
|------|------|------|------|
| `M` | 每个节点的连接数 | 16 | 越大召回越高，内存越大 |
| `ef_construction` | 构建时候选数 | 200 | 越大构建越慢，质量越好 |
| `ef_search` | 查询时候选数 | - | 越大召回越高，查询越慢 |

**经验**：

- 通用场景 `M=16, ef_construction=200, ef_search=64` 召回率 ~95%
- 高召回要求 `M=32, ef_construction=400, ef_search=128` 召回率 ~99%
- 高 QPS 要求 `M=8, ef_construction=100, ef_search=32` 召回率 ~85%（牺牲召回换速度）

## 3. pgvector 实战（首选推荐）

### 3.1 为什么先讲 pgvector

- **已有 PostgreSQL 直接加扩展**，无运维成本
- **支持事务**：embedding 和业务数据同库
- **支持 SQL filter**：可以 `WHERE tenant_id = 'A' AND ...`
- **Spring AI 原生支持**：开箱即用
- **满足 80% 场景**：1000 万向量以内毫无压力

**唯一缺点**：超大规模（> 亿级）需要专业向量库。

### 3.2 安装与建表

```sql
-- 安装扩展（PG 14+）
CREATE EXTENSION vector;

-- 建表
CREATE TABLE documents (
    id UUID PRIMARY KEY,
    tenant_id VARCHAR(64),
    business_code VARCHAR(64),
    content TEXT,
    metadata JSONB,
    embedding vector(1536),     -- 维度按 embedding 模型定
    created_at TIMESTAMP DEFAULT NOW()
);

-- HNSW 索引
CREATE INDEX documents_embedding_idx
ON documents USING hnsw (embedding vector_cosine_ops)
WITH (m = 16, ef_construction = 200);

-- 复合 filter 索引（关键：加速 hybrid 查询）
CREATE INDEX documents_tenant_business_idx
ON documents (tenant_id, business_code);
```

### 3.3 Spring AI 接入

```yaml
spring:
  ai:
    vectorstore:
      pgvector:
        index-type: hnsw
        distance-type: cosine_distance
        dimensions: 1536
        schema-name: public
        table-name: documents
        initialize-schema: true
```

```java
// 本代码仅作学习材料参考
@Service
@RequiredArgsConstructor
public class DocumentVectorStore {

    private final VectorStore vectorStore;

    public void add(List<Document> docs) {
        vectorStore.add(docs);
    }

    public List<Document> search(String query, String tenantId, String businessCode) {
        SearchRequest req = SearchRequest.builder()
            .query(query)
            .topK(10)
            .similarityThreshold(0.75)
            .filterExpression("tenant_id == '" + tenantId + "' && business_code == '" + businessCode + "'")
            .build();
        return vectorStore.similaritySearch(req);
    }
}
```

### 3.4 pgvector 性能基准（参考）

环境：AWS r6i.4xlarge（16C 128G），1536 维向量

| 数据量 | 索引 | 索引大小 | 查询 p95 | 召回率 |
|--------|------|---------|---------|--------|
| 10 万 | HNSW | 800 MB | 8 ms | 98% |
| 100 万 | HNSW | 8 GB | 25 ms | 96% |
| 500 万 | HNSW | 40 GB | 60 ms | 95% |
| 1000 万 | HNSW | 80 GB | 120 ms | 93% |

**结论**：

- < 500 万：pgvector 完全够
- 500 万 - 1000 万：需要调优（ivfflat + 大内存）
- > 1000 万：考虑专用向量库

### 3.5 pgvector 调优技巧

1. **配置 `maintenance_work_mem`**：建索引时给足内存

   ```sql
   SET maintenance_work_mem = '4GB';
   CREATE INDEX ... USING hnsw ...;
   ```

2. **配置 `hnsw.ef_search`**：查询时动态调整

   ```sql
   SET hnsw.ef_search = 100;  -- 全局
   -- 或单查询
   BEGIN;
   SET LOCAL hnsw.ef_search = 50;
   SELECT ...;
   COMMIT;
   ```

3. **预过滤再 ANN**：用 SQL 把候选集缩小

   ```sql
   -- ❌ 慢：先 ANN 再过滤
   SELECT * FROM documents ORDER BY embedding <-> $1 LIMIT 10 OFFSET 0;
   -- 然后在应用层 filter tenant_id

   -- ✅ 快：先 filter 再 ANN
   SELECT * FROM documents
   WHERE tenant_id = $2 AND business_code = $3
   ORDER BY embedding <-> $1 LIMIT 10;
   ```

   注意：pgvector 2.0+ 的 partial index 才支持这种"先 filter 后 ANN"，老版本要建带 WHERE 的 partial index。

## 4. Milvus 实战（大规模首选）

### 4.1 什么时候该上 Milvus

- 数据 > 1000 万
- 需要高写入吞吐（如日志/对话历史向量化）
- 需要多种索引共存（一个 collection 用 HNSW，另一个用 IVF_SQ8）
- 有运维 Kubernetes 的能力

### 4.2 部署形态

```
Milvus 集群
├── Proxy（接入层，无状态）
├── QueryNode（读节点）
├── DataNode（写节点）
├── IndexNode（建索引）
└── 依赖
    ├── etcd（元数据）
    ├── MinIO / S3（数据存储）
    └── Pulsar / Kafka（消息）
```

**最小生产配置**：3 节点 etcd + 3 节点 MinIO + 2 Proxy + 4 QueryNode + 2 DataNode + 2 IndexNode。**很重**，没有 K8s 团队不建议碰。

轻量替代：**Milvus Lite**（嵌入式，单机）或 **Zilliz Cloud**（托管）。

### 4.3 Spring AI 接入

```yaml
spring:
  ai:
    vectorstore:
      milvus:
        client:
          host: milvus.internal
          port: 19530
          username: ${MILVUS_USER}
          password: ${MILVUS_PASSWORD}
        database-name: default
        collection-name: documents
        embedding-dimension: 1536
        index-type: HNSW
        metric-type: COSINE
        index-parameters: '{"M": 16, "efConstruction": 200}'
        search-parameters: '{"ef": 64}'
```

### 4.4 Milvus 的特殊能力

| 能力 | 用途 |
|------|------|
| **分区（Partition）** | 按租户分区，查询只扫特定分区 |
| **分片（Shard）** | 写入水平拆分，提升吞吐 |
| **动态字段** | 不需要预定义 metadata schema |
| **标量字段索引** | 加速 hybrid filter |
| **多向量字段** | 一个 collection 存多个 embedding（如文本 + 图像） |

**多租户最佳实践**：

```python
# 用 partition 做租户隔离
collection.create_partition(partition_name=f"tenant_{tenant_id}")
collection.insert(data, partition_name=f"tenant_{tenant_id}")
collection.search(query, partition_names=[f"tenant_{tenant_id}"])
```

> 单 collection partition 数上限 4096，超过要按 collection 拆。

## 5. Qdrant 实战（Rust 性能怪兽）

### 5.1 Qdrant 的优势

- **Rust 实现**，性能优秀
- **filter 性能强**：hybrid 查询比 pgvector / Milvus 都快
- **资源占用小**：单节点能扛百万级向量
- **API 友好**：REST + gRPC

### 5.2 适合的场景

- 中等规模（100 万 - 1000 万）
- 高 QPS + 复杂 filter
- 想要单节点起步，按需扩集群

### 5.3 Spring AI 接入

```yaml

    vectorstore:

        host: qdrant.internal
        port: 6334
        use-tls: false

```

### 5.4 Qdrant 的 Payload（filter 字段）


## 7. Embedding 调用治理

向量库本身只是"存"。**真正的成本大头在 embedding 调用**——每次写入和查询都要算一遍。

### 7.1 成本陷阱

OpenAI `text-embedding-3-large`：$0.13 / 1M tokens。看着不贵，但：

- RAG 系统启动一次性索引 1 亿文档（平均 1000 tokens）= 1 亿 tokens = **$13,000**
- 每天用户查询 10 万次（query 改写 + 原始）= 20 万 query × 50 tokens = 1 亿 tokens / 天 ≈ **$13 / 天**

写入是大头，查询相对可控。**治理重点在写入侧**。

### 7.2 三层治理

#### 7.2.1 Embedding 缓存

```java
// 本代码仅作学习材料参考
@Component
@RequiredArgsConstructor
public class CachedEmbeddingModel implements EmbeddingModel {

    private final EmbeddingModel delegate;     // OpenAI / 本地
    private final Cache<String, float[]> cache;

    @Override
    public float[] embed(Document document) {
        String key = "emb:" + DigestUtils.sha256Hex(document.getText());
        float[] cached = cache.getIfPresent(key);
        if (cached != null) return cached;

        float[] vec = delegate.embed(document);
        cache.put(key, vec);
        return vec;
    }

    @Override
    public List<float[]> embed(List<Document> documents) {
        // 批量：先查缓存，未命中的再批量调 API
        List<Document> missing = new ArrayList<>();
        List<float[]> result = new ArrayList<>();
        for (Document d : documents) {
            float[] hit = cache.getIfPresent(hashKey(d));
            if (hit != null) result.add(hit);
            else missing.add(d);
        }
        if (!missing.isEmpty()) {
            List<float[]> fresh = delegate.embed(missing);
            // 写回缓存
            for (int i = 0; i < missing.size(); i++) {
                cache.put(hashKey(missing.get(i)), fresh.get(i));
            }
            result.addAll(fresh);
        }
        return result;
    }
}
```

**缓存命中率参考**：RAG 系统通常 50-80%（同一文档不会重复 embedding，但同一 query 可能反复 embedding）。

#### 7.2.2 增量更新

不要每次 RAG 启动都重新 embed 全量。维护"已处理标记"：

```sql
CREATE TABLE embedding_progress (
    source_table VARCHAR(64),
    source_id    VARCHAR(64),
    content_hash VARCHAR(64),    -- 内容变了才重新 embed
    embedded_at  TIMESTAMP,
    PRIMARY KEY (source_table, source_id)
);
```

只有 `content_hash` 变了的记录才重新 embed。

#### 7.2.3 本地 embedding 模型

高频低成本场景用本地模型：

- `bge-large-zh-v1.5`（中文，1024 维）—— 中文场景性价比之选
- `bge-m3`（多语言，1024 维）—— 多语言通用
- `gte-large`（多语言，1024 维）—— 阿里达摩

用 Ollama 或 Infinity 部署，**零 token 成本**（但有 GPU/服务器成本）。

```yaml
spring:
  ai:
    ollama:
      base-url: http://ollama.internal:11434
      embedding:
        model: bge-m3
```

### 7.3 Embedding 维度选择

| 模型 | 维度 | 性价比 | 适用 |
|------|------|--------|------|
| text-embedding-3-small | 1536 | 高 | 通用首选 |
| text-embedding-3-large | 3072 | 中 | 高质量检索 |
| text-embedding-3-large + MRL | 768/256 | 极高 | 量大降维（见下） |
| bge-m3 | 1024 | 极高 | 本地多语言 |

**MRL（Matryoshka Representation Learning）**：OpenAI 3 系列支持"截断"——3072 维向量砍掉后 256 维仍可用，召回率只降一点。**1/12 存储 + 1/12 查询成本**。

```java
// 本代码仅作学习材料参考
// 调用时指定 dimensions

    List.of(new EmbeddingRequest(List.of("text"), Map.of("dimensions", 256)))
);
```

## 8. 数据治理

### 8.1 向量的生命周期

向量不是越多越好。要管理生命周期：

```
新建 → 使用 → 沉淀 → 归档 → 删除
```

| 阶段 | 处理 |
|------|------|
| 新建 | embed + 写库 + 标 metadata |
| 使用 | 频繁查询 |
| 沉淀 | 30 天没被命中 → 降级（移到冷库） |
| 归档 | 90 天没命中 → 移到对象存储 |
| 删除 | 1 年没命中 / 业务下线 → 删 |

### 8.2 评估检索质量

定期评估召回率，避免索引漂移：

```java
// 本代码仅作学习材料参考
@Component
public class RetrievalQualityEvaluator {

    // 维护一个 golden set（query + 期望命中的文档 ID）
    private final List<EvalCase> goldenSet;

    public QualityReport evaluate() {
        int hits = 0;
        for (EvalCase c : goldenSet) {
            List<Document> results = vectorStore.similaritySearch(
                SearchRequest.builder().query(c.query()).topK(10).build());
            Set<String> ids = results.stream().map(Document::getId).collect(toSet());
            if (ids.contains(c.expectedDocId())) hits++;
        }
        return new QualityReport(hits, goldenSet.size(), (double) hits / goldenSet.size());
    }
}
```

**质量基线**：

- 召回率 < 80%：可能要换 embedding 模型 / 调索引参数
- 召回率波动 > 5%：可能数据分布变了，需要重新 embed

### 8.3 多租户隔离

向量库的多租户有三种模式：

| 模式 | 实现 | 优点 | 缺点 |
|------|------|------|------|
| **共享 collection + filter** | tenant_id 在 metadata | 简单 | filter 开销 |
| **per-tenant partition** | Milvus partition / Qdrant payload | 中等 | partition 数有上限 |
| **per-tenant collection** | 完全隔离 | 性能最好 | 数量爆炸难管理 |

**生产推荐**：小租户用 filter，大租户用独立 collection，中等的用 partition。

## 9. 实战 benchmark

### 9.1 测试方案

```
数据集：随机生成 100 万 1536 维向量（模拟 OpenAI text-embedding-3-small）
查询集：1000 个 query
环境：AWS c6i.4xlarge（16C 32G）
指标：召回率（vs Flat 暴力搜索）+ p50/p95/p99 latency + QPS
```

### 9.2 参考结果（实测数值会因配置而异，仅供参考）

| 库 | 索引 | 召回率@10 | p50 (ms) | p95 (ms) | QPS | 内存占用 |
|---|------|----------|---------|---------|-----|---------|
| pgvector | HNSW M=16 | 96.2% | 12 | 28 | 800 | 6 GB |
| pgvector | IVFFlat | 92.5% | 18 | 45 | 600 | 4 GB |
| Qdrant | HNSW M=16 | 96.5% | 8 | 18 | 1200 | 5 GB |
| Milvus | HNSW M=16 | 96.3% | 10 | 22 | 1000 | 6 GB |
| Milvus | IVF_SQ8 | 90.1% | 14 | 30 | 900 | 2 GB |

**结论**：

- 同样 HNSW，Qdrant 最快（Rust + 内存布局优化）
- pgvector 性能不差，差距在 2x 以内，**简单场景完全够**
- 想省内存用 IVF_SQ8，但召回率会掉

### 9.3 怎么自己做 benchmark

```java
// 本代码仅作学习材料参考
@SpringBootTest
class VectorStoreBenchmark {

    @Autowired VectorStore store;

    @Test
    void benchmark() throws Exception {
        // 1. 准备数据
        List<Document> docs = generateDocs(1_000_000);
        store.add(docs);

        // 2. 等索引构建完成
        Thread.sleep(60_000);

        // 3. 准备查询集
        List<String> queries = loadGoldenQueries();

        // 4. 跑查询
        List<Long> latencies = new ArrayList<>();
        int hits = 0;
        for (String q : queries) {
            long t0 = System.nanoTime();
            List<Document> r = store.similaritySearch(
                SearchRequest.builder().query(q).topK(10).build());
            long t1 = System.nanoTime();
            latencies.add((t1 - t0) / 1_000_000);
            if (containsExpected(r, q)) hits++;
        }

        // 5. 输出报告
        Collections.sort(latencies);
        System.out.printf("Recall: %.1f%%%n", 100.0 * hits / queries.size());
        System.out.printf("p50: %d ms%n", latencies.get(latencies.size() / 2));
        System.out.printf("p95: %d ms%n", latencies.get((int)(latencies.size() * 0.95)));
    }
}
```

## 10. 实战避坑

### 10.1 "召回率突然变低"

**原因 1**：数据分布变了，索引漂移。

**解决**：定期重建索引（pgvector：`REINDEX INDEX ...`；Milvus：compact 操作）。

**原因 2**：`ef_search` / `nprobe` 被改小了。

**解决**：在配置里固化，不允许运行时改。

### 10.2 "查询越来越慢"

**原因**：向量太多但索引没更新。

**解决**：

- HNSW 增量插入会有"碎片"，定期重建
- 设置 `autovacuum` 更激进（pgvector）
- 监控索引深度（Milvus 的 sealed segments 数量）

### 10.3 "embedding 费用失控"

**原因**：每次 RAG 启动都全量 re-embed。

**解决**：

- 用 §7.2.2 的 `embedding_progress` 表跟踪
- 维护内容哈希，只对变化的文档 re-embed
- 强制本地 embedding 模型做兜底

### 10.4 "filter 把性能拖死"

**原因**：hybrid filter 不走索引。

**解决**：

- 给 filter 字段建独立索引（不只是 ANN 索引）
- 用 partial index（pgvector）
- 用 payload index（Qdrant）
- 必要时改成两阶段查询：先 SQL filter 拿候选集 ID，再用向量检索这些 ID

### 10.5 "向量库宕机业务全瘫"

**原因**：单点。

**解决**：

- 主从复制（pgvector：PG streaming replication；Qdrant：replication factor）
- 客户端兜底（向量库挂了降级到关键词搜索）
- 多机房部署
