> 理念：**先动手，再补理论**。每个阶段都对应可运行的项目，遇到概念卡点再去 `reference/` 查阅。
>
> **学习模式**：LangChain4j 入门（API 设计最像传统 Java，建立心智模型）→ Spring AI 主线（与 Spring Boot 深度融合，最终交付栈）。
>
> **⚠️ 重要校正（2026-07-13 最新）**：基于 [`reference/选型与对比/09`](../reference/选型与对比/09-企业级Java-AI架构选型真相.md) + [`reference/选型与对比/10`](../reference/选型与对比/10-SpringAI-vs-LangChain4j何时用何框架.md) 的调研：
> - **Spring AI 2.0.0 GA 已于 2026-06-12 发布**（基于 Spring Boot 4 + Jackson 3 + JSpecify），`ToolCallingAdvisor` 原生支持 Agent Loop，LangChain4j 优势大幅缩水
> - 企业级 Java AI 以**单框架（Spring AI）+ Anthropic Workflow 模式**为主流，"混用"是理论范式不是工程标配
> - **LangChain4j 仅作阶段 1 入门**（理解 LLM 调用本质），**不再作为最终技术栈**
>
> **三个学习方向（按 ROI 排序）**：
> - **方向 1（最高 ROI）**：Spring AI 2.0 全栈 + Anthropic 5 大 Workflow 模式 + MCP（80% 企业场景）
> - **方向 2（中 ROI）**：编排引擎（Spring AI Alibaba Graph 或 LangGraph4j）
> - **方向 3（高 ROI 赌性强）**：MCP / A2A 协议生态、Agent 可靠性工程
>
> **本仓库当前技术栈**：Spring Boot 3.5.10 + Spring AI 1.0.0 + LangChain4j 1.0.1 + JDK 21。
> **阶段 3.5 起升级到**：Spring Boot 4 + Spring AI 2.0 + JDK 21。

## 阶段 0.5：LLM 基础速通（Day 1-3）

### 目标
作为 Java 工程师，把 LLM 当成"一个有概率出错的远程 RPC"来建立心智模型，避免后续阶段被概念卡住。

### ⚠️ 为什么插这一段
你是 Java 工程师，但计划里之前没有专门讲 LLM 本身的机制（tokenization/embedding/温度等）。不补这一段，后面写 prompt、调 RAG、看 token 计费会一直懵。

### 核心概念速通

| 概念 | Java 类比 | 学到什么程度 |
|------|----------|------------|
| **Token** | 字符串字段的"长度单位" | 知道 1 token ≈ 0.75 英文单词 / 0.5 汉字；知道为什么按 token 计费 |
| **Context Window** | RPC 请求 body 的最大长度 | 知道模型上下文窗口大小（4K/32K/128K/200K），影响能塞多少历史 |
| **Temperature / Top-p** | RPC 的随机种子 | 知道 temp=0 输出稳定、temp=1 输出有创造性；知道何时调 |
| **System / User / Assistant** | RPC 的 header / body / response | 知道三种 role 的作用，能写出最小可用 prompt |
| **Embedding** | 把文档转成数据库索引的"向量指纹" | 知道 embedding 是 RAG 的基础；维度（768/1024/1536）影响精度和成本 |
| **Function Calling / Tool Use** | RPC 的"反向调用"——服务端告诉客户端执行本地方法 | 知道 LLM 输出结构化 JSON 让 Java 反射调用 |
| **Streaming（SSE）** | RPC 的 chunked transfer | 知道为什么 token 流式返回能改善体验 |
| **Prompt Cache** | HTTP 缓存 | 知道 prompt 命中缓存能省 10 倍钱（Anthropic Prompt Caching） |

### 任务清单

- [ ] 读 [`reference/心智模型与决策树/01-心智模型与决策树.md`](../reference/心智模型与决策树/01-心智模型与决策树.md) 的"心智模型"部分
- [ ] 用 LM Studio Playground 或 DeepSeek 控制台**手动**测：
  - [ ] 同一个 prompt，temperature=0 跑 3 次，看输出是否一致
  - [ ] 同一个 prompt，temperature=1 跑 3 次，看变化
  - [ ] 写一个带 `@Tool` 描述的 prompt，观察 LLM 是否输出 function call JSON
- [ ] 理解"上下文窗口" vs "输出窗口"的区别（输入 + 输出 ≤ 总窗口）
- [ ] 知道 Token 计费：input tokens（便宜）/ output tokens（贵 3-5 倍）/ cache read（便宜 10 倍）

### 验收
- ✅ 能解释"为什么 LLM 是无状态的"（一句话：每次调用都是独立 HTTP 请求）
- ✅ 能解释"为什么多轮对话需要 ChatMemory"（一句话：HTTP 无状态，要客户端塞回历史）
- ✅ 能解释"temperature=0 时为什么输出还是会变"（一句话：top-p 采样 + 模型浮点不确定性）

## 阶段 3.5：升级到 Spring AI 2.0（Week 4 末，3-5 天）

### ⚠️ 为什么插这一阶段（2026-07-13 新增）

原方案阶段 5 才升级，但阶段 4 在 1.0 上自研 `AgentLoop`，**升级后这套自研代码要废弃**——白学。

提前到阶段 3.5：
- ✅ 阶段 4 直接用 `ToolCallingAdvisor` 原生 Agent Loop，无需自研
- ✅ 学习收益最大化（学的是最终技术栈，不是过渡方案）
- ✅ 升级成本可控（3-5 天，依赖 + Jackson 3 + JSpecify）

### 升级理由（详见 [`reference/选型与对比/10`](../reference/选型与对比/10-SpringAI-vs-LangChain4j何时用何框架.md) §2）

| 能力 | 1.0 | 2.0 |
|------|-----|-----|
| Tool Calling 循环 | 手动管理 | `ToolCallingAdvisor` 自动注册 + 递归迭代 |
| 结构化输出校验 | `BeanOutputConverter` | + `StructuredOutputValidationAdvisor`（校验+重试） |
| MCP 集成 | client only | MCP Java SDK 2.0，client + server，`@McpTool` |
| 会话持久化 | `MessageWindowChatMemory` | `spring-ai-session`（event-sourced） |

### 任务清单

- [ ] **依赖升级**：Spring Boot 3.5.10 → 4.0.x，Spring AI 1.0.0 → 2.0.0
- [ ] **Jackson 2 → 3**：序列化代码全量检查
- [ ] **JSpecify null-safety**：类型系统增强，Optional 使用审视
- [ ] **回归测试**：阶段 1-3 的所有功能在新版本下都能跑
- [ ] **结构化输出迁移**：`BeanOutputConverter` → `StructuredOutputValidationAdvisor`
- [ ] **重跑评估集**：阶段 3 的 30 条 QA，升级后指标不退化

## 阶段 4：Agent + Anthropic 5 大 Workflow 模式（Week 5-6）

### 目标

### ⚠️ 重点认知（必读 [`reference/选型与对比/09`](../reference/选型与对比/09-企业级Java-AI架构选型真相.md)）

企业级 Agent 的主流形态**不是自主 ReAct Agent**，而是**Anthropic Workflow 模式**：
1. Prompt Chaining（串联）
2. Parallelization（并行/分段/投票）
3. Routing（路由分流）
4. Orchestrator-Workers（编排-工人）
5. Evaluator-Optimizer（评估-优化循环）

**"Workflow > Agent"**：能用工作流（确定性 DAG）解决的问题，不要用自主 Agent。

### 项目 P3：单文件代码评审助手

**功能**：
- 用户提交一个 Java 文件

**为什么选这个项目**：
- 演示 5 大模式最自然（Routing 按语言分 / Parallelization 三类问题并行 / Orchestrator-Workers 派工 / Evaluator-Optimizer 多轮改进 / Chaining 串接全流程）
- Java 工程师天天用，演示效果好
- **后续可作为 P5 客服系统的代码评审 Tool 复用**

### 8.4 成本工程（4 层缓存 + 路由）

> LLM 调用是按 token 计费的 RPC，不优化成本会烧钱。Stripe 用 vLLM 自部署省 73% 成本。

**4 层成本优化**：
- [ ] **L1：Prompt Cache**（Anthropic 5min/1hr cache，10x 成本节省）
- [ ] **L2：Semantic Cache**（语义相同请求复用，GPTCache / Langfuse Cache）
- [ ] **L3：Model Routing**（简单请求走 Haiku，复杂走 Sonnet/Opus）
- [ ] **L4：Long Context Tradeoff**（200K context 不替代 RAG，"Lost in the Middle"问题）

### 8.5 Agent 可靠性工程（Java 工程师的护城河）

> Red Hat 2026-06 文章核心论点：**Agent 可靠性本质是分布式系统问题，Java 工程师在类型系统 + 工程纪律上有天然优势**。

**为什么是 Java 工程师的护城河**：
- Python/算法工程师擅长调 prompt，但不擅长分布式系统
- Agent 失败模式（重试 / 幂等 / 补偿 / 超时 / 熔断）= 分布式系统的经典问题
- Java 生态有 Temporal / Restate / Spring Cloud / Resilience4j 等成熟方案

**长期任务**：
- [ ] **接入 Temporal 或 Restate**：把 Agent 执行包成工作流，获得 Checkpoint / 重试 / 补偿
- [ ] **幂等性设计**：所有 Tool 调用设计 idempotencyKey，防 Agent 重试导致副作用重复
- [ ] **审计合规**：EU AI Act Article 12 要求 append-only 审计日志，提前实现

| **Agent 可靠性工程** | ⭐⭐⭐⭐⭐ | 3-6 个月 | ⭐⭐⭐⭐⭐ | 有分布式系统经验 | 4 |
| **JDK 21+ 现代特性** | ⭐⭐⭐ | 持续 | ⭐⭐ | 所有 Java 工程师 | 穿插 |

### 8.8 不推荐投入的方向（跟进不押注）

- ❌ **Embabel / Koog / Google ADK Java**：Beta / 实验性，等 GA 后再说
- ❌ **AGNTCY / ASL**：Cisco Outshift 的死路方向
- ❌ **A2A 协议深度实现**：2-3 年内不会普及，跟进规范即可
- ❌ **形式化验证 Agent**：学术界方向，工业界不实用
- ❌ **GraphRAG 全量替换 RAG**：维护成本高，18 个月降本 99% 但仍未普及

## 选型原则速记（来自 [`reference/选型与对比/09`](../reference/选型与对比/09-企业级Java-AI架构选型真相.md) + [`reference/选型与对比/10`](../reference/选型与对比/10-SpringAI-vs-LangChain4j何时用何框架.md)）

- **80% 场景**：Spring AI 2.0 单框架 + Anthropic Workflow 模式
- **Quarkus 项目**：LangChain4j（Red Hat 官方，与 Spring AI 互斥）
- **复杂多 Agent**：Spring AI Alibaba Graph（国内）或 LangGraph4j（社区）
- **跨进程 Agent**：MCP（已可用）+ A2A（2-3 年内跟进不押注）
- **不要混用 Spring AI + LangChain4j 做两层架构**
- **Workflow > Agent**：能用确定性 DAG 解决的，不要用自主 Agent
- **LangChain4j 仅作阶段 1 入门**：理解 LLM 调用本质后转 Spring AI，最终位置是历史知识
- **评估驱动**：没有评估集的改动都是玄学（阶段 3 起所有改动都跑评估）
