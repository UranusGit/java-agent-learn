# Java 工程师 AI 实战学习计划（LangChain4j + Spring AI 双轨）

> 理念：**先动手，再补理论**。每个阶段都对应可运行的项目，遇到概念卡点再去 `desc/01~09` 查阅。
> 双轨学习：LangChain4j 主力起步（API 设计最像传统 Java），Spring AI 在第二阶段切入（与 Spring Boot 深度融合）。

---

## 总体路线图

```
阶段 0：环境准备 + 第一次调用（1 天）
    ↓
阶段 1：LangChain4j 基础（1 周）—— 命令行聊天机器人
    ↓
阶段 2：Spring AI 切入（1 周）—— 同样的机器人，用 Spring Boot 重写
    ↓
阶段 3：RAG 入门（2 周）—— 知识库问答（双框架各做一遍）
    ↓
阶段 4：Agent + Tool（2 周）—— 智能工具调用助手
    ↓
阶段 5：生产化（2-3 周）—— 流式、记忆持久化、可观测
    ↓
阶段 6：综合项目（持续）—— 自选业务方向，双框架融合
```

**总时长估算**：8-12 周（每周 10-15 小时投入）。

## 阶段 0：环境准备（Day 0）

### 目标
本地能跑通一个 LLM，准备好开发环境。

### 任务清单

- [ ] JDK 21（已确认）
- [ ] Maven 3.9+（`mvn -v` 验证）
- [ ] IDEA（安装 Markdown 插件 + 可选 Continue 插件）
- [ ] 选一个模型服务（二选一）：
  - **方案 A：Ollama（推荐，零成本）**
    - 安装：`brew install ollama`
    - 拉模型：`ollama pull qwen2.5:7b`
    - 启动：`ollama serve`（默认端口 11434）
  - **方案 B：DeepSeek API（推荐付费起步）**
    - 注册 `platform.deepseek.com`，充值 10 元够用很久
    - 获取 API Key
- [ ] 验证模型可用（curl 测试一次）

### 验收
能通过 curl 或 Postman 调通 LLM 并拿到回复。

## 阶段 1：LangChain4j 基础（Week 1）

### 目标
用纯 Java（无 Spring Boot）跑通最小 AI 应用，理解 LLM 调用的本质。

### 为什么不上 Spring Boot
**降低复杂度，专注理解 AI 调用本身**。Spring 的 IoC / 自动配置会遮蔽底层细节，对新手不利。

### Day 1-2：第一次调用

**任务**：
- [ ] 在 `pom.xml` 加 LangChain4j 依赖
- [ ] 写一个 `Main.java`，调用 LLM，打印回复
- [ ] 理解 `ChatLanguageModel` 抽象

**预期代码量**：10 行以内。

**学习要点**：
- `ChatLanguageModel` 是什么？它和 OpenAI SDK 的关系？
- 为什么 LangChain4j 能屏蔽不同模型供应商差异？

### Day 3-4：多轮对话 + 记忆

**任务**：
- [ ] 加 `ChatMemory`，实现多轮对话
- [ ] 在命令行做一个交互式 REPL（输入 → 回复 → 再输入）

**学习要点**：
- 为什么 LLM 是"无状态"的？（HTTP 是无状态的）
- ChatMemory 的存储结构（本质是消息列表）
- Token 窗口是什么？为什么会"忘事"？

### Day 5-6：第一个 Tool（Function Calling）

**任务**：
- [ ] 写一个 `@Tool` 方法，比如"获取当前时间"或"生成随机数"
- [ ] 让 LLM 自己决定何时调用工具
- [ ] 观察日志，理解 LLM 输出的工具调用 JSON

**学习要点**：
- Tool 的本质：LLM 输出结构化 JSON → Java 反射调用
- Tool 描述的重要性（描述写不好 LLM 就不会调）
- Function Calling 与传统 RPC 的区别

### Day 7：声明式 Agent

**任务**：
- [ ] 用 `AiServices` 重构，把上面三块整合成一个声明式接口
- [ ] 写一篇学习笔记（300 字左右）

### 阶段 1 验收标准

- ✅ 命令行能多轮对话
- ✅ 能调用至少 1 个自定义 Tool
- ✅ 能用自己的话讲清"ChatMemory 怎么工作的"
- ✅ Git 提交记录至少 5 次

### 交付物
- 项目路径：`demo01` 当前项目
- 包结构建议：
  ```
  src/main/java/org/example/
    ├── step01_basic/        // Day 1-2
    ├── step02_memory/       // Day 3-4
    ├── step03_tool/         // Day 5-6
    └── step04_agent/        // Day 7
  ```

## 阶段 2：Spring AI 切入（Week 2）

### 目标
用 Spring Boot 重写阶段 1 的功能，理解 Spring AI 的抽象设计，对比两个框架的差异。

### 为什么现在才上 Spring Boot
理解了 LangChain4j 的"手动装配"后，再看 Spring AI 的"Bean 注入"，你会立刻明白 Spring 在帮你做什么。

### Day 1-2：Spring Boot 项目搭建

**任务**：
- [ ] 新建 `demo02-spring-ai` 项目（与 demo01 平行）
- [ ] 加 `spring-boot-starter` + `spring-ai-openai-spring-boot-starter`
- [ ] 配置 `application.yml`
- [ ] 写一个 `/hello` 接口，调通 LLM

### Day 3-4：ChatClient + Advisor

**任务**：
- [ ] 用 `ChatClient.Builder` 注入 Bean
- [ ] 加 `MessageChatMemoryAdvisor` 实现多轮对话
- [ ] 加 `QuestionAnswerAdvisor`（RAG 的雏形，先体验，原理下阶段讲）
- [ ] 理解 Advisor 链（类似 Spring AOP）

**学习要点**：
- Advisor = 拦截器链，类比 Servlet Filter
- `ChatClient` vs `ChatLanguageModel` 的设计差异

### Day 5-6：Tool（@Tool 注解）

**任务**：
- [ ] 用 Spring AI 的 `@Tool` 注解，重写阶段 1 的工具
- [ ] 加一个调用数据库的 Tool（用 JdbcTemplate 即可）
- [ ] 让 LLM 自己选择调用

### Day 7：双框架对比笔记

**任务**：写一篇对比文档，至少回答：

1. **同样是 ChatMemory，两个框架的实现差异？**
2. **同样是 Tool，两个框架的描述机制差异？**
3. **生产场景你会选哪个？为什么？**

### 阶段 2 验收标准

- ✅ Spring Boot 应用能跑，提供 REST 接口
- ✅ 多轮对话 + 工具调用都能在 HTTP 上工作
- ✅ 完成"双框架对比"文档

### 交付物
- 新项目：`demo02-spring-ai/`
- 对比文档：`desc/对比-LangChain4j-vs-SpringAI.md`

## 阶段 3：RAG 入门（Week 3-4）

### 目标
做一个能问答本地 PDF/Markdown 文档的应用，理解 RAG 全链路。

### Week 3：用 LangChain4j 做

**Day 1-3：基础 RAG**
- [ ] 引入向量库（推荐先用 **本地 Chroma** 或 **Milvus Lite**）
- [ ] 加载 Markdown 文档 → 分块 → 向量化 → 入库
- [ ] 用 `ContentRetriever` 实现检索增强

**Day 4-7：体验分块策略**
- [ ] 体验 `DocumentSplitter` 的不同参数（chunk size、overlap）
- [ ] 准备 10 条 QA 测试集，对比不同参数的召回质量
- [ ] 理论补充：读 `desc/01-第三阶段总览与RAG优化.md` 的前两节

### Week 4：用 Spring AI 重做

**Day 1-3：Spring AI 的 RAG**
- [ ] 用 `VectorStore` 抽象 + `QuestionAnswerAdvisor`
- [ ] 体验 Spring AI 的 ETL 管道（`DocumentReader` / `DocumentTransformer` / `DocumentWriter`）

**Day 4-7：升级体验（可选进阶）**
- [ ] 引入混合检索（BM25 + 向量）—— 用 Elasticsearch
- [ ] 引入重排模型（bge-reranker 通过 TEI 服务）
- [ ] 用 RAGAS 跑自动评估

### 阶段 3 验收标准

- ✅ 能上传 PDF → 自动回答问题
- ✅ 能讲清"分块 / 向量化 / 检索 / 拼接 prompt"四步
- ✅ 有至少 10 条测试集和量化结果

### 交付物
- 在两个项目里都实现 RAG 模块
- 测试报告：`desc/RAG评测报告.md`

## 阶段 4：Agent + Tool 进阶（Week 5-6）

### 目标
做一个真实的 Agent 应用，能用自然语言完成多步骤任务。

### 推荐项目：个人助理 Agent

**功能**：
- 用户："我明天要去北京开会，帮我查下天气，再加一个提醒"
- Agent：① 调天气 API → ② 调日历 API → ③ 综合回复

### Week 5：LangChain4j 实现

- [ ] 设计 3-5 个 Tool（天气、日历、待办、查询数据库等）
- [ ] 用 `AiServices` 整合多 Tool
- [ ] 处理 Tool 调用失败（异常转 Observation）
- [ ] 限制 `max_iterations` 防止死循环

### Week 6：Spring AI 实现 + 对比

- [ ] 用 Spring AI 重做
- [ ] 对比两个框架的 Tool 路由准确率
- [ ] 写一篇"生产环境如何防 Agent 失控"的笔记

### 阶段 4 验收标准

- ✅ Agent 能串联 3+ 个 Tool 完成任务
- ✅ 有失败兜底逻辑
- ✅ 能讲清 Function Calling 的 JSON 协议

## 阶段 5：生产化（Week 7-9）

### 目标
让 Demo 走向"能上线"的状态。

### 任务清单（按优先级）

#### 必做
- [ ] **流式输出**：SSE 把 token 推到前端（解决"等 30 秒"的体验问题）
- [ ] **会话持久化**：ChatMemory 存 Redis（重启不丢）
- [ ] **超时 + 重试**：WebClient 配置合理的超时和降级
- [ ] **token 计费监控**：每次调用记录 input/output token
- [ ] **JSON 严格解析**：用 Structured Output / JSON Mode

#### 推荐
- [ ] **接入 Langfuse**：全链路 trace（见 `desc/06-LLMOps.md`）
- [ ] **限流**：基于用户 + token 双维度（Bucket4j）
- [ ] **多模型路由**：主模型失败切备用

#### 可选
- [ ] **多租户**：每个租户独立知识库 + prompt
- [ ] **审计日志**：所有 prompt 落库

### 阶段 5 验收标准

- ✅ 流式输出工作正常
- ✅ 重启后会话不丢
- ✅ Langfuse 能看到完整 trace
- ✅ 单接口压测能稳定运行

## 阶段 6：综合项目（持续）

### 目标
把前 5 阶段的能力融合到一个真实业务项目里。

### 推荐方向（任选其一）

#### 方向 A：智能运维助手
- 对接 K8s / Prometheus / 日志系统
- 用自然语言查服务状态、CPU、错误日志
- **优势**：Java 工程师最熟悉的领域

#### 方向 B：企业知识库客服
- 对接公司文档（Confluence / Notion / 内部 Wiki）
- 多租户 + 权限隔离
- **优势**：业务价值明显，简历加分

#### 方向 C：代码评审助手
- 接 GitLab / GitHub Webhook
- PR 创建时自动评审
- 可选：阶段 7 后接微调模型

### 交付要求
- README 文档清晰
- 至少一个架构图
- 部署文档（Docker Compose）
- 录一个 3 分钟演示视频
- 写一篇博客发出去

## 学习纪律

### 每周必做
1. **Git 提交**：至少 5 次，commit message 写清楚
2. **学习笔记**：写一篇（200-500 字），存到 `desc/notes/` 下
3. **代码注释**：关键理解点写一行 `//` 说明（不要堆 docstring）

### 每个阶段结束
- 写一篇阶段性总结
- 整理本阶段遇到的坑（`desc/pitfalls.md`）
- 决定是否进入下一阶段（不要赶进度）

### 反模式（不要做）
- ❌ 跳过基础直接看框架源码
- ❌ 同一天切换两个框架（混乱）
- ❌ 不写测试集就改 prompt
- ❌ 没跑通就上复杂特性（RAG/Agent）
