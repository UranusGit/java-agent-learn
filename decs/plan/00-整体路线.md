# Java 工程师 AI 实战学习计划（LangChain4j + Spring AI 双轨）

> 理念：**先动手，再补理论**。每个阶段都对应可运行的项目，遇到概念卡点再去 `reference/01~11` 查阅。
> 双轨学习：LangChain4j 主力起步（API 设计最像传统 Java），Spring AI 在第二阶段切入（与 Spring Boot 深度融合）。
>
> **⚠️ 重要校正（2026-07-13）**：基于 [`reference/11-企业级Java-AI架构选型真相.md`](../reference/11-企业级Java-AI架构选型真相.md) 的调研结论，企业级 Java AI 项目以**单框架 + Anthropic Workflow 模式**为主流，"混用 Spring AI + LangChain4j"是理论范式而非工程标配。因此本计划的重心**向 Spring AI 2.0 倾斜**：
> - **方向 1（最高 ROI）**：Spring AI 2.0 全栈 + Anthropic 5 大 Workflow 模式 + MCP
> - **方向 2（中 ROI）**：编排引擎（Spring AI Alibaba Graph 或 LangGraph4j）
> - **方向 3（高 ROI 赌性强）**：MCP / A2A 协议生态
>
> LangChain4j 仍作为阶段 1 入门（理解 LLM 调用本质），但**不再作为最终技术栈**。最终交付技术栈以 Spring AI 为主。

---

## 总体路线图

```
阶段 0：环境准备 + 第一次调用（1 天）
    ↓
阶段 1：LangChain4j 基础（1 周）—— 命令行聊天机器人【理解 LLM 调用本质】
    ↓
阶段 2：Spring AI 2.0 切入（1-2 周）—— 主技术栈定型，Web 化重写
    ↓
阶段 3：RAG 入门（2 周）—— 知识库问答（Spring AI 主，LangChain4j 对照）
    ↓
阶段 4：Agent + Anthropic Workflow 模式（2 周）—— 智能工具调用 + 5 大工作流
    ↓
阶段 5：生产化（2-3 周）—— 流式、记忆持久化、可观测、MCP 接入
    ↓
阶段 6：综合项目（持续）—— Spring AI 单框架全栈交付
    ↓
阶段 7（可选进阶）：编排引擎（1-2 周）—— Alibaba Graph / LangGraph4j
```

**总时长估算**：8-12 周主线（每周 10-15 小时投入）；阶段 7 是进阶方向，时间另算。

## 阶段 1：LangChain4j 基础（Week 1）

### 目标
用纯 Java（无 Spring Boot）跑通最小 AI 应用，理解 LLM 调用的本质。

### 为什么不上 Spring Boot
**降低复杂度，专注理解 AI 调用本身**。Spring 的 IoC / 自动配置会遮蔽底层细节，对新手不利。

### Day 1-2：第一次调用

**任务**：
- [ ] 在 `pom.xml` 加 LangChain4j 依赖
- [ ] 写一个 `Main.java`，调用 LLM，打印回复
- [ ] 理解 `ChatLanguageModel` 抽象

**预期代码量**：10 行以内。

**学习要点**：
- `ChatLanguageModel` 是什么？它和 OpenAI SDK 的关系？
- 为什么 LangChain4j 能屏蔽不同模型供应商差异？

### Day 3-4：多轮对话 + 记忆

**任务**：
- [ ] 加 `ChatMemory`，实现多轮对话
- [ ] 在命令行做一个交互式 REPL（输入 → 回复 → 再输入）

**学习要点**：
- 为什么 LLM 是"无状态"的？（HTTP 是无状态的）
- ChatMemory 的存储结构（本质是消息列表）
- Token 窗口是什么？为什么会"忘事"？

### Day 5-6：第一个 Tool（Function Calling）

**任务**：
- [ ] 写一个 `@Tool` 方法，比如"获取当前时间"或"生成随机数"
- [ ] 让 LLM 自己决定何时调用工具
- [ ] 观察日志，理解 LLM 输出的工具调用 JSON

**学习要点**：
- Tool 的本质：LLM 输出结构化 JSON → Java 反射调用
- Tool 描述的重要性（描述写不好 LLM 就不会调）
- Function Calling 与传统 RPC 的区别

### Day 7：声明式 Agent

**任务**：
- [ ] 用 `AiServices` 重构，把上面三块整合成一个声明式接口
- [ ] 写一篇学习笔记（300 字左右）

## 阶段 4：Agent + Tool 进阶 + Anthropic Workflow 模式（Week 5-6）

### 目标
做一个真实的 Agent 应用，能用自然语言完成多步骤任务；**掌握 Anthropic 5 大 Workflow 模式**（企业 Agent 的真实形态）。

### ⚠️ 重点认知（必读 `reference/11`）

企业级 Agent 的主流形态**不是自主 ReAct Agent**，而是**Anthropic Workflow 模式**：
1. Prompt Chaining（串联）
2. Parallelization（并行/分段/投票）
3. Routing（路由分流）
4. Orchestrator-Workers（编排-工人）
5. Evaluator-Optimizer（评估-优化循环）

**"Workflow > Agent"**：能用工作流（确定性 DAG）解决的问题，不要用自主 Agent。

### 推荐项目：个人助理 Agent

**功能**：
- 用户："我明天要去北京开会，帮我查下天气，再加一个提醒"
- Agent：① 调天气 API → ② 调日历 API → ③ 综合回复

### Week 5：Spring AI 主实现（含 Workflow 模式）

- [ ] 设计 3-5 个 Tool（天气、日历、待办、查询数据库等）
- [ ] 用 Spring AI 的 `@Tool` + `ToolCallingAdvisor` 整合多 Tool
- [ ] 处理 Tool 调用失败（Advisor 捕获异常，返回降级响应）
- [ ] **实践 Anthropic 5 大 Workflow 模式**（每种至少一个示例）
- [ ] 限制 `max_iterations` 防止死循环
- [ ] 用 `StructuredOutputValidationAdvisor` 做结构化输出

### Week 6：LangChain4j 对照 + 工作流深化

- [ ] 用 LangChain4j `AiServices` 重做一遍（理解两框架差异）
- [ ] 写一篇"Workflow vs Agent：何时用哪个"的笔记
- [ ] 写一篇"生产环境如何防 Agent 失控"的笔记

### 阶段 4 验收标准

- ✅ Agent 能串联 3+ 个 Tool 完成任务
- ✅ 有失败兜底逻辑
- ✅ 能讲清 Function Calling 的 JSON 协议
- ✅ **能讲清 Anthropic 5 大 Workflow 模式分别适用什么场景**
- ✅ 知道为什么 "Workflow > Agent"

## 阶段 5：生产化（Week 7-9）

### 目标
让 Demo 走向"能上线"的状态。**以 Spring AI 2.0 单框架为主线**。

### 任务清单（按优先级）

#### 必做
- [ ] **流式输出**：SSE 把 token 推到前端（解决"等 30 秒"的体验问题）
- [ ] **会话持久化**：`spring-ai-session`（事件溯源记忆）或 ChatMemory 存 Redis
- [ ] **超时 + 重试**：WebClient 配置合理的超时和降级
- [ ] **token 计费监控**：每次调用记录 input/output token
- [ ] **JSON 严格解析**：`StructuredOutputValidationAdvisor`

#### 推荐（企业级标配）
- [ ] **Advisor 链**：鉴权 / 限流（Bucket4j）/ 审计 / 多租户 / 敏感词
- [ ] **可观测性**：Micrometer + OpenTelemetry GenAI Semantic Conventions
- [ ] **多模型路由**：主模型失败切备用（ChatClient fallback）
- [ ] **MCP 接入**：用 `@McpTool` / `@McpResource` 暴露 Tool（押注协议层）

#### 可选
- [ ] 接入 Langfuse（全链路 trace，见 `reference/06-LLMOps.md`）
- [ ] 审计日志：所有 prompt 落库

### 阶段 5 验收标准

- ✅ 流式输出工作正常
- ✅ 重启后会话不丢
- ✅ 至少有 3 个 Advisor（鉴权 / 限流 / 审计）
- ✅ Micrometer + Prometheus 指标可视化
- ✅ 至少 1 个 MCP Tool 可用
- ✅ 单接口压测能稳定运行

## 选型原则速记（来自 `reference/11`）

- **80% 场景**：Spring AI 2.0 单框架 + Anthropic Workflow 模式
- **复杂多 Agent**：Spring AI Alibaba Graph（国内）或 LangGraph4j（社区）
- **跨进程 Agent**：MCP + A2A 协议
- **不要混用 Spring AI + LangChain4j 做两层架构**
- **Workflow > Agent**：能用确定性 DAG 解决的，不要用自主 Agent
