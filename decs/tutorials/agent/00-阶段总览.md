# 阶段 4 总览 - Agent + Tool 进阶

> 阶段 1-2 你学会了"如何用框架"。阶段 4 你要学的是"如何让 Agent 在生产中稳定运行"。
> **重点不是新 API，而是工程化、防失控、可评估**。

---

## 1. 阶段目标

### 1.1 你要掌握的核心能力

| 能力 | 现状 | 阶段 4 后 |
|------|------|----------|
| 写 Tool | ✅ 会写基础 Tool | 会设计高质量 Tool（描述、参数、错误处理） |
| 多 Tool 编排 | ❌ 没做过 | 会做 Tool 路由、并行调用、子 Agent 分工 |
| 防止 Agent 失控 | ❌ 没经验 | 会设迭代上限、超时、循环检测、成本控制 |
| Agent 评估 | ❌ 凭感觉 | 会做准确率测试、回归测试 |
| 完整项目 | ❌ Demo 级 | 能做出可写进简历的项目 |

### 1.2 关键认知转变

```
阶段 1-2: "能跑通就行"
阶段 4  : "能稳定跑、能监控、能评估、能扩展"
```

## 2. 推荐的两个实战项目

### 项目 A：个人助理 Agent（推荐入门）

**业务场景**：
> 用户："我明天要去北京开会，帮我查天气，再加一个提醒"
> Agent：① 调天气 API → ② 调日历 API → ③ 综合回复

**为什么推荐**：
- 业务简单，能专注 Agent 工程
- Tool 都是外部 API（OpenWeather、Calendar API），不需要复杂内部系统
- 能完整覆盖 Tool 设计、编排、防失控

### 项目 B：智能运维 Agent（推荐进阶）

**业务场景**：
> 用户："用户服务有几个副本？CPU 高不高？"
> Agent：① 调 K8s API → ② 调 Prometheus → ③ 综合

**为什么推荐**：
- Java 工程师最熟悉的领域
- 企业级场景，简历加分
- 涉及更多工程问题（鉴权、限流、错误码）

### 项目 C：知识库客服（备选）

**业务场景**：
> 用户："你们的退货政策是什么？"
> Agent：① RAG 检索 → ② LLM 回答 → ③ 工单生成（必要时）

**为什么推荐**：
- 阶段 3 的延伸，把 RAG 和 Agent 结合
- 业务价值明显

## 5. 框架选择

阶段 4 你可以继续用：
- **LangChain4j**（API 更简洁，适合做单 Agent）
- **Spring AI**（生态融合好，适合做企业级多 Tool）
- **两者混用**（推荐：核心 Agent 用 LangChain4j，业务系统用 Spring AI）

教程里我会**两个框架的代码都给**，你自己选。

## 6. 准备工作

### 6.1 模型准备

阶段 4 强烈建议用 **DeepSeek API**，原因：
- 本地 Ollama 在 Tool 调用上准确率不如云端模型
- Function Calling 测试需要稳定结果，本地小模型会让你怀疑人生
- 充值 20 元够学完整个阶段

### 6.2 外部服务准备（按项目选）

#### 项目 A（个人助理）
- 天气：OpenWeatherMap 免费版（注册拿 Key）或伪造数据
- 日历：先用本地内存伪造，后续可接 Google Calendar
- 待办：本地内存伪造

#### 项目 B（智能运维）
- Minikube / Kind 本地 K8s
- Prometheus（Docker 起）
- Grafana（可选）

### 6.3 心态准备

**阶段 4 会有挫败感**：
- Agent 偶尔会"智障"，调用错的 Tool 或填错参数
- 同样的 prompt，10 次跑可能有 8 次成功 2 次失败
- 调试比阶段 1-2 难

**这是正常的**。生产级 Agent 工程就是把这些失败概率压到最低，**不是追求 100%**。

## 7. 阶段 4 验收标准

完成阶段 4 后，你应该能：

- [ ] 设计 5+ 个高质量 Tool，描述清晰、参数完整
- [ ] 实现 Tool 调用失败兜底
- [ ] 实现 Agent 迭代上限（防死循环）
- [ ] 实现成本控制（token 监控 + 告警）
- [ ] 完成 1 个完整项目（A 或 B），可以演示
- [ ] 准备至少 20 条测试用例，统计 Tool 调用准确率
- [ ] **能讲清"Agent 失败的常见模式有哪些"**
