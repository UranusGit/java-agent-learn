# LangChain4j 09 - 常见错误与排查手册

> 按场景分类的错误排查指南，遇到问题先来这里查。
> 每个错误都给出：**症状 → 原因 → 诊断步骤 → 解决方案**。

---

## 1. 连接类错误

### 1.1 Connection refused

**症状**：
```
java.net.ConnectException: Connection refused: /127.0.0.1:11434
```

**诊断流程**：
```bash
# 1. 服务是否启动？
ollama serve           # Mac/Linux
# 或
lsof -i :11434         # 看端口

# 2. 防火墙是否拦截？
nc -zv localhost 11434

# 3. Docker 内的 Chroma 是否运行？
docker ps | grep chroma
```

**常见原因**：
- Ollama 服务没起
- Chroma 容器挂了
- 端口被其他进程占用
- Docker 没启动

### 1.2 SSL 握手失败

**症状**：`SSLHandshakeException`
**原因**：JDK 版本太旧 / TLS 协议不匹配
**解决**：升级 JDK 21+

### 1.3 超时（Timeout）

**症状**：`SocketTimeoutException: Read timed out`

**原因分类**：

| 原因 | 诊断 |
|------|------|
| 模型推理慢（小模型/长上下文） | 调大 `timeout` |
| 网络问题 | `curl` 测试直连 |
| 服务挂了 | 检查 Ollama/DeepSeek 状态 |

**解决方案**：
```java
OllamaChatModel.builder()
        .timeout(Duration.ofMinutes(5))  // 默认太短
        .build();
```

## 2. 认证类错误

### 2.1 401 Unauthorized

**症状**：`{"error":{"message":"Invalid API key"}}`

**诊断**：
```bash
echo $DEEPSEEK_API_KEY   # 看环境变量是否设置
```

**常见坑**：
- IDEA 启动后改了环境变量 → **重启 IDEA**
- 用了 zshrc 但 IDEA 启动时没读 → 在 IDEA Run Configuration 里设置
- Key 前后有空格 → `tr -d ' '` 清理

### 2.2 余额不足（402）

**症状**：`Insufficient balance`
**解决**：充值。

### 2.3 限流（429）

**症状**：`Rate limit exceeded`
**解决**：
```java
OpenAiChatModel.builder()
        .maxRetries(3)
        .build();
```

或加全局限流（见 `reference/06-LLMOps.md`）。

## 3. 模型相关错误

### 3.1 Model not found

**症状**：`model 'qwen2.5:7b' not found`

**原因**：
- 本地没拉模型 → `ollama pull qwen2.5:7b`
- 模型名拼错（区分大小写、tag）
- DeepSeek 用了不存在的模型名

**诊断**：
```bash
ollama list              # 本地有哪些模型
ollama show qwen2.5:7b   # 看具体模型信息
```

### 3.2 模型加载失败（OOM）

**症状**：`CUDA out of memory` 或 `Killed`

**原因**：内存/显存不够。

**解决**：
- 换更小的模型（14B → 7B → 0.5B）
- 用更激进的量化（`q8_0` → `q4_K_M`）
- 调小 `num_ctx`

### 3.3 输出乱码

**症状**：返回内容是乱码或非预期语言

**原因**：
- 模型不支持中文
- temperature 太高导致发散
- prompt 措辞不当

**解决**：
```java
@SystemMessage("你必须用中文回答")
```

## 4. 依赖与构建错误

### 4.1 ClassNotFoundException

**症状**：`java.lang.ClassNotFoundException: dev.langchain4j.xxx.XxxModel`

**原因**：
- 该模块没引入 pom.xml
- 版本不匹配

**解决**：
```bash
mvn dependency:tree | grep langchain4j
```

确认所有 langchain4j 模块版本一致。

### 4.2 版本冲突

**症状**：`NoSuchMethodError` 或 `NoClassDefFoundError`

**原因**：多个依赖引入了不同版本的 langchain4j。

**解决**：用 BOM 统一管理
```xml
<dependencyManagement>
    <dependencies>
        <dependency>
            <groupId>dev.langchain4j</groupId>
            <artifactId>langchain4j-bom</artifactId>
            <version>1.0.1</version>
            <type>pom</type>
            <scope>import</scope>
        </dependency>
    </dependencies>
</dependencyManagement>
```

### 4.3 中文乱码（编译期）

**症状**：Java 文件中文字符串编译后变 `???`

**解决**：
- pom.xml 加 `<project.build.sourceEncoding>UTF-8</project.build.sourceEncoding>`
- IDEA 设置 File Encoding 为 UTF-8
- JVM 参数 `-Dfile.encoding=UTF-8`

## 5. AiServices 相关错误

### 5.1 接口无法实例化

**症状**：`IllegalArgumentException: Interface must be...`

**原因**：接口里混了不兼容的方法。

**规则**：
- AiServices 接口每个方法都必须能用 LLM 实现
- 不能混入普通业务方法

### 5.2 SystemMessage 没生效

**诊断**：开 `logRequests(true)` 看请求体，第一条 message 是不是 system。

**常见原因**：
- `@SystemMessage` 写在方法上而非接口上 → 类级别的才稳定生效
- 用 `systemMessageProvider` 时 lambda 写错

### 5.3 结构化输出失败

**症状**：`JsonProcessingException` 或字段为 null

**排查步骤**：
1. 打印 LLM 原始返回（开 logResponses）
2. 看返回是否合法 JSON
3. 字段名是否和 record 属性名匹配
4. 用 `responseFormat("json_object")` 强制 JSON

## 6. Tool 调用错误

### 6.1 Tool 不被调用

**诊断**：开 `logRequests(true)`，看请求体的 `tools` 数组里有没有你的 Tool。

**常见原因**：
- 忘了 `.tools(new XxxTools())`
- Tool 方法没加 `@Tool`
- Tool 描述模糊，LLM 不知道何时用

**调试技巧**：让 LLM 输出推理过程
```java
@SystemMessage("思考过程用 <thought>...</thought> 标记输出")
```

### 6.2 Tool 参数错误

**症状**：LLM 调了 Tool，但参数填错（类型不对、值不对）

**原因**：
- 参数描述不清晰
- 参数类型复杂（嵌套对象）

**解决**：
- 用基本类型（String/int/long）
- 参数描述加示例：`@P("日期格式 yyyy-MM-dd，如 2026-07-12")`
- 在 Tool 内部做参数校验

### 6.3 死循环调用

**症状**：LLM 反复调用同一个 Tool，永不结束。

**解决**：
```java
AiServices.builder(Xxx.class)
        // 限制迭代次数（不同版本 API 可能不同，查文档）
        .chatMemory(MessageWindowChatMemory.withMaxMessages(20))
        .build();
```

或在 Tool 内部加短路：
```java
private final Set<String> queriedKeys = ConcurrentHashMap.newKeySet();

@Tool("查询用户")
public User query(String name) {
    if (queriedKeys.contains(name)) {
        return User.empty("已查询过");
    }
    queriedKeys.add(name);
    // ...
}
```

## 7. RAG 错误

### 7.1 检索结果与问题无关

**诊断**：开日志看拼到 prompt 里的上下文。

**排查路径**：
```
查到的是垃圾 → Embedding 模型不适合中文？换 bge-zh
                → 分块太大太小？调整 chunk size
                → 没用 minScore 过滤？加阈值
```

### 7.2 索引后搜不到

**诊断**：直接看向量库里有几条数据
```java
System.out.println("总数: " + store.getAll().size());
```

**常见原因**：
- 索引脚本没跑成功
- collection 名字不一致
- 文档分块后内容为空

### 7.3 答案有幻觉

**症状**：检索到了正确片段，但 LLM 编造。

**解决**：强化 system prompt
```java
@SystemMessage("""
    严格根据下面的上下文回答。如果上下文中没有，回答"我不知道"。
    不要编造。引用上下文中的原话。
    
    上下文：
    {{it}}
    """)
```

## 8. 流式输出错误

### 8.1 浏览器一直 pending

**原因**：
- Spring MVC 模式下用 Flux 不工作
- `produces` 没设对

**解决**：
```java
@GetMapping(value = "/chat", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
public Flux<String> chat(...) { ... }
```

### 8.2 中文被切断

**原因**：SSE 按字节推送，UTF-8 多字节字符被分割。
**解决**：LangChain4j 内部已处理。如自定义实现，注意用 `String` 而非 `byte[]`。

### 8.3 客户端断开后服务端不知道

**解决**：
```java
emitter.onCompletion(() -> System.out.println("客户端断开"));
emitter.onTimeout(() -> System.out.println("超时"));
emitter.onError(e -> e.printStackTrace());
```

## 9. 性能问题

### 9.1 响应慢

**诊断步骤**：
1. 测裸调 `curl` 看模型本身速度
2. 如果裸调快，是 LangChain4j 层慢
3. 开日志看请求/响应大小

**常见原因**：
- prompt 太长 → 精简
- ChatMemory 太大 → 缩小窗口
- 每次都重新加载模型 → 设 `OLLAMA_KEEP_ALIVE`

### 9.2 内存持续增长

**原因**：
- ChatMemory 没设上限
- Session 没清理
- 缓存无淘汰策略

**解决**：
```java
ChatMemory memory = MessageWindowChatMemory.withMaxMessages(20);  // 别用 Integer.MAX_VALUE
```

### 9.3 并发上不去

**原因**：
- Ollama `OLLAMA_NUM_PARALLEL=1`（默认）
- HttpClient 用了阻塞 IO

**解决**：
- Ollama 设 `OLLAMA_NUM_PARALLEL=4`
- 用 WebFlux 替代 MVC

## 10. 调试工具箱

### 10.1 一键开关日志

```java
// 在 Main 开头加这个，全局开关
System.setProperty("org.slf4j.simpleLogger.defaultLogLevel", "DEBUG");
```

### 10.2 抓 HTTP 包

```bash
# 抓本地 Ollama 的 HTTP 流量
sudo tcpdump -i lo0 port 11434 -A

# 或用 mitmproxy 做反向代理
mitmproxy --mode reverse:http://localhost:11434 -p 8080
```

### 10.3 Langfuse 追踪

接入 Langfuse 看完整链路（见 `reference/06-LLMOps.md`）。

### 10.4 检查日志的清单

开 `logRequests(true)` + `logResponses(true)` 后，看：

| 检查项 | 关注点 |
|--------|--------|
| `messages` 数组 | system 在第一位？历史被正确拼入？ |
| `tools` 数组 | Tool 被正确序列化？描述清晰？ |
| `tool_calls` | LLM 决策正确？参数填对？ |
| `usage` | token 数合理？没爆掉？ |
| `finish_reason` | 是 `stop` 还是 `length`（被截断）？ |

## 11. 求助前的自查清单

提交 issue 或问 AI 前，先准备好：

1. **LangChain4j 版本**：`mvn dependency:tree | grep langchain4j`
2. **JDK 版本**：`java -version`
3. **模型**：用的什么模型？多大？
4. **完整报错栈**：不要截断
5. **可复现的最小代码**：删掉业务逻辑，留核心调用
6. **请求体**：开 logRequests 后的实际 JSON

## 12. 学习建议

### 12.1 遇到错误的正确姿势

1. **先读完整错误信息**（包括 caused by）
2. **定位错误类型**（连接？认证？模型？代码？）
3. **查本手册**
4. **搜索 GitHub issues**：`github.com/langchain4j/langchain4j/issues`
5. **构造最小复现**：删到不能再删

### 12.2 避坑纪律

- **不要一报错就改代码**：先理解错误
- **不要无脑重试**：相同错误再试 10 次还是错
- **不要绕过检查**：`@ts-ignore` 类似手法在 Java 里也别用
- **记录踩坑**：写进 `plan/progress.md` 的踩坑表

## 13. 错误速查表（打印出来贴墙上）

| 错误关键字 | 可能原因 | 立即排查 |
|----------|---------|---------|
| `Connection refused` | 服务没起 | `lsof -i :port` |
| `401 Unauthorized` | API Key 错 | `echo $API_KEY` |
| `404 Not Found` | baseUrl 错 | 检查末尾斜杠 |
| `429 Too Many Requests` | 限流 | 加重试 |
| `402 Payment Required` | 余额不足 | 充值 |
| `model not found` | 模型名错 | `ollama list` |
| `OutOfMemoryError` | 内存不够 | 换小模型 |
| `ClassNotFoundException` | 缺依赖 | `mvn dependency:tree` |
| `NoSuchMethodError` | 版本冲突 | 用 BOM |
| `JsonProcessingException` | JSON 解析失败 | 加 `responseFormat("json_object")` |
| `SocketTimeoutException` | 超时 | 调大 `timeout` |
